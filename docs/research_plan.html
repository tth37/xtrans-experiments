1. Executive Summary

Core Thesis: Every GPU collective communication library (NCCL, RCCL, HCCL, oneCCL) assumes a shared OS namespace for intra-node GPU-to-GPU communication. Container isolation breaks this assumption universally. We propose XTrans (Cross-container Transport), a fabric-agnostic broker layer that mediates topology discovery and memory handle exchange across container boundaries, enabling any CCL to recover native interconnect bandwidth (NVLink, Infinity Fabric, Xe Link, HCCS/UnifiedBus) without sacrificing per-GPU isolation.

Our exp1–2 results proved that NCCL’s cuMem VMM path can recover 100% NVLink bandwidth in per-GPU containers on NVIDIA hardware. But this solution is:

  • NVIDIA-only — relies on NCCL-specific env vars (NCCL_CUMEM_ENABLE, NCCL_HOSTID) and CUDA VMM API
  • Fragile — depends on three undocumented “gates” (hostHash, shmDev, IPC socket) that could change in any NCCL release
  • Not extensible — NCCL’s intra-node transport array is hardcoded; no plugin interface exists for P2P/SHM transports

The research question becomes: Can we build a general mechanism that enables high-bandwidth cross-container GPU communication for any vendor’s hardware, any CCL, and any container runtime?

The research challenge is discovering whether a common broker protocol can unify these divergent mechanisms while preserving native performance. DMA-BUF FDs emerge as a potential universal substrate (available on NVIDIA, AMD, and Intel), but this has never been tested for cross-container GPU P2P.

2.2 Will NVIDIA Solve This Themselves?

No, not in the next 2–3 years. Pre-research identifies structural reasons:

  1. IMEX is GB200 NVL72 only. Requires 4th-gen NVSwitches + Grace CPUs + Blackwell GPUs. Does not work on A100, H100, or even B200 DGX without NVSwitch fabric. The vast majority of deployed GPU infrastructure will never run IMEX.
  2. NVIDIA’s container strategy moves toward stronger isolation, not sharing. Confidential Containers, Kata GPU support, namespace-based security — all point away from making cross-container GPU communication easier.
  3. NCCL has no intra-node transport plugin interface and shows no signs of adding one. P2P and SHM transports are hardcoded in transport.cc. The plugin API only covers NET (inter-node), Tuner, Profiler, and Env.
  4. NCCL 2.29 acknowledges the problem but only offers a broken env var toggle. NCCL_IPC_USE_ABSTRACT_SOCKET=0 is documented as a workaround for “multiple containers in certain situations,” but does not reliably work.
  5. Cloud providers avoid the problem entirely by giving exclusive node access to training jobs. AWS EFA and Azure InfiniBand focus on inter-node RDMA only.

2.6 What Elevates XTrans to a Research Contribution

Based on analysis of successful systems papers (Xen/SOSP’03, Exokernel/SOSP’95, “Simple Testing”/OSDI’14, Linux Scalability/OSDI’10), the key is that XTrans must teach the community something new about the problem space, not just solve a specific instance. Four potential research contributions:

  1. Cross-vendor failure taxonomy (measurement contribution): First systematic analysis showing that NCCL, RCCL, and oneCCL all fail through the same three-step pattern (topology → transport → IPC) under container isolation, despite having completely different internal implementations. Precedent: “Simple Testing Can Prevent Most Critical Failures” (OSDI’14) discovered a universal pattern across 5 distributed systems and was accepted at OSDI as a measurement paper.
  2. DMA-BUF as universal GPU IPC substrate (systems insight): Pre-research reveals that DMA-BUF file descriptors are available on all three major GPU platforms (NVIDIA via cuMem, AMD via hsa_amd_portable_export_dmabuf, Intel via Level Zero IPC). DMA-BUF FDs can be passed via SCM_RIGHTS across any container boundary. If we can demonstrate that DMA-BUF provides a universal cross-container GPU memory sharing primitive with native performance, that is a genuine systems discovery — not transplanting cuMem, but identifying a Linux kernel subsystem that unifies vendor-divergent IPC mechanisms.
  3. GPU isolation spectrum model (conceptual contribution): Define “GPU communication isolation” formally as a spectrum from L0 (no isolation, --ipc=host) through L1 (namespace-only, broken communication) to L2 (brokered, controlled P2P) to L3 (hardware, MIG/SR-IOV). If we can build a model that predicts bandwidth recovery for a given isolation configuration, that is a theoretical contribution.
  4. Minimal interception surface (completeness argument): Prove that intercepting exactly N functions (e.g., gethostname, stat, connect) is necessary and sufficient to recover native GPU communication performance. This is analogous to the Exokernel’s argument about “secure bindings” being the minimal kernel interface.

Recommended framing for the paper: “We present the first systematic cross-vendor measurement of how container isolation breaks GPU collective communication. We discover that all major CCLs fail through the same three mechanisms [taxonomy]. We identify DMA-BUF file descriptors as a universal cross-container GPU memory sharing primitive, and show that a lightweight broker architecture recovers ≥95% of native interconnect bandwidth across NVIDIA, AMD, and Intel GPUs.”

3. The Problem: Container Isolation Breaks GPU Communication Universally

3.1 The Universal Pattern

Every GPU CCL follows the same intra-node communication pattern:

Step 1: TOPOLOGY DISCOVERY
  • Detect which GPUs are on the same physical node
  • Detect interconnect type and bandwidth between GPU pairs
  • Build a topology graph (NVLink mesh, PCIe tree, xGMI mesh, etc.)

Step 2: TRANSPORT SELECTION
  • For each GPU pair, select the fastest available transport
  • Intra-node: P2P (NVLink/xGMI/Xe Link) > SHM (host staging) > NET (socket/RDMA)
  • Inter-node: RDMA > TCP

Step 3: IPC HANDLE EXCHANGE
  • Allocate GPU memory for communication buffers
  • Export memory handles (vendor-specific: cudaIpcMemHandle, cuMem FD, HSA signal, ...)
  • Exchange handles between GPU processes via IPC mechanism
  • Map peer memory into local address space

Step 4: DATA TRANSFER
  • Use the selected transport for collective operations
  • P2P: direct load/store over NVLink/xGMI
  • SHM: GPU → host buffer → GPU via shared memory
  • NET: GPU → NIC → network → NIC → GPU

Container isolation breaks Steps 1–3 in vendor-specific but structurally identical ways:

4.4 Key Design Principles

  1. Zero CCL modification: XTrans works via LD_PRELOAD, environment variables, and filesystem mounts. No CCL source code changes required. This is critical for adoption — users should not need to maintain a fork of NCCL/RCCL.
  2. Minimal privilege escalation: Only the host daemon needs elevated privileges (to see all GPUs and pass FDs). Containers remain unprivileged with single-GPU visibility. The security boundary is the daemon’s access control policy.
  3. Fabric-agnostic core, vendor-specific backends: The topology service and handle broker define vendor-neutral protocols. Vendor-specific logic is isolated in backend plugins (one per vendor). Adding a new vendor means writing one backend, not redesigning the system.
  4. Graceful degradation: If the daemon is unavailable or a backend is missing, the CCL falls back to its default behavior (TCP/socket). XTrans is an optimization layer, not a hard dependency.
  5. Container-runtime agnostic: Works with Docker, containerd, CRI-O, Kata Containers (with VFIO passthrough). The only requirement is the ability to bind-mount a UDS socket directory into the container.

6. Research Plan: Phased Experiments

The plan is structured as a series of experiments that progressively build the XTrans system, starting from what we already know (NVIDIA/NCCL) and expanding to multi-vendor and multi-fabric support.

6.1 Phase 1: Systematize the NCCL Container Gates (Exp A)

Exp A: NCCL Gate Taxonomy and Minimal Interception

Goal: Transform our ad-hoc exp2 workaround (env vars + shared mounts) into a principled, minimal interception layer. Understand exactly which system calls NCCL makes for each gate, and build the thinnest possible LD_PRELOAD shim that satisfies them.

TaskMethodDeliverable
A1: Syscall tracing Run NCCL under strace -e trace=network,ipc,file in both shared-namespace and isolated-container configurations. Diff the syscall traces to identify every divergence point. Complete syscall-level map of NCCL’s three gates. Identify any additional gates beyond hostHash/shmDev/IPC socket.
A2: Minimal LD_PRELOAD shim Write a C library that intercepts only the identified syscalls and returns values that satisfy NCCL’s gates. No daemon yet — hardcoded values. libxtrans_nccl.so (~200 LoC) that recovers NVLink P2P in per-GPU containers without any NCCL_* env vars or shared /dev/shm.
A3: Version stability test Test the shim against NCCL 2.18, 2.19, 2.20, 2.21, 2.25, 2.27, 2.28. Identify which gates changed across versions. Compatibility matrix. Understand how fragile the interception points are.
A4: Security analysis Analyze what the shim exposes. Can a malicious container use the shim to access another container’s GPU memory? What is the minimal trust model? Threat model document. Define what “isolation” means when P2P is enabled.

Success criteria: The shim recovers ≥99% of NVLink bandwidth on A100 (node192) without any NCCL-specific env vars, without shared /dev/shm, and without --network=host. The only requirement is the shim library and a bind-mounted UDS socket directory.

Why this matters: This experiment transforms our exp2 finding from “here are some env vars that happen to work” into “here is a principled interception layer with a known syscall surface, tested across NCCL versions, with a defined security model.” This is the foundation for everything that follows.

6.2 Phase 1.5: DMA-BUF Feasibility Spike (Exp A′)

Exp A′: DMA-BUF as Universal Cross-Container GPU Memory Sharing Primitive

Goal: Test whether Linux DMA-BUF file descriptors can serve as a vendor-agnostic cross-container GPU memory sharing mechanism. This de-risks Phase 2 (AMD) and Phase 3 (unified daemon) by validating the key insight from pre-research: DMA-BUF FDs are available on NVIDIA, AMD, and Intel, and can be passed via SCM_RIGHTS across any container boundary.

Motivation from pre-research: We discovered that all three GPU vendors expose DMA-BUF export capabilities:

  • NVIDIA: cuMem VMM handles are internally DMA-BUF FDs on Linux
  • AMD: hsa_amd_portable_export_dmabuf(ptr, size, &fd, &offset) — already used by RCCL for RDMA NIC registration
  • Intel: Level Zero ze_ipc_mem_handle_t contains a DMA-BUF FD in its first 4 bytes; oneCCL’s sockets mode passes these via UDS

If DMA-BUF export → FD passing → DMA-BUF import → GPU P2P mapping works on NVIDIA (where we have hardware), that’s strong evidence for a universal mechanism.

TaskMethodDeliverable
A′1: NVIDIA DMA-BUF P2P test On node192 (A100): allocate GPU memory via cuMem VMM, export as DMA-BUF FD, pass FD via UDS SCM_RIGHTS to a process in a different container, import and map as GPU-accessible VA. Measure P2P bandwidth. Proof that DMA-BUF FD passing enables cross-container GPU P2P on NVIDIA. Bandwidth comparison vs native cuMem IPC path.
A′2: DMA-BUF vs cuMem overhead Compare three IPC paths on NVIDIA: (a) native cuMem FD (what exp2 uses), (b) DMA-BUF export/import, (c) legacy cudaIpcMemHandle (requires shared IPC NS). Measure init latency and steady-state bandwidth. Quantify whether DMA-BUF adds overhead vs vendor-native IPC. If overhead is <1%, DMA-BUF is viable as the universal substrate.
A′3: Kernel requirements Identify minimum kernel version, required modules, and capabilities for DMA-BUF GPU memory export on each vendor. Test on kernel 5.15 (our node192) and 6.x. Compatibility matrix for DMA-BUF GPU IPC across kernel versions.

Success criteria: DMA-BUF FD passing achieves ≥95% of native cuMem IPC bandwidth on NVIDIA A100 across containers. If successful, this validates DMA-BUF as the universal substrate for XTrans and significantly de-risks the AMD/Intel phases.

Why this is a potential key contribution: If DMA-BUF works as a universal GPU IPC primitive across vendors, the paper’s story becomes: “We discovered that a standard Linux kernel subsystem (DMA-BUF) — originally designed for graphics buffer sharing — can serve as a universal cross-container GPU memory sharing mechanism, eliminating the need for vendor-specific IPC workarounds.” This is a genuine systems insight, not “transplanting cuMem.”

6.3 Phase 2: AMD RCCL Cross-Container Recovery (Exp B)

Exp B: RCCL Gate Discovery and Infinity Fabric P2P Recovery

Goal: Replicate the exp2 methodology on AMD hardware. Discover RCCL’s equivalent of NCCL’s three gates, and build a shim that recovers Infinity Fabric bandwidth in per-GPU containers.

Pre-research findings that refine this phase:

  • RCCL uses the same three gates as NCCL (hostHash, shmDev, pidHash) — it’s a hard fork
  • RCCL explicitly disables cuMem VMM on AMD: ncclIsCuMemSupported() returns 0 on __HIP_PLATFORM_AMD__
  • AMD has three potential FD-based IPC paths (HIP VMM, DMA-BUF, opaque HSA handles via KFD) — none tested for cross-container P2P
TaskMethodDeliverable
B1: RCCL source analysis Analyze RCCL source (forked from NCCL, diverged significantly). Identify topology detection (/sys/class/kfd, rocm_smi), transport selection, and IPC handle exchange mechanisms. Pre-research already confirmed: same fillInfo() with hostHash + shmDev, same transport priority (P2P > SHM > NET), cuMem disabled. RCCL gate taxonomy. Equivalent of our NCCL deep-dive document.
B2: Three-path IPC test Test three AMD IPC paths in priority order:
(a) Opaque HSA IPC handles (hsa_amd_ipc_memory_create/attach) across containers with shared /dev/kfd — cheapest test, may already work since KFD resolves handles kernel-side
(b) DMA-BUF export (hsa_amd_portable_export_dmabuf) + FD passing + import for GPU P2P — validates Phase 1.5 finding on AMD
(c) HIP VMM FD path (hipMemHandleTypePosixFileDescriptor) — Beta API, direct analog of NVIDIA cuMem		 Determine which AMD IPC path(s) work across containers. Bandwidth comparison for each working path vs native (shared-namespace) baseline.
B3: Baseline measurement Measure RCCL allreduce bandwidth on MI250X or MI300X in: (a) shared container, (b) per-GPU containers with default config, (c) per-GPU containers with progressive fixes using the working IPC path from B2. Bandwidth recovery curve for RCCL, analogous to our NCCL exp2 results.
B4: RCCL shim Build libxtrans_rccl.so using the same architecture as the NCCL shim. Intercept the same gates (gethostname, stat) plus any AMD-specific gates discovered in B1. Infinity Fabric P2P recovery in per-GPU containers on AMD hardware.

Hardware requirement: Access to an AMD MI250X or MI300X system. Options: Azure ND-MI300X-v5, AMD Accelerator Cloud, academic cluster access, or collaboration with AMD research.

Risk (updated from pre-research): RCCL explicitly disables cuMem on AMD, so the NVIDIA cuMem path cannot be directly transplanted. However, pre-research identified two alternative FD-based paths (DMA-BUF via hsa_amd_portable_export_dmabuf and HIP VMM via hipMemHandleTypePosixFileDescriptor) plus the possibility that opaque HSA handles already work across containers via shared /dev/kfd. If none of these work, the daemon broker (Phase 3) becomes necessary earlier — and this strengthens the paper by showing vendors require fundamentally different broker strategies.

6.4 Phase 3: XTrans Daemon and Unified Protocol (Exp C)

Exp C: Host Daemon with Vendor-Pluggable Backends

Goal: Unify the NCCL and RCCL shims into a single system with a host daemon that provides topology service and handle brokering, and per-container shims that communicate with the daemon via a vendor-neutral protocol.

ComponentDesignKey Challenge
C1: Daemon topology service Host daemon enumerates all GPUs (NVIDIA via NVML, AMD via rocm_smi, Intel via Level Zero) and interconnects. Builds a unified topology graph. Serves topology queries via UDS. Unified topology representation across vendors. NVLink mesh vs xGMI mesh vs Xe Link star have different structures.
C2: Handle broker protocol Define a simple protocol for handle exchange:
EXPORT(container_id, gpu_id, handle_type, handle_data) → handle_id
IMPORT(handle_id) → handle_data
For FD-based handles (cuMem): use SCM_RIGHTS relay.
For opaque handles (HSA IPC, L0 IPC): use SHM segment relay.		 FD passing requires the daemon to receive and re-send FDs. Latency of the relay must be <1ms to not affect NCCL init time.
C3: Vendor backend plugins Each vendor backend implements:
discover_gpus() → [GpuInfo]
discover_links(gpu_a, gpu_b) → LinkInfo
export_handle(gpu_id, alloc) → Handle
import_handle(handle) → mapping		 Abstracting vendor-specific GPU memory allocation and IPC mechanisms behind a common interface without losing performance.
C4: Unified shim library Single libxtrans.so that detects which CCL is loaded (NCCL vs RCCL vs oneCCL) and applies the appropriate interceptions. Communicates with daemon for topology and handle exchange. Detecting CCL type at runtime. Handling version differences within each CCL.

XTrans is strictly better than current practice.
Performance overhead of daemon relay Low Confirmed low. Handle exchange happens once during NCCL init, not on the data path. LD_PRELOAD overhead is ~1–5ns per call. Steady-state communication goes directly over NVLink/xGMI with zero daemon involvement.
Paper seen as “just engineering” High Key risk — requires careful framing. Pre-research identified four elevation strategies (Section 2.6): (1) cross-vendor failure taxonomy, (2) DMA-BUF as universal GPU IPC substrate, (3) GPU isolation spectrum model, (4) minimal interception surface completeness argument. Frame around the understanding of why GPU communication breaks universally, not the daemon. Precedent: “Simple Testing” (OSDI’14) was accepted for discovering a universal failure pattern across diverse systems.
NVIDIA closes the gap — IMEX expansion or NCCL container-aware transport Low De-risked. Pre-research confirms: IMEX is GB200-only with no expansion plans. NCCL has no intra-node transport plugin interface. NVIDIA’s container strategy focuses on stronger isolation (Confidential Containers), not easier sharing. Even if NVIDIA expands IMEX, it remains single-vendor and proprietary.
Scope creep Medium Unchanged. Paper B requires only NVIDIA + AMD (or NVIDIA + Intel if AMD hardware unavailable). Phase 1.5 DMA-BUF spike provides early go/no-go signal. Three fallback directions defined (Section 2.8) if full XTrans scope proves infeasible.
