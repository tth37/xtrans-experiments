Literature Study: Container Techniques in
Top Systems Conferences (2020–2026)

Companion to the XTrans Research Plan · 2026-04-09

1. Overview

Scope: This survey covers container-related systems research published at OSDI, SOSP, EuroSys, ASPLOS, USENIX ATC, NSDI, and select adjacent venues (SoCC, FAST, ISCA, SIGCOMM, SC) from 2020 to 2026. We focus on techniques directly relevant to the XTrans research agenda: GPU isolation and sharing in containers, cross-container communication, container runtimes and security, serverless GPU computing, checkpoint/restore and live migration, resource disaggregation, and GPU cluster scheduling.

Containers have become the dominant deployment abstraction for cloud-native workloads. Over the past six years, top systems conferences have produced a rich body of work addressing the fundamental tension: containers provide isolation and manageability, but isolation breaks the shared-memory and shared-namespace assumptions that high-performance GPU communication depends on.

We organize this literature into seven research directions, each representing a distinct thread of investigation with its own community, techniques, and open problems:

  1. GPU Sharing and Virtualization in Containers — multiplexing GPU resources across tenants
  2. GPU Cluster Scheduling and Orchestration — topology-aware, elastic, and fair allocation
  3. Container Runtime, Isolation, and Security — microVMs, sandboxes, and the isolation–performance trade-off
  4. Serverless Computing and Cold Start Optimization — fast container provisioning for GPU workloads
  5. GPU Checkpoint/Restore and Live Migration — transparent GPU process mobility
  6. Resource Disaggregation — decoupling memory and accelerators from compute nodes
  7. LLM Serving Infrastructure — container-based systems for large model inference

2. GPU Sharing and Virtualization in Containers

The standard practice of binding entire GPUs to individual containers leads to severe underutilization—production clusters at Alibaba, Microsoft, and SenseTime report only 25–50% GPU utilization. This has motivated extensive work on spatial sharing (partitioning SMs and memory), temporal sharing (time-slicing), and spatio-temporal sharing (combining both dimensions).

2.2 Techniques and Trade-offs

GPU sharing approaches fall into three categories of interposition:

  • API-level interception: Systems like GaiaGPU and gShare intercept CUDA Driver APIs via LD_LIBRARY_PATH or LD_PRELOAD, managing GPU memory allocation and kernel launches transparently. This is the most portable approach but provides coarse control.
  • Kernel-level scheduling: Orion and Paella intercept and schedule individual GPU kernels, achieving fine-grained interference avoidance. This requires deeper integration with the ML framework (e.g., PyTorch) but delivers superior isolation.
  • Hardware partitioning: NVIDIA MIG (Multi-Instance GPU) provides hardware-enforced isolation but at coarse granularity (up to 7 instances on A100/H100). AMD SR-IOV offers up to 16 virtual partitions per GPU.

Gap relevant to XTrans: All existing GPU sharing systems assume either a single container or co-located containers sharing a GPU. None addresses the orthogonal problem of high-bandwidth inter-container GPU communication when GPUs are in separate containers on the same node. XTrans fills this gap.

3.2 Evolution of Scheduling Approaches

The field has evolved through several generations:

  1. 2020: Topology and fairness awareness — HiveD introduced topology-aware guarantees; Themis and Gavel addressed fairness in heterogeneous clusters.
  2. 2021–2022: Co-adaptive and elastic scheduling — Pollux pioneered joint resource–hyperparameter optimization; Synergy expanded the resource model beyond GPUs alone.
  3. 2023–2024: Heterogeneity and scale — Sia solved the scaling problem that plagued Pollux on heterogeneous clusters; MAST tackled geo-distributed scheduling; AlpaServe brought scheduling innovations to inference.
  4. 2025+: Disaggregation and decomposition — DeDe and emerging work decompose the monolithic scheduling problem for hyperscale clusters.

Relevance to XTrans: Topology-aware schedulers like HiveD and CASSINI assume that GPUs within a topology group share a container or VM. XTrans relaxes this assumption: per-GPU containers can be independently scheduled while still achieving high-bandwidth interconnect communication. This opens new scheduling possibilities (e.g., fine-grained GPU-level preemption and migration).

5.2 Optimization Strategies

Cold start mitigation techniques form a hierarchy:

  1. Container image optimization: On-demand loading (AWS Lambda), layer sharing (RainbowCake), lazy pulling (containerd stargz).
  2. Process-level snapshotting: CRIU-based checkpoint/restore, fork-based cloning (REAP, Catalyzer). Avoids full container creation for warm starts.
  3. GPU-specific optimization: GPU context pooling (PhoenixOS), CUDA library remoting (Torpor), pipelined context switching (PipeSwitch).
  4. Model-level optimization: Multi-tier checkpoint loading (ServerlessLLM), selective state restoration (gCROP).

6. GPU Checkpoint/Restore and Live Migration

GPU checkpoint/restore (C/R) enables fault tolerance, live migration, and preemption—all critical for per-GPU containerized deployments. This is one of the most technically challenging areas due to opaque GPU driver state and lack of OS-level introspection.

6.1 Key Papers

PaperVenueYearKey Contribution
PhoenixOS (PhOS) (SJTU-IPADS) SOSP 2025 First concurrent OS-level GPU C/R system using validated speculation; probes GPU memory via kernel argument analysis + binary instrumentation; orders of magnitude faster than cuda-checkpoint
Singularity (Microsoft) arXiv 2022 Planet-scale preemptive and elastic AI scheduling; all jobs are preemptable, migratable, and dynamically resizable by default; transparent to user code
Oobleck (Jang et al.) SOSP 2023 Resilient distributed training using pipeline templates; tolerates f simultaneous failures via f+1 heterogeneous pipeline replicas; up to 13.9× faster recovery than Bamboo/Varuna
Tenplex (Wagenländer et al.) SOSP 2024 Dynamic parallelism via parallelizable tensor collections (PTCs); enables changing multi-dimensional parallelism at runtime for elastic scaling, redeployment, and failure recovery
ReCycle (various) SOSP 2024 Resilient DNN training via pipeline adaptation; reconfigures pipeline parallelism without full restart after node failures
gCROP (various) SoCC 2024 On-demand and parallel GPU checkpoint/restore; achieves <100ms startup for GPU apps with up to 774M parameters (GPT-2 Large)

6.2 Technical Challenges

GPU C/R is fundamentally harder than CPU C/R for several reasons:

  • No dirty bits: GPUs lack the page-table dirty bit mechanism that enables efficient incremental checkpointing on CPUs. PhoenixOS addresses this through speculative memory access tracking based on kernel launch arguments.
  • Opaque driver state: GPU driver contexts, command queues, and page tables are managed by proprietary drivers (NVIDIA, AMD) with no standardized introspection API.
  • Large state size: GPU memory (80 GB on A100) dwarfs typical CPU process state, making full-state checkpoints expensive. Concurrent C/R (PhOS) and selective restore (gCROP) address this.
  • Multi-GPU coherence: Distributed training state spans multiple GPUs with NCCL/RCCL communication state. Oobleck and Tenplex sidestep this by operating at the training framework level rather than the OS level.

Key insight for XTrans: PhoenixOS demonstrates that OS-level GPU C/R is feasible and performant. Combined with XTrans’s cross-container GPU IPC, this enables true per-GPU container mobility: individual GPUs can be checkpointed, migrated, and restored independently while maintaining high-bandwidth communication with peer containers. This is a powerful combination that neither system achieves alone.

7.3 CXL: The Emerging Interconnect

CXL (Compute Express Link) is rapidly becoming the key enabler for hardware disaggregation:

  • CXL 1.x: Memory expansion — attach more memory to a single server at ~150–250ns latency
  • CXL 2.0: Memory pooling — shared memory pool across servers via CXL switches
  • CXL 3.0: Memory sharing — cache-coherent shared memory across nodes

Key systems papers: DirectCXL (ATC 2022), Pond (ASPLOS 2023), TPP (ASPLOS 2023), CXL-ANNS (ATC 2023), CXLfork (OSDI 2025).

Relevance to XTrans: GPU disaggregation (Prism, DistServe) separates GPUs from CPUs across the network. XTrans addresses the complementary intra-node problem: GPUs that are physically on the same node but isolated in different containers. CXL’s “My CXL Pool Obviates Your PCIe Switch” (HotOS 2025) hints at a future where CXL-attached GPUs could use the same DMA-BUF FD passing that XTrans explores for cross-container IPC.

8.2 Cross-Vendor GPU Communication

An emerging thread directly relevant to XTrans:

PaperVenueYearKey Contribution
HetCCL (Kim et al.) arXiv 2026 Cross-vendor collective communication library enabling NVIDIA + AMD GPU communication via RDMA without driver modifications; unified vendor abstraction over NCCL/RCCL with IB Verbs API for GPU memory registration

HetCCL validates a key XTrans assumption: cross-vendor GPU communication is achievable via standardized transport mechanisms. While HetCCL uses RDMA for inter-node communication, XTrans addresses the intra-node case using DMA-BUF FD passing—the two are complementary.

8.3 Container Implications

LLM serving systems face a container dilemma: tensor parallelism across GPUs requires high-bandwidth NCCL communication, which assumes shared namespaces. Current deployments handle this in three ways:

  • All-GPUs-in-one-container: The common approach (used by vLLM, TGI). Simple but coarse-grained: cannot independently scale, migrate, or upgrade individual GPUs.
  • Process-per-GPU with shared namespaces: Uses hostIPC, hostNetwork, and shared /dev/shm in Kubernetes. Functionally works but defeats container isolation.
  • Per-GPU containers with XTrans: (proposed) Maintain true isolation while recovering high-bandwidth interconnect communication. Enables GPU-granular operations (preemption, migration, scaling) without sacrificing NCCL performance.

9. Cross-Cutting Themes and Research Directions

9.1 The Isolation–Performance Fundamental Trade-off

Across all seven research areas, a unifying theme emerges: stronger isolation degrades communication performance. This trade-off manifests differently at each level:

9.2 Emerging Research Directions

Based on this literature survey, we identify the following under-explored directions:

Direction A: Cross-Container GPU IPC (the XTrans problem)

Gap: No existing work addresses fabric-agnostic, high-bandwidth GPU communication across container boundaries. All GPU sharing papers assume co-location within a single container. All container isolation papers ignore GPU IPC entirely.

Opportunity: DMA-BUF file descriptors provide a vendor-neutral handle that can be passed via UDS SCM_RIGHTS across containers. This has been validated on NVIDIA (our exp2 results) but not systematically studied across AMD and Intel.

Direction B: GPU-Aware Secure Container Runtimes

Gap: Firecracker, gVisor, and Kata have no GPU support. NVIDIA Container Toolkit works only with runc, providing no hardware isolation for GPU memory.

Opportunity: Combining VFIO-based GPU passthrough with microVM isolation, or extending gVisor’s syscall interposition to GPU driver ioctls.

Direction C: Per-GPU Container Migration and Elasticity

Gap: PhoenixOS enables GPU C/R but assumes a monolithic process. Tenplex enables elastic parallelism but operates at the framework level. Neither supports transparent per-GPU container migration.

Opportunity: Combine XTrans (cross-container IPC) + PhoenixOS (GPU C/R) to enable truly transparent, GPU-granular live migration in containerized clusters.

Direction D: CXL-Enabled GPU Memory Pooling Across Containers

Gap: CXL memory pooling (Pond, DirectCXL) has been studied for CPU memory but not for GPU memory across containers.

Opportunity: As CXL 3.0 enables cache-coherent shared memory, GPU memory pools could be shared across per-GPU containers without the NCCL namespace assumptions.

Direction E: Topology-Aware Scheduling for Per-GPU Containers

Gap: Schedulers like HiveD and CASSINI are topology-aware but treat containers as multi-GPU units. Per-GPU containers enable finer scheduling granularity.

Opportunity: A scheduler that understands both the physical interconnect topology (NVLink, PCIe, CXL) and the container topology (which containers share which broker) could make better placement decisions for distributed training.

12. Conclusions

Key finding: Despite 84 papers on container-related GPU systems in the past six years, no published work addresses cross-container GPU communication as a first-class research problem. The closest related work (TGS, Shimmy, C-GDR) addresses intra-container GPU sharing or inter-container CPU communication, but not high-bandwidth GPU-to-GPU IPC across container boundaries.

Our survey yields the following conclusions:

  1. GPU sharing is mature but orthogonal to XTrans. Systems like TGS, Orion, and Paella multiplex one GPU across containers. XTrans enables multiple GPUs in separate containers to communicate at full interconnect bandwidth. These are complementary.
  2. GPU C/R is now feasible. PhoenixOS (SOSP ’25) demonstrates concurrent, OS-level GPU checkpoint/restore with production-grade performance. This is the missing piece for per-GPU container migration—combined with XTrans’s cross-container IPC, it enables true GPU-granular elasticity.
  3. The scheduling community assumes coarse GPU allocation. All major schedulers (HiveD, Pollux, Sia, MAST) allocate GPUs in groups to a single job container. Per-GPU containers with XTrans would enable a fundamentally finer scheduling granularity.
  4. Memory disaggregation is converging on CXL. The RDMA-based disaggregated memory era (Clio, Hermit, FineMem) is transitioning to CXL (Pond, DirectCXL, CXLfork). GPU memory disaggregation via CXL is a natural next step but unexplored.
  5. LLM serving drives urgent demand. vLLM, DistServe, and ServerlessLLM all use multi-GPU tensor parallelism that breaks under container isolation. XTrans directly addresses this pain point.
  6. The isolation–performance trade-off is the fundamental challenge. Every paper in this survey navigates this trade-off differently. XTrans proposes a principled solution: maintain container-level isolation (namespaces, cgroups) while providing a minimal, well-defined broker for GPU IPC handle exchange.

Positioning XTrans in the literature: XTrans sits at the intersection of three research threads—container isolation (§4), GPU communication (§2), and disaggregation (§7)—addressing a gap that none of the 53 surveyed papers fills. The closest related work is Shimmy (HotEdge ’19, shared-memory inter-container communication for CPUs) and C-GDR (container-aware GPUDirect MPI, cross-container but inter-node). No prior work combines intra-node, GPU-to-GPU, cross-container, and vendor-agnostic into a single system.
