Research Plan v4: GPU-Granular Scheduling Across Container Boundaries

2026-04-15 (updated 2026-04-16 with Exp A1 findings)

What this document is: A self-contained research plan for studying what happens when GPU scheduling techniques — designed and validated on bare metal — meet the containerized production environments where GPU workloads increasingly run. We select concrete, open-source scheduling systems as baselines and progressively containerize them, observing what challenges emerge at each stage.

1. Motivation

Three independent trends are converging. Each is well-evidenced and individually uncontroversial. Their intersection creates an under-explored problem space.

1.1 GPU Scheduling Got Finer

Over six years, GPU cluster scheduling research progressed from coarse to fine granularity:

EraGranularityKey SystemsWhat Changed
2020 Node-level HiveD (OSDI), Gavel (OSDI), Themis (NSDI) Whole nodes allocated to jobs; topology affinity within cells
2021–22 GPU-group Pollux (OSDI), Synergy (OSDI), AntMan (OSDI) Dynamically adjust GPU count per job; co-adaptive batch size tuning
2023–24 Per-GPU Sia (SOSP), CASSINI (NSDI), Crux (SIGCOMM) Per-GPU allocation with topology and communication awareness
2024–25 Per-GPU elastic Tenplex (SOSP ’24), NTP (arXiv ’25), Kale (SoCC ’24) Add/remove individual GPUs from running jobs at runtime

Two systems exemplify the frontier:

  • Tenplex (SOSP ’24): Changes tensor, pipeline, and data parallelism dimensions via Parallelizable Tensor Collections (PTCs). Described as “without restart,” but Exp A1 revealed this is stop-transform-restart: the training process fully terminates, state is transformed on the host, and a new process starts. Evaluated on up to 64 GPUs with ~122s reconfiguration for a 16→8 resize.
  • NTP (arXiv ’25): When a GPU fails mid-training in a tensor-parallel group of 8, the remaining 7 GPUs continue with non-uniform tensor parallelism — redistributing the failed GPU’s workload across survivors without checkpoint-and-restart.

Critical assumption: Both Tenplex and NTP — along with every other system in the table — assume bare metal. All GPUs see each other through shared OS namespaces. The collective communication library (NCCL) discovers NVLink topology and communicates at full interconnect bandwidth without any container boundary in the way.

1.2 Hardware and Operations Demand Containers

Hardware trajectory. NVLink domains are growing from 8 GPUs (DGX A100) to 72 (GB200 NVL72) to 576 (NVL576). You cannot give 72 GPUs to every job. Multiple jobs must share the NVLink domain. NVIDIA acknowledges this: their ComputeDomains RFC (Nov 2025) envisions “many single-GPU pods” where “the unit of interest is the individual GPU, and not the individual node.” But today, ComputeDomains have severe limitations: one pod per node, one ComputeDomain per node — falling far short of this vision.

Operational reality. Kubernetes has 92% market share in orchestration. AI/ML workloads on K8s at 54% and growing (CNCF survey: 90%+ expect increase). K8s Dynamic Resource Allocation (DRA, GA in K8s 1.32) enables per-GPU device allocation. NVIDIA’s KAI Scheduler (open-sourced 2025, CNCF Sandbox) provides topology-aware GPU scheduling, fractional allocation, and bin-packing. The orchestration layer is ready for per-GPU scheduling.

Isolation requirements. GPU failure rates of 9% annualized (Meta Llama 3.1, 16K H100s) demand fault isolation. Multi-tenancy on shared GPU nodes demands security isolation. Both require container boundaries.

1.3 The Container Wall Breaks GPU-Granular Scheduling

When containers meet GPU-granular scheduling, a fundamental tension appears.

Today, GPU workloads in production run in monolithic multi-GPU containers: one container holds all GPUs for a job. This works for NCCL — all GPUs share namespaces, NVLink is discovered, full bandwidth is achieved. But monolithic containers are incompatible with fine-grained scheduling:

  • Elasticity is trapped (for in-place systems): If a framework scales from 4 to 2 GPUs while training continues running, the container still holds 4 GPUs. K8s cannot reclaim the freed GPUs for other jobs (In-Place Pod Resize does not support GPU resources). Note: for stop-restart systems like Tenplex, this is solvable by simply creating the new container with fewer GPUs. The trap is only fundamental when the training process survives the scaling event.
  • Failures are amplified: One GPU failure inside an 8-GPU container may crash the entire container, killing all 8 GPUs. Even if the framework recovers internally (Oobleck, NTP), the failed GPU remains allocated to the container. K8s cannot replace it without restarting the entire pod.
  • Bin-packing is coarse: If Job A needs 3 GPUs and Job B needs 5 GPUs on an 8-GPU node, they must be in separate containers for isolation. But each container sees only its own GPUs — cross-container NVLink communication is not supported.

The natural solution is per-GPU containers: each GPU in its own container, independently schedulable. Scaling down means stopping containers (GPUs return to the cluster). GPU failure kills only one container. K8s manages GPU lifecycle with standard primitives (HPA, health checks, rolling updates).

But per-GPU containers introduce their own problems. NCCL assumes all GPUs on a node share OS namespaces. When they don’t, NCCL falls back from NVLink to TCP — a 2–10× bandwidth loss. The scheduling operation that prompted the containerization now faces degraded or broken communication.

The three-act gap: GPU scheduling got finer (Act 1). Production demands containers (Act 2). The container wall breaks GPU-granular scheduling (Act 3). No system today bridges this gap. The scheduling community assumes bare metal; the container community does not consider GPU collectives. We study the intersection.

The gap: Every scheduling system assumes either bare metal or all GPUs in one container. NVIDIA’s ComputeDomains is the only system that addresses cross-container GPU communication, but it is limited to GB200 hardware, one pod per node, and proprietary. Nobody has studied what happens when general-purpose GPU scheduling techniques are deployed in standard containerized environments.

3. Research Methodology: Progressive Containerization

3.1 Philosophy

We do not pre-imagine what challenges exist. Instead, we take existing, working, bare-metal GPU scheduling techniques and progressively move them into containerized environments. The challenges reveal themselves through this process.

This is a principled empirical methodology:

  • The baseline techniques are published, peer-reviewed, and validated on bare metal. Any failure in containers is a genuine finding about the container–scheduling gap, not a bug in our setup.
  • The challenges we discover are ground truth, not speculation. They come with concrete failure modes, error messages, and performance measurements.
  • Starting from the obvious production approach (multi-GPU containers) before exploring alternatives (per-GPU containers) ensures that each step is motivated by the limitations of the previous step.

3.2 Three-Phase Experiment Structure

Each experiment follows the same three-phase progression:

PhaseSetupPurpose
Phase 1:
Bare Metal		 Run the scheduling technique on bare metal, exactly as designed. All GPUs visible in one process group, shared namespaces. Confirm the technique works as documented on our hardware. Establish baseline performance metrics.
Phase 2:
Multi-GPU Container		 Put all GPUs into a single container — the standard production approach today. Run the same scheduling technique inside. Test the obvious containerization approach. Observe whether the scheduling operation provides its intended benefit at the cluster level (not just framework level). Record any limitations that arise from the monolithic container model.
Phase 3:
Per-GPU Containers		 Each GPU in its own isolated container. Containers communicate over a Docker bridge network. Only if Phase 2 reveals limitations that per-GPU containers would address. Attempt the same scheduling operation. Record all new challenges that arise from container boundaries. Attempt workarounds.

Key principle: Phase 3 is motivated by Phase 2, not assumed a priori. If multi-GPU containers work perfectly for a given scheduling technique, there is no reason to try per-GPU containers. The experiments discover whether and where the monolithic container model breaks down, and what new challenges arise when you move to finer granularity.

3.3 What We Record at Each Phase

  • Does the scheduling operation succeed? (e.g., does elastic resize complete? does fault recovery restore training?)
  • Does the benefit reach the cluster level? (e.g., are freed GPUs actually available to other jobs? can K8s replace a failed GPU?)
  • Performance metrics: training throughput, reconfiguration/recovery time, communication bandwidth.
  • Failure modes: error messages, stack traces, NCCL logs, container events. We record everything without filtering — the unexpected observations are often the most valuable.
  • Workaround results: for each challenge in Phase 3, what workarounds were attempted, which succeeded, and at what cost (performance, isolation, complexity).

4. Baseline Technique Selection

4.1 Selection Criteria

We select scheduling techniques that are:

  1. Open-source with available, runnable code (not paper-only).
  2. Valuable: implements a scheduling operation broadly needed in production (elasticity, fault tolerance, etc.).
  3. Bare-metal only: designed without considering containerized environments, so the migration genuinely tests what happens.
  4. Feasible on our hardware: can run on a 4×A100 node.
  5. Challenging in containers: the scheduling operation involves dynamic changes to the GPU group — exactly where containers create friction.

4.2 Candidate Survey

Exp A1 finding (Tenplex): Tenplex’s elasticity is stop-restart, not in-place — training fully stops at each reconfiguration. Multi-GPU containers handle stop-restart elasticity adequately (just restart with a different GPU count). Per-GPU containers are only motivated by in-place elasticity, where training processes survive GPU allocation changes and Docker’s immutable --gpus flag becomes a genuine barrier. This insight reshapes candidate selection: we now distinguish stop-restart vs. in-place systems.

The experiments now cover three distinct motivations for per-GPU containers:

  • Elasticity (stop-restart) (Exp A1, done): Multi-GPU containers suffice. Per-GPU containers not needed. This is a negative result that sharpens the argument.
  • Fault isolation (Exp A2): One GPU failure in a multi-GPU container may kill all GPUs. Per-GPU containers limit blast radius.
  • In-place elasticity (Exp A3, new): Training processes survive GPU allocation changes. Per-GPU containers are the only way to express this in Docker/K8s, because running containers cannot change GPU assignments.

Why MoE/Expert Parallelism is the strongest direction for Exp A3: Experts are naturally independent computation units. Each expert maps to a GPU (or container). Adding/removing experts is a natural elastic operation. The “Expert-as-a-Service” paradigm literally disaggregates the model into independently deployable services — which is exactly what per-GPU containers provide. The strongest candidate is vLLM’s Elastic EP (RFC): open source, production-deployed, in-place scaling on real MoE models (DeepSeek-V3/R1, Qwen3.5, Llama 4). If vLLM’s Elastic EP currently requires all EP workers in the same process group (one container), demonstrating it across per-GPU containers would be a direct proof-of-concept for the per-GPU container value proposition.

6. Exp A1: Elastic GPU Scaling (Tenplex)

6.1 Technique Overview

Tenplex (SOSP ’24) enables elastic GPU scaling for distributed training. Its core innovation is Parallelizable Tensor Collections (PTCs): a data structure that represents model and optimizer state in a way that can be repartitioned across different parallelism configurations at runtime. When the GPU count changes, Tenplex recomputes the parallelism strategy (TP, PP, DP dimensions) and redistributes state across the new set of GPUs.

AspectDetail
Code github.com/kungfu-team/tenplex (Go + Python, Megatron-LM integration)
Capability Change TP, PP, DP dimensions at runtime. Repartition model/optimizer state. GPU add, remove, and redeployment without restart.
Performance ~122s reconfiguration for 16→8 GPU resize (dominated by state I/O)
Deployment assumption Bare metal. All GPUs in shared namespace. NCCL communicators created freely across all GPUs.

7. Exp A2: Fault-Tolerant Training (Oobleck)

7.1 Technique Overview

Oobleck (SOSP ’23) enables resilient distributed training that tolerates GPU failures without full checkpoint-and-restart. Its core innovation is pipeline templates: pre-computed pipeline parallelism configurations for different numbers of surviving GPUs. When a GPU fails, Oobleck selects the appropriate template and redistributes pipeline stages to the remaining GPUs.

AspectDetail
Code github.com/SymbioticLab/Oobleck (Python + Rust, Colossal-AI / PyTorch)
Capability Tolerate f simultaneous GPU failures using f+1 heterogeneous pipeline replicas. Up to 13.9× faster recovery than Bamboo/Varuna.
Recovery mechanism Pre-computed templates → detect failure → select template → redistribute stages → resume training
Deployment assumption Bare metal. Detects failures via process exit / NCCL timeout. All GPUs in shared namespace.

8. Exp A3: In-Place Elastic Expert Parallelism (vLLM)

8.1 Technique Overview

vLLM Elastic EP (shipped in vLLM v0.17, 2026) enables live rescaling of expert parallelism for Mixture-of-Experts (MoE) models. When GPU resources change — added during a traffic spike, removed when utilization drops — vLLM brings up or tears down engine-cores, reinitializes EP/DP/TP/PP communicators, and rebalances expert loads across the new set of GPUs. This happens in-place: the serving process survives the scaling event without restart.

AspectDetail
Code github.com/vllm-project/vllm (Python, v0.17+); RFC #20323
Capability In-place scale-up/down of expert parallel workers. Reinitialize NCCL communicators. Rebalance expert placement via EPLB algorithms. Zero-downtime for live serving.
Models DeepSeek-V3/R1, Qwen3.5-MoE, Llama 4 Maverick, and other MoE architectures
Deployment assumption All EP workers in the same process group, typically within a single container or set of containers with shared network. NCCL All-to-All for expert dispatch.

Why this experiment matters: Exp A1 showed that stop-restart elastic systems (Tenplex) do not need per-GPU containers. vLLM Elastic EP is in-place: the serving process keeps running while GPUs are added or removed. Under the multi-GPU container model, Docker’s immutable --gpus flag prevents the running container from releasing or acquiring GPUs. Per-GPU containers resolve this: each expert worker lives in its own container, and scaling is expressed as container lifecycle. This experiment directly tests whether per-GPU containers enable in-place elasticity that multi-GPU containers cannot.

8.2 Why MoE + Expert Parallelism Is the Ideal Test Case

Experts are naturally independent computation units that map directly to containers:

  • Each expert ≈ one GPU ≈ one container. Unlike tensor parallelism (which shards a single tensor across GPUs, requiring tight coupling), expert parallelism routes tokens to independent expert modules on separate GPUs. Each expert is a self-contained computation unit.
  • Scaling = adding/removing experts. Scale-up means starting a new container with a new expert. Scale-down means stopping a container. The remaining experts continue serving without interruption.
  • Fault isolation is natural. If one expert’s GPU fails, only that container dies. Other experts continue serving (with load rebalancing).
  • The “Expert-as-a-Service” paradigm (arXiv ’25) already proposes disaggregating experts into independent services — which is precisely what per-GPU containers provide at the infrastructure level.

8.3 Phase 1: vLLM Elastic EP Native (Baseline)

Run vLLM with a MoE model using expert parallelism on 4 GPUs. Confirm that live rescaling works as designed.

StepActionRecord
1 Start vLLM with a MoE model (e.g., Qwen3.5-MoE or a smaller MoE model that fits on 4×A100). Configure EP across 4 GPUs. Serving throughput (tokens/s), latency, expert placement, NCCL transport for All-to-All.
2 Trigger live scale-down: 4 → 2 EP workers Rescaling time, communicator reinitialization, expert rebalancing, serving continuity (are in-flight requests dropped?).
3 Trigger live scale-up: 2 → 4 EP workers New worker join time, expert redistribution, throughput recovery.
4 Record NCCL communicator lifecycle How are communicators destroyed and recreated? What is the All-to-All downtime during rescaling?

8.4 Phase 2: Multi-GPU Container — The In-Place Barrier

Run vLLM Elastic EP inside a single container with all 4 GPUs. This is where in-place elasticity hits the container wall.

  +-----------------------------------------------+
  |     Single Container (4 GPUs, --gpus all)      |
  |  Expert0  Expert1  Expert2  Expert3            |
  |  (GPU 0)  (GPU 1)  (GPU 2)  (GPU 3)           |
  |                                                |
  |  vLLM scales 4 → 2 EP workers...              |
  |  Expert0  Expert1  [idle]   [idle]             |
  |  (GPU 0)  (GPU 1)  (GPU 2)  (GPU 3)           |
  |                                                |
  |  GPU 2, 3: idle but TRAPPED in container       |
  +-----------------------------------------------+
StepActionObserve
1 Start vLLM in a single container with all 4 GPUs Does Elastic EP work inside a container? Performance vs. Phase 1?
2 Trigger live scale-down: 4 → 2 EP workers vLLM successfully scales internally — but can the cluster reclaim GPUs 2 and 3? Docker cannot remove GPUs from a running container. Unlike Tenplex (stop-restart), the process is still running — you cannot recreate the container with fewer GPUs without killing the service.
3 From outside: attempt to use the “freed” GPUs GPUs trapped. This is the fundamental limitation for in-place systems (unlike Tenplex where it was merely an engineering detail).
4 Trigger live scale-up: attempt to add GPUs from outside Docker cannot hot-add GPUs to a running container. Scale-up is impossible without restarting — negating the in-place benefit.

The critical difference from Exp A1: Tenplex’s GPU trapping was a fixable engineering detail because Tenplex stops the process anyway. For vLLM Elastic EP, the process stays running. There is no opportunity to recreate the container with different GPUs. The multi-GPU container model fundamentally prevents in-place elasticity from reaching the cluster level.

8.5 Phase 3: Per-GPU Containers — Enabling In-Place Elasticity

Each EP worker in its own container. Scaling = container lifecycle. This is the direct proof-of-concept for the per-GPU container value proposition.

  Start (4 EP workers):                Scale down (4 → 2):
  +----+ +----+ +----+ +----+          +----+ +----+
  | C0 | | C1 | | C2 | | C3 |          | C0 | | C1 |    C2, C3 stopped
  | E0 | | E1 | | E2 | | E3 |          | E0 | | E1 |    GPU 2,3 FREE
  |GPU0| |GPU1| |GPU2| |GPU3|          |GPU0| |GPU1|    (cluster-reclaimable)
  +----+ +----+ +----+ +----+          +----+ +----+

  Scale up (2 → 4):
  +----+ +----+ +----+ +----+
  | C0 | | C1 | |C2' | |C3' |          C2', C3' = new containers
  | E0 | | E1 | | E2 | | E3 |          New experts loaded
  |GPU0| |GPU1| |GPU2| |GPU3|          Communicators re-formed
  +----+ +----+ +----+ +----+
StepActionObserve
1 Start 4 per-GPU containers, each running one EP worker. Establish cross-container NCCL All-to-All for expert dispatch. Does expert parallelism work across container boundaries? NCCL transport (NVLink P2P or TCP fallback)? Throughput vs. Phase 1 and Phase 2? Is the LD_PRELOAD shim (common/shim/) needed?
2 Trigger live scale-down: stop containers C2 and C3 Do surviving containers (C0, C1) continue serving? Can vLLM’s Elastic EP reinitialize communicators with only 2 containers? Are GPUs 2, 3 immediately free at the cluster level? Serving latency during rescaling?
3 Trigger live scale-up: start new containers C2′, C3′ Can new EP workers join the existing serving group? Can NCCL All-to-All communicator expand to include new containers? Expert rebalancing across old + new containers?
4 Fault tolerance: kill container C3 unexpectedly Do surviving containers detect the failure? Can vLLM continue serving with 3 EP workers? Is the failed GPU immediately reclaimable? Contrast with multi-GPU container (all 4 GPUs lost).

8.6 Challenges to Investigate

Per-GPU containers introduce specific challenges for cross-container expert dispatch:

ChallengeDescriptionPotential Workaround
NCCL All-to-All across containers Expert dispatch uses All-to-All to route tokens to the correct expert. NCCL may fall back to TCP when GPUs are in separate containers (different network/IPC namespaces). LD_PRELOAD shim (common/shim/) to make NCCL’s namespace checks pass; or use --network host with relaxed isolation.
Dynamic communicator creation vLLM creates NCCL communicators at initialization. Adding/removing containers requires creating new communicators spanning a different set of processes across containers. Investigate vLLM’s communicator reinitialization path. May need framework patches to support cross-container NCCL init.
Expert state migration When experts are redistributed during scaling, expert weights must move between containers (potentially across GPU memory boundaries that cross container namespaces). Shared volume for expert weights; or P2P GPU memory transfer with IPC handle passing across containers.
Service discovery New EP worker containers must discover existing workers and vice versa. Standard container orchestration (Docker Compose, K8s DNS) should handle this. Docker Compose service names; environment variables for peer addresses.

9. Decision Points and Extensions

9.1 After Phase 2 of Each Experiment

Phase 2 determines whether we proceed to Phase 3:

Phase 2 OutcomeImplicationNext Step
Multi-GPU containers work well The scheduling technique provides its intended benefit within standard production containers Document this as a finding (the container wall doesn’t affect this technique). Move to the next experiment.
Multi-GPU containers have limitations The framework works internally, but the container model prevents the benefit from reaching the cluster (trapped GPUs, amplified failures, etc.) Proceed to Phase 3. The limitations motivate per-GPU containers.
Multi-GPU containers break the technique The technique does not work at all in containers Investigate why. Is it a container-specific issue (fixable) or a fundamental assumption mismatch?

9.2 After Phase 3

Phase 3 OutcomeWhat It MeansNext Step
Works with minor workarounds The container wall is thinner than expected for this operation Document exactly which workarounds are needed. Expand to more techniques (Bamboo, Pollux). Begin cross-vendor testing.
Works with significant effort Real but surmountable challenges exist Characterize the challenges. Generalize workarounds. Design systematic solutions.
Fails fundamentally New system-level mechanisms are needed This is the strongest research finding. Characterize exactly what mechanisms are missing. Design and build them.

Every outcome is a research finding. Phase 2 success means multi-GPU containers suffice. Phase 2 failure motivates per-GPU containers. Phase 3 success maps the feasibility frontier. Phase 3 failure reveals fundamental challenges. The experiments are designed so there is no wasted outcome.

10. Summary

Research question: What happens when GPU scheduling techniques — designed for bare metal — are deployed in containerized production environments?

Motivation. GPU scheduling evolved toward per-GPU granularity (Tenplex, NTP). Production demands containers. Nobody has studied what happens when these two realities collide.

Methodology. Progressive containerization. For each scheduling technique: bare metal (baseline) → multi-GPU container (production approach) → per-GPU containers (if motivated by limitations). Challenges are discovered empirically, not imagined.

Experiments.

  • Exp A1 (Tenplex): Stop-restart elastic scaling. Done. Found that multi-GPU containers suffice for stop-restart systems. Does not motivate per-GPU containers.
  • Exp A2 (Oobleck): Fault-tolerant training. Does framework-level recovery work when the failure unit is a container, not a process?
  • Exp A3 (vLLM Elastic EP): In-place elastic expert parallelism. The strongest test: does per-GPU-container deployment enable cluster-level elasticity that multi-GPU containers fundamentally cannot?

Design principle. Every phase is motivated by the previous. Every outcome is a finding. The experiments discover the problem space; solutions follow.
