Research Plan v2: Understanding and Recovering GPU
Communication Under Container Isolation on Kubernetes

A Research-First Reframing · 2026-04-10 (revised)

1. What Changed from v1 and Why

Problem with v1: The v1 plan reads like an engineering project: “We will build XTrans, a fabric-agnostic broker daemon.” It starts from a solution and works backwards to justify it. This makes it hard to discover unexpected challenges, and reviewers will see it as “just engineering.”

v2 approach: Root on a specific setting — multi-GPU workloads on Kubernetes that need to move from monolithic multi-GPU containers to per-GPU containers — and let the research questions and challenges emerge from studying that setting. The system (if we build one) is a consequence of the research, not its starting point.

The difference is subtle but fundamental:

2. The Setting: Multi-GPU Workloads on Kubernetes

2.1 The Starting Point: It Works Fine Today

Multi-GPU workloads (distributed training, tensor-parallel serving) work well on Kubernetes today. The standard practice is straightforward:

Current K8s practice: All GPUs for a job go into one container. A pod requests nvidia.com/gpu: 4 (or 8), all GPUs are visible inside one container, NCCL sees them all in the same namespace, NVLink P2P works perfectly. A memory-backed emptyDir mounted at /dev/shm provides the shared memory NCCL needs. No special privileges required for intra-node communication.

  Today's K8s practice (works fine):
  +-----------------------------------------------+
  |              Single Pod / Container            |
  |   nvidia.com/gpu: 4                            |
  |                                                |
  |   GPU0    GPU1    GPU2    GPU3                 |
  |   ==========NVLink mesh============           |
  |   NCCL sees all 4 GPUs, P2P works             |
  |   /dev/shm mounted as emptyDir                |
  +-----------------------------------------------+

This works. So why change it?

2.2 The Need: Why Per-GPU Containers Become Necessary

The monolithic multi-GPU container works for NCCL, but it creates a coarse-grained management unit that sacrifices key operational capabilities. Several real, pressing scenarios force a move to per-GPU containers:

Scenario A: GPU-level fault isolation

In a monolithic 8-GPU container, one GPU failure (ECC error, driver crash, thermal throttling) kills the entire job. All 8 GPUs are lost, and the workload must restart from the last checkpoint. With per-GPU containers, only the failed GPU’s container dies; the remaining 7 can continue or reconfigure.

Scale of the problem: Meta reports 5–20% of training GPUs experience failures per week at scale (OPT-175B training log). At 10,000 GPUs, that’s 500–2,000 GPU failures per week. Each failure in a monolithic container kills an 8-GPU group, wasting the other 7 healthy GPUs until restart.

Scenario B: GPU-granular scheduling and preemption

A shared K8s cluster runs multiple workloads at different priorities. A high-priority serving job arrives and needs 4 GPUs urgently. With monolithic containers, the scheduler must kill an entire 8-GPU training job to free 4 GPUs — wasting the other 4. With per-GPU containers, the scheduler preempts exactly the 4 GPUs it needs.

Industry direction: Kubernetes DRA (GA in K8s 1.34) explicitly models per-GPU-device allocation. NVIDIA’s GPU Operator and DRA driver support single-GPU-per-container assignment. Singularity (Microsoft, 2022) was designed around per-GPU preemption and migration.

Scenario C: Multi-tenant GPU sharing on one node

A cloud provider has an 8-GPU node. Tenant A needs 4 GPUs for Llama-70B serving (TP=4). Tenant B needs 4 GPUs for training. Each tenant’s GPUs must be in isolated containers (security, billing, compliance). But Tenant A’s 4 GPUs need NVLink for tensor parallelism. Inside their separate containers, NCCL cannot use NVLink.

Reality: This is how cloud GPU instances work (AWS p5, Azure ND, GCP a3). Today, providers avoid sub-node sharing entirely — each tenant gets the whole node. This wastes GPUs when a tenant needs fewer than 8.

Scenario D: Disaggregated inference (DistServe model)

LLM serving architectures increasingly separate prefill (compute-bound) from decode (memory-bound) into different containers for independent scaling (DistServe, OSDI ’24). KV cache transfer between the prefill and decode containers on the same node should use NVLink, but container isolation prevents it.

Scenario E: Rolling updates and independent GPU lifecycle

You want to update the model on GPU 0–1 while GPU 2–3 continue serving. With a monolithic container, any update requires restarting all GPUs. Per-GPU containers enable rolling updates, canary deployments, and independent GPU lifecycle management — standard K8s practices that work for every other resource type but break for multi-GPU workloads.

2.3 The Breakdown: What Happens When You Try Per-GPU Containers

  Per-GPU containers (what we need):
  +-------+ +-------+ +-------+ +-------+
  |Cont 0 | |Cont 1 | |Cont 2 | |Cont 3 |
  | GPU0  | | GPU1  | | GPU2  | | GPU3  |
  +---+---+ +---+---+ +---+---+ +---+---+
      |         |         |         |
  ====NVLink mesh (physically present)=======
  But NCCL can't use it! Falls back to TCP.
  Bandwidth drops 2-10x. Latency SLO violated.

When each GPU goes into its own container, NCCL (and RCCL, oneCCL) falls back to TCP/socket communication, bypassing the high-bandwidth interconnect entirely. This is not a configuration error — it is a fundamental consequence of how CCLs detect co-located GPUs using OS-level mechanisms (hostnames, shared memory devices, Unix domain sockets) that container namespaces isolate by design.

The core tension: Multi-GPU workloads on K8s work today only because all GPUs are packed into one container. The moment you need per-GPU containers — for fault isolation, scheduling granularity, multi-tenancy, or independent lifecycle — GPU communication breaks. The only reason K8s still uses monolithic multi-GPU containers is that nobody has solved the cross-container GPU communication problem.

2.4 The Fundamental Tension (Isolation vs. Communication)

This is a structural conflict between two properties that GPU cloud infrastructure needs simultaneously:

PropertyWhat It RequiresWho Provides It
Isolation Separate PID/IPC/network namespaces, independent lifecycle, fault boundaries Linux containers (namespaces + cgroups)
High-bandwidth GPU communication Shared memory, shared IPC namespace, same-host detection, direct P2P access GPU CCLs (NCCL, RCCL, oneCCL) over NVLink/xGMI/Xe Link

Today’s workarounds all sacrifice one for the other:

WorkaroundIsolation?Bandwidth?Problem
Monolithic multi-GPU container No (all GPUs in one container) Full NVLink Coarse-grained management, no GPU-level fault/scheduling/tenancy
hostIPC + hostNetwork No (namespaces shared with host) Full NVLink Defeats container isolation entirely
Per-GPU containers (default) Yes TCP fallback (2–10x slower) NVLink/xGMI bandwidth wasted
Per-GPU containers + ??? Yes Full NVLink This is the research problem

2.5 Why This Setting Is Research, Not Engineering

The question is not “can we build a daemon that passes file descriptors between containers?” (that is engineering). The questions are:

  • What exactly breaks? Is it the same mechanisms across all CCLs and vendors, or fundamentally different?
  • Is there a minimal fix? What is the thinnest possible intervention that recovers native bandwidth? Can we prove it is both necessary and sufficient?
  • Is there a universal primitive? Do DMA-BUF file descriptors provide a cross-vendor solution, or is each vendor irreducibly different?
  • What is the isolation–bandwidth trade-off space? Can we formalize it?

These are questions whose answers teach the community something new about how GPU communication interacts with container isolation — regardless of whether we end up building a system.

3. Research Questions

Rooting on the setting of multi-GPU K8s workloads that need per-GPU containers, the following research questions emerge naturally. Note that these are genuine questions — we do not presuppose the answers.

RQ1: What exactly breaks, and why?

Type: Measurement / characterization

When a CCL (NCCL, RCCL, oneCCL) runs inside a per-GPU container, what specific mechanisms fail? Is the failure pattern the same across vendors, or does each vendor break differently? Our preliminary exp1–2 work showed that NCCL has three “gates” (hostHash, shmDev, IPC socket), but:

  • Are there more gates we haven’t found?
  • Does RCCL (AMD) have the same gates or different ones?
  • Does oneCCL (Intel) have the same pattern?
  • Is there a universal pattern (topology → transport → IPC) that all CCLs follow, or is each vendor truly different?

Why this matters: Without a systematic understanding of what breaks, any fix is ad hoc. If we discover a universal failure pattern, that’s a research contribution in itself (cf. “Simple Testing Can Prevent Most Critical Failures,” OSDI ’14 — discovered a universal pattern across 5 systems).

RQ2: What is the minimal intervention to recover native bandwidth?

Type: Systems design / minimality argument

Given the failure mechanisms identified in RQ1, what is the smallest possible change to the container’s environment that recovers full interconnect bandwidth? Can we characterize the interception surface formally — i.e., prove that intercepting exactly N functions is necessary and sufficient?

  • Is LD_PRELOAD on 2–3 libc calls enough, or do we need more?
  • Can bind mounts replace some interceptions?
  • Does the intervention surface grow with CCL version, or is it stable?
  • What is the trade-off between intervention scope and isolation preservation?

Why this matters: A minimality result (“these N functions are necessary and sufficient”) is an Exokernel-style contribution — it defines the minimal interface between container isolation and GPU communication.

RQ3: Is there a universal cross-container GPU memory sharing primitive?

Type: Systems insight / discovery

Pre-research reveals that DMA-BUF file descriptors are available on all three major GPU platforms (NVIDIA, AMD, Intel). DMA-BUF FDs can be passed via UDS SCM_RIGHTS across any container boundary. But:

  • Does DMA-BUF import actually produce a GPU-accessible virtual address?
  • Does the imported buffer support P2P access (NVLink/xGMI) or only CPU access?
  • Is the performance equivalent to native vendor-specific IPC?
  • What kernel version, driver version, and configuration is required?

Why this matters: If DMA-BUF works as a universal GPU IPC primitive, this is a genuine systems discovery: a standard Linux kernel subsystem (designed for graphics buffer sharing) turns out to solve the cross-container GPU communication problem across all vendors. If it doesn’t work, understanding why (e.g., the import path doesn’t support P2P mapping) is equally valuable.

RQ4: What does the isolation–communication spectrum look like?

Type: Conceptual model / characterization

Between “no isolation” (--ipc=host) and “full isolation” (separate everything, TCP only), there is a spectrum. Can we formally characterize this spectrum?

  • What isolation properties does each intervention sacrifice?
  • Can we define “levels” of GPU communication isolation analogous to database isolation levels?
  • Can we build a model that predicts bandwidth recovery for a given isolation configuration?
  • What level of isolation do real workloads actually need?

Why this matters: A formal isolation–communication model would be a conceptual contribution that guides future systems design. It would let operators make informed trade-offs rather than choosing between “no isolation” and “no performance.”

4. Research Methodology: Study-First, Build-If-Needed

The experiments are structured as a study of what happens when multi-GPU K8s workloads move to per-GPU containers, not as a system build plan. Each phase answers specific research questions and informs whether the next phase is needed.

Key difference from v1: In v1, every phase was “build the next piece of XTrans.” In v2, every phase is “answer a research question about what breaks when multi-GPU workloads move to per-GPU containers on K8s.” We may end up building a system, but only if the research findings call for one.

4.1 Phase 1: NCCL Deep Characterization (RQ1 + RQ2)

“What exactly breaks when NCCL runs in per-GPU containers, and what is the minimal fix?”

Setting: node192, 4×A100 80GB with NVLink 3.0. Per-GPU Docker containers (1 GPU per container) running NCCL allreduce benchmarks.

ExperimentQuestionMethodExpected Finding
1a: Failure characterization What syscalls does NCCL make during init that are sensitive to container isolation? Run NCCL under strace in shared-namespace vs per-GPU-container configs. Diff the syscall traces. Correlate with NCCL source code to identify each “gate” — the decision point where NCCL chooses to fall back. Complete gate taxonomy. Possibly more gates than the three we know of.
1b: Bandwidth degradation curve How much bandwidth do we lose at each gate failure? Progressively enable/disable each gate. Measure allreduce bandwidth at each configuration point. Build a degradation curve: which gate matters most? Quantified per-gate impact. Understanding of which failures are catastrophic vs marginal.
1c: Minimal intervention What is the smallest change that recovers full NVLink bandwidth? Build the thinnest possible LD_PRELOAD shim (~200 LoC C). Intercept only the identified gates. Measure: does it recover ≥99% bandwidth? Which intercepted calls are truly necessary vs redundant? The minimal interception surface for NCCL. Count of intercepted functions and proof that each is necessary.
1d: Version stability Is the interception surface stable across NCCL versions? Test the shim against NCCL 2.18, 2.21, 2.25, 2.27, 2.28, 2.29. Document any gate changes. Identify which versions add, remove, or modify gates. Stability characterization. If the surface is unstable, this is a finding that argues for a different approach.
1e: Topology visibility How does NCCL discover GPU topology in containers, and what goes wrong? Analyze NCCL’s sysfs reads (/sys/bus/pci). Test NCCL_TOPO_FILE as a topology injection mechanism. Does daemon-generated XML produce correct transport decisions? Understanding of whether topology virtualization is needed in addition to gate interception, and whether NCCL’s existing NCCL_TOPO_FILE mechanism is sufficient.

Output: A comprehensive characterization paper on NCCL’s behavior under per-GPU containerization. This is publishable on its own (measurement contribution) regardless of what follows.

Decision point: After Phase 1, we know exactly what breaks in NCCL and how to fix it. The question becomes: is this NCCL-specific, or is it a universal pattern?

4.2 Phase 1.5: DMA-BUF Hypothesis Test (RQ3)

“Can DMA-BUF file descriptors serve as a universal cross-container GPU memory sharing primitive?”

Setting: node192, A100s. Pure GPU memory IPC experiment — outside of any CCL, test the raw DMA-BUF mechanism.

ExperimentQuestionMethodExpected Finding
1.5a: DMA-BUF export + FD pass + import Can a DMA-BUF FD from one container be imported in another container and used for GPU P2P? Write standalone C programs: Container A allocates GPU memory via cuMem, exports DMA-BUF FD, sends via UDS. Container B receives FD, imports, maps to GPU VA, performs P2P copy. Yes/no answer. If yes: DMA-BUF is a viable universal primitive. If no: understanding of why (e.g., import doesn’t produce GPU VA).
1.5b: DMA-BUF vs native IPC performance Does the DMA-BUF path match vendor-native IPC performance? Compare three paths: (a) cuMem FD native, (b) DMA-BUF export/import, (c) legacy cudaIpcMemHandle (shared IPC NS). Measure latency and bandwidth for P2P copies of various sizes. Performance characterization. Quantify any overhead from the DMA-BUF path.
1.5c: Kernel requirements What kernel/driver version is needed for DMA-BUF GPU IPC? Test on kernel 5.15 (node192) and 6.x. Document required modules and capabilities. Compatibility matrix.

Output: Definitive answer on whether DMA-BUF is a viable universal GPU IPC primitive. This finding is significant regardless of direction — if DMA-BUF works, it simplifies the multi-vendor story; if it doesn’t, we understand why and what vendor-specific paths are needed instead.

Decision point: If DMA-BUF works on NVIDIA with full performance, we have strong evidence it will work on AMD/Intel (since DMA-BUF is a kernel subsystem). This de-risks Phase 2.

4.3 Phase 2: Cross-Vendor Generality (RQ1 across vendors)

“Do RCCL (AMD) and oneCCL (Intel) break in the same way as NCCL, or differently?”

Setting: AMD MI250X/MI300X (cloud) and/or Intel Data Center GPU Max (if accessible). Per-GPU containers running vendor-native CCL benchmarks.

ExperimentQuestionMethod
2a: RCCL gate taxonomy Does RCCL have the same three gates as NCCL, or different ones? Source analysis (RCCL is open source, forked from NCCL) + strace on AMD hardware. Compare gate-by-gate with NCCL findings.
2b: AMD IPC path exploration Which AMD IPC mechanism(s) work across containers? Test three paths in order: (a) opaque HSA handles via shared /dev/kfd, (b) DMA-BUF export/import, (c) HIP VMM FD. Each is a different point in the isolation–performance spectrum.
2c: RCCL bandwidth recovery Can we recover Infinity Fabric bandwidth in per-GPU containers on AMD? Apply Phase 1 methodology (gate interception + working IPC path) to RCCL. Measure allreduce bandwidth vs baseline.
2d: Cross-vendor comparison Is the failure pattern universal or vendor-specific? Compare NCCL and RCCL gate taxonomies side-by-side. Identify shared patterns vs vendor-specific divergences. Test whether the same shim approach works.

Output: Cross-vendor characterization. The key finding is whether the topology → transport → IPC failure pattern is universal (strong contribution) or vendor-specific (still useful, but different paper).

Decision point: Based on Phases 1–2 findings, choose paper direction:

  • If universal pattern + DMA-BUF works: Strong systems paper. “We discover that all CCLs break through the same three mechanisms, and DMA-BUF provides a universal recovery path.”
  • If patterns diverge but each is solvable: Characterization + system paper. “We present a taxonomy of failure modes and vendor-specific recovery strategies.”
  • If only NCCL is tractable: Deep NCCL paper. Still publishable as a thorough single-system study with measurement + mechanism contribution.

4.4 Phase 3: Isolation–Communication Model (RQ4)

“What does the isolation–communication trade-off space look like?”

This phase synthesizes findings from Phases 1–2 into a conceptual model.

ExperimentQuestionMethod
3a: Isolation level taxonomy What are the meaningful levels between “no isolation” and “full isolation”? Enumerate all namespace/mount combinations. For each, measure: (a) which CCL transport is selected, (b) what bandwidth is achieved, (c) what isolation properties are preserved.
3b: Security analysis What does each intervention expose? Can a malicious container exploit the intervention to access another container’s GPU memory? Threat model analysis. For each intervention (shim function, bind mount, shared socket), enumerate what a malicious actor could do.
3c: Predictive model Can we predict bandwidth recovery from isolation configuration? Build a simple model: given a set of shared/isolated namespaces, predict which CCL transport will be selected and what bandwidth will result. Validate against experimental measurements.

Output: A formal model of GPU communication under container isolation. This is a conceptual contribution that frames the problem space for future work.

4.5 Phase 4 (Conditional): System Construction

Only if Phases 1–3 findings call for a system.

If the research findings reveal that:

  1. The failure pattern is universal across vendors
  2. A common mechanism (e.g., DMA-BUF) enables recovery
  3. The recovery requires coordination (topology, handle brokering) beyond what a simple shim can provide

Then building a broker/daemon system is the natural next step — and it will be informed by deep understanding of the problem rather than upfront architecture decisions.

If the findings show that a simple shim per vendor is sufficient and no coordination is needed, then the paper is the characterization + shims, not a system. That is equally valid and publishable.

6. Potential Paper Framings

Depending on what we discover, the paper can take several forms. All start from the same setting (per-GPU containers) but emphasize different findings:

Framing A: Measurement + Discovery Paper

“We present the first systematic study of how container isolation breaks GPU collective communication. We discover that all major CCLs (NCCL, RCCL, oneCCL) fail through a universal three-stage pattern (topology → transport → IPC), despite having completely different internal implementations. We identify DMA-BUF file descriptors as a universal cross-container GPU memory sharing primitive, and show that minimal interception (N syscall wrappers) recovers ≥95% of native interconnect bandwidth across vendors.”

Contribution: Taxonomy + discovery + minimality proof. No system needed.

Venue: OSDI/SOSP (if the discovery is surprising enough), ATC/EuroSys (safe bet).

Framing B: Deep Single-Vendor Study

“We present a comprehensive analysis of NCCL’s behavior under per-GPU containerization: the complete set of isolation-sensitive decision points, a minimal LD_PRELOAD interception layer that recovers native NVLink bandwidth, a version compatibility study across NCCL 2.18–2.29, and a formal isolation–communication trade-off model.”

Contribution: Depth over breadth. Thorough single-system study.

Venue: ATC, EuroSys, NSDI.

Framing C: Systems Paper (if Phase 4 happens)

“We study the per-GPU container setting and discover that [universal pattern]. Based on this understanding, we build [system name], a lightweight broker that mediates topology discovery and GPU memory handle exchange across container boundaries. [System] recovers ≥95% of native interconnect bandwidth on NVIDIA NVLink, AMD Infinity Fabric, and Intel Xe Link, while preserving container isolation.”

Contribution: Measurement + system. Strongest paper but requires most work.

Venue: OSDI, SOSP, EuroSys.

Framing D: DMA-BUF Discovery Paper

“We discover that Linux DMA-BUF — a kernel subsystem designed for graphics buffer sharing — can serve as a universal cross-container GPU memory sharing primitive. We demonstrate that DMA-BUF file descriptors, passed via standard Unix domain sockets, enable GPU P2P communication across container boundaries on NVIDIA, AMD, and Intel hardware with ≤1% overhead compared to vendor-native IPC. This eliminates the need for vendor-specific IPC workarounds and enables per-GPU containerization without sacrificing interconnect bandwidth.”

Contribution: Focused discovery. Clean story around one key insight.

Venue: ATC, ASPLOS, EuroSys.

8. What Makes This Research, Not Engineering

An engineering project asks: “How do I build a system that does X?” A research project asks: “What do we not yet understand about domain Y, and what can we discover?”

Here is what makes our work research:

Engineering MindsetResearch Mindset (This Plan)
“Build a broker daemon” “Characterize what breaks and discover whether a universal pattern exists”
Success = system works Success = community learns something new about how GPU communication interacts with container isolation
Negative result = failure Negative result = valuable finding (e.g., “DMA-BUF import does not produce P2P-capable GPU VAs on AMD — here’s why”)
Architecture decided upfront Architecture emerges from understanding
Multi-vendor support is a feature Cross-vendor comparison reveals whether the problem is universal or coincidental

8.1 The Potential Surprises

Research should have room for surprise. Here are things we might discover that we don’t currently expect:

  • NCCL might have more than three gates that only manifest in specific configurations (e.g., NVSwitch systems, multi-node, CUDA MPS)
  • DMA-BUF import might work on NVIDIA but not on AMD for P2P (different kernel driver implementations) — this would be a finding about vendor divergence in Linux kernel GPU subsystems
  • The minimal intervention on AMD might be fundamentally larger than on NVIDIA (because RCCL disables cuMem), meaning the isolation–communication trade-off is vendor-dependent
  • oneCCL’s sockets mode might already solve the problem for Intel, meaning the “universal” problem is actually NVIDIA+AMD specific
  • The topology virtualization problem (making containers see the full interconnect topology) might be harder than the IPC problem (NCCL cross-validates sysfs)
  • NCCL version changes might break the shim in non-obvious ways, revealing that the interception surface is inherently unstable

Each of these surprises would change the paper’s story. That’s the point of doing research rather than engineering — we follow the findings.
