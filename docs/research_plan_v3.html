Research Plan v3: The Case for Per-GPU Containers

Motivation, Setting, and Evidence · 2026-04-10

What this document is: An evidence-based argument for why per-GPU containers are an inevitable paradigm shift in GPU cloud infrastructure, and why this creates a rich, under-explored research problem space. This document focuses entirely on motivation and setting — not solutions.

1. The Status Quo: The Monolithic Multi-GPU Container

1.1 How Multi-GPU Workloads Run Today

Today, multi-GPU workloads on Kubernetes (training, tensor-parallel serving) run in a simple model: one container gets all GPUs on a node.

  Standard K8s multi-GPU deployment (2024–2026):
  +-----------------------------------------------+
  |              Single Pod / Container            |
  |   requests: nvidia.com/gpu: 8                  |
  |                                                |
  |   GPU0  GPU1  GPU2  GPU3  GPU4  GPU5  GPU6  GPU7
  |   ================NVLink mesh==================|
  |   NCCL sees all 8 GPUs, full P2P bandwidth     |
  +-----------------------------------------------+

This works for NCCL. The CCL sees all GPUs in the same namespace, detects NVLink, and achieves full interconnect bandwidth. vLLM, Megatron-LM, DeepSpeed, and every other multi-GPU framework use this model.

So what’s the problem?

1.2 The Monolithic Container Is a Coarse-Grained Management Unit

Packing all GPUs into one container means the GPU is not the unit of management — the node is. This creates fundamental operational problems:

ProblemWhat HappensImpact
One GPU fails → all GPUs lost A single GPU ECC error, driver crash, or thermal event kills the entire container. All 8 GPUs stop. The job must restart from the last checkpoint. The 7 healthy GPUs are wasted until the job restarts.
Cannot preempt individual GPUs A high-priority job needs 4 GPUs. The scheduler must kill an entire 8-GPU job to free them, wasting the other 4 GPUs. Coarse preemption granularity. GPU utilization drops.
Cannot add or remove GPUs at runtime K8s containers have a fixed GPU count. Resizing tensor parallelism requires recreating the container, which restarts the entire workload. No elasticity. Resources locked for the job’s lifetime.
Cannot share a node across tenants If Tenant A needs 4 GPUs and Tenant B needs 4 GPUs, they cannot safely coexist on the same 8-GPU node without sharing namespaces. Nodes allocated exclusively to single tenants. Severe underutilization.
Cannot do rolling updates Updating the model on GPU 0–1 requires restarting all 8 GPUs. Standard K8s rolling update strategies don’t apply. GPU workloads can’t use standard cloud-native operational patterns.

These are not hypothetical concerns. They are backed by production data.

2. Evidence Pillar 1: GPU Failures Make Per-GPU Fault Isolation Critical

2.1 GPU Failure Rates in Production

The most detailed public data comes from Meta’s Llama 3.1 405B training (2024), run on 16,384 NVIDIA H100 GPUs over 54 days:

Meta Llama 3.1 405B training (16,384 H100s, 54 days):
• 419 unexpected failures in 54 days — one failure every 3 hours
• 58.7% caused by GPUs (GPU faults: 30.1%, HBM3 memory: 17.2%, SRAM: 4.5%, system processor: 4.1%)
• 9% annualized GPU failure rate (27% over 3 years)
• Only ~90% effective training time despite extensive automation
• 2.1% of training time spent on checkpointing and failure recovery
Source: Meta, “The Llama 3 Herd of Models,” 2024

Earlier, during OPT-175B training (992 A100s), Meta reported 56% of training time was lost to failure handling. The improvement to 90% effective time with Llama 3 required massive engineering investment in automated recovery.

2.2 The Scaling Problem

GPU failure rates compound with scale:

Cluster SizeExpected Failure IntervalImpact with Monolithic Containers
1,000 GPUs (125 nodes) ~1 failure every 50 hours One 8-GPU node dies every ~2 days
16,384 GPUs (Meta Llama 3) ~1 failure every 3 hours Every failure kills 8 GPUs; checkpoint & restart
100,000 GPUs ~1 failure every 30 minutes Continuous cascading restarts of 8-GPU groups
1,000,000 GPUs (projected 2027) ~1 failure every 3 minutes Monolithic containers become unworkable

2.3 Why Per-GPU Containers Help

With per-GPU containers, a single GPU failure kills only that GPU’s container. The remaining GPUs continue running. The system can:

  1. Isolate the failure: Only the failed GPU’s process dies, not all 8.
  2. Reconfigure without full restart: The training framework can reduce tensor parallelism (7 instead of 8), switch to a different parallelism strategy (NTP, Nonuniform-Tensor-Parallelism, arXiv 2025), or wait for a replacement GPU.
  3. Replace the failed GPU: K8s can reschedule a new single-GPU container while the other 7 continue, rather than restarting the entire 8-GPU group.

The argument: At the failure rates observed in production (one every 3 hours at 16K GPUs), the monolithic 8-GPU container is an amplifier of failure impact. Per-GPU containers reduce the blast radius from 8 GPUs to 1 GPU. As clusters scale to 100K+ GPUs, this reduction becomes essential — without it, the entire cluster spends more time recovering from failures than training.

Related work confirming this direction:

  • Oobleck (SOSP ’23): Resilient training using pipeline templates. Tolerates f simultaneous failures via f+1 heterogeneous pipeline replicas. Up to 13.9× faster recovery than Bamboo/Varuna. Operates at the framework level — per-GPU containers would make this transparent at the OS level.
  • NTP (Nonuniform-Tensor-Parallelism, arXiv 2025): When a GPU fails in a TP group of 8, NTP redistributes work to the remaining 7 GPUs with minimal overhead. Requires the ability to isolate and replace individual GPUs — exactly what per-GPU containers provide.
  • Singularity (Microsoft, 2022): Designed around the principle that “all jobs are preemptable, migratable, and dynamically resizable by default.” This requires GPU-granular lifecycle management.

3. Evidence Pillar 2: Hardware Is Dissolving the “Node”

3.1 The GB200 NVL72: 72 GPUs in One Rack

NVIDIA’s GB200 NVL72 fundamentally changes the node concept:

GB200 NVL72 architecture: 72 Blackwell GPUs connected via NVLink 5.0 (1.8 TB/s per GPU, 130 TB/s aggregate) across an entire rack. There is no direct GPU-to-GPU connectivity within a single server — all 72 GPUs connect through external NVSwitches. The entire rack acts as a single NVLink domain.
Source: NVIDIA, Nebius, CoreWeave documentation

This creates a new reality:

  • You cannot give all 72 GPUs to one job. Most workloads need 4–32 GPUs. Giving 72 to one job wastes the rest. Multiple jobs must share the rack.
  • Each job’s GPUs must be in separate containers for isolation, billing, security, and independent lifecycle management.
  • But all 72 GPUs share one NVLink domain. Jobs on the same rack should use NVLink (1.8 TB/s), not InfiniBand (~400 GB/s), for communication between their GPUs.
  • Container isolation breaks NVLink communication. This is exactly our problem.

3.2 NVIDIA Acknowledges the Problem

NVIDIA built ComputeDomains and IMEX specifically to address this. IMEX is “software at the GPU-driver level that lets GPUs communicate across nodes. With IMEX, every individual GPU memory export/import operation is subject to fine-grained access control.” ComputeDomains are the K8s-level abstraction on top: they “dynamically create, manage, and tear down IMEX domains as multi-node workloads are scheduled to nodes and run to completion.” When a workload is submitted, a ComputeDomain “will form just in time around workload pods when they are actually scheduled onto nodes.” When the workload finishes, “its corresponding IMEX domain and channels are automatically cleaned up.”

NVIDIA identifies three requirements that drove the design:

  1. “Security isolation: In a zero-trust environment, there is a clear need for neighboring GPU jobs to be securely isolated.”
  2. “Fault isolation: Neighboring jobs, even if trusted, must not step onto each other’s toes.”
  3. “Cost efficiency: Resource utilization must be kept high even with (1) and (2) in place.”

Current limitation: one pod per node, one ComputeDomain per node

Despite these goals, ComputeDomains today have a coarse-grained allocation model that falls far short of per-GPU granularity:

Constraint 1: “Currently, only one pod per node can be part of any given ComputeDomain. Users have to be aware of how many GPUs are available in a node, and then typically grab them all from within a single workload pod. The application in that pod then needs to subdivide its work across those GPUs.”

Constraint 2: “Currently, at most one ComputeDomain is supported per node.” If a ComputeDomain uses only some GPUs on a node, “the remaining GPUs in that node cannot be part of any other ComputeDomain.”

Source: NVIDIA Developer Blog, “Enabling Multi-Node NVLink on Kubernetes for GB200 and Beyond,” Nov 2025

This means that even on the state-of-the-art GB200 NVL72, one job exclusively owns entire nodes. On an 18-node rack (72 GPUs), a job needing 8 GPUs occupies 2 full nodes (8 GPUs) while the remaining GPUs on those nodes sit idle — they cannot be used by any other ComputeDomain. This is the same monolithic, node-level allocation that plagues existing A100/H100 clusters, just at a larger scale.

NVIDIA’s stated future direction: per-GPU pods

“We are planning to remove this constraint to make the notion of individual nodes less relevant. It will then be possible for the application to be composed of many single-GPU pods that may or may not be placed next to each other on the same node. In that mode, the unit of interest is the individual GPU, and not the individual node — node boundaries become almost transparent.”
Source: NVIDIA Developer Blog, Nov 2025. Tracked in issue #309.

They also plan to lift the one-ComputeDomain-per-node constraint (tracked in issue #353), but acknowledge the trade-off explicitly: “Lifting that constraint allows for increased resource utilization on the one hand but, on the flipside, may weaken fault isolation between workloads. No universal remedy exists, and we will allow users to pick their sweet spot in the trade-off spectrum.”

Limitations of IMEX + ComputeDomains

Even when these constraints are lifted, fundamental limitations remain:

  • GB200/Blackwell only. Requires 4th-gen NVSwitches + IMEX kernel module. Does not work on A100, H100, or any AMD/Intel hardware. The vast majority of deployed GPU infrastructure will never run IMEX.
  • NVIDIA-proprietary. No equivalent exists for AMD Infinity Fabric or Intel Xe Link. The isolation–communication trade-off is not NVIDIA-specific, but their solution is.
  • Requires specific kernel/driver versions (GPU Operator 25.3+, K8s 1.32+).
  • The isolation–communication trade-off remains unsolved. NVIDIA acknowledges there is “no universal remedy” — they push the trade-off to users rather than solving it. Our research aims to characterize this trade-off space formally.

3.3 The Trajectory: From Node to GPU as the Unit

EraUnit of AllocationGPUs per UnitInter-GPU Fabric
DGX A100 (2020) Node (8 GPUs) 8 NVLink 3.0 (intra-node only)
DGX H100 (2023) Node (8 GPUs) 8 NVLink 4.0 (intra-node only)
DGX GB200 NVL72 (2025) Rack (72 GPUs) 72 NVLink 5.0 (cross-node, intra-rack)
NVL576 (announced) Multi-rack (576 GPUs) 576 NVLink (cross-rack)
UALink 1.0 (2027?) Domain (up to 1,024 accelerators) 1,024 Open standard, multi-vendor

The argument: As the NVLink domain grows from 8 GPUs (one node) to 72 (one rack) to 576+ (multi-rack), the “one container per NVLink domain” model becomes absurd. You cannot put 72 GPUs in one container for each job. Multiple containers must share the NVLink domain, and each container’s GPUs must communicate at NVLink bandwidth, not InfiniBand/TCP. This is not a future scenario — GB200 NVL72 is shipping now and NVIDIA built ComputeDomains specifically because per-GPU pods on shared NVLink are necessary.

4. Evidence Pillar 3: GPU Utilization Is Catastrophically Low

4.1 The Numbers

SourceReported GPU UtilizationContext
CNCF member survey (2025) 13–37% baseline Production K8s GPU clusters. Advanced scheduling triples utilization from 13% to 37%.
Microsoft Azure (2024) <15% for ML inference Inference workloads are bursty; GPUs idle during off-peak.
Alibaba PAI (2023–24) 25–50% Production clusters. MIG sharing introduced 21–42% fragmentation.
Industry average (2024) 20–30% Cited by CIO, NVIDIA, multiple vendors.

At H100 prices ($27,000–$40,000 per GPU, $2–$5/hour cloud), a 1,000-GPU cluster at 25% utilization wastes ~$50–$100M annually.

4.2 Why Utilization Is Low: The Fragmentation Problem

The root cause is coarse-grained allocation: K8s allocates whole GPUs (or whole nodes) to jobs, and jobs rarely need exactly what’s allocated.

  Example: 8-GPU node, three jobs arrive:

  Job A: needs 3 GPUs     Job B: needs 3 GPUs     Job C: needs 4 GPUs

  Monolithic allocation:
  +--Node 1--+  +--Node 2--+  +--Node 3--+
  | A A A . . . . . |  | B B B . . . . . |  | C C C C . . . . |
  | 3/8 used (38%)  |  | 3/8 used (38%)  |  | 4/8 used (50%)  |
  +----------+  +----------+  +----------+
  Total: 10 GPUs used, 24 allocated.  Utilization: 42%.

  Per-GPU allocation:
  +------Node 1------+  +--Node 2--+
  | A A A  B B B  C C |  | C C . . . . . . |
  | 8/8 used (100%)  |  | 2/8 used (25%)  |
  +------------------+  +----------+
  Total: 10 GPUs used, 16 allocated.  Utilization: 63%.
  (+50% improvement in allocation efficiency)

This is the classic bin-packing advantage. NVIDIA’s Volcano Scheduler achieved 90% GPU occupancy using bin-packing, up from much lower baselines. But effective bin-packing requires fine-grained allocation units — per-GPU containers.

4.3 GPU Sharing (MIG) Creates New Problems

MIG (Multi-Instance GPU) can partition an A100/H100 into up to 7 slices. But:

  • Fragmentation: Alibaba’s PAI platform found that MIG converted 21–42% of unallocated GPU resources into fragmentation — partially allocated GPUs that can’t fit any pending job.
  • Coarse partitions: MIG slices have fixed sizes (e.g., 1g.10gb, 2g.20gb, 3g.40gb on A100). Jobs rarely fit exactly.
  • No cross-slice communication: MIG partitions are hardware-isolated. Two MIG slices on the same GPU cannot use NVLink to communicate.

Per-GPU containers avoid these problems: each GPU is a whole unit, and the communication layer (if solved) allows any subset of GPUs to form a group.

The argument: GPU utilization is 20–30% in production, wasting billions of dollars annually. The root cause is coarse-grained allocation at the node level. Per-GPU containers enable GPU-level bin-packing, which can dramatically improve utilization. MIG is an alternative but introduces its own fragmentation problems and prevents high-bandwidth inter-GPU communication. Per-GPU containers are the more flexible, software-defined approach.

5. Evidence Pillar 4: GPU Scheduling Has Evolved Toward GPU Granularity — but Stopped at the Container Wall

5.1 The Evolution of GPU Scheduling Granularity

GPU cluster scheduling research has progressively moved from coarse to fine granularity over the past six years:

EraGranularityKey SystemsScheduling Unit
2020 Node-level HiveD (OSDI ’20), Gavel (OSDI ’20), Themis (NSDI ’20) Allocate whole nodes or GPU groups (“cells”) to jobs
2021–22 GPU-group-level Pollux (OSDI ’21), Synergy (OSDI ’22), AntMan (OSDI ’20) Dynamically adjust GPU count per job, co-adaptive batch size tuning
2023–24 GPU-level Sia (SOSP ’23), CASSINI (NSDI ’24), Crux (SIGCOMM ’24) Per-GPU allocation with topology and communication awareness
2025+ GPU-level (elastic) Tenplex (SOSP ’24), NTP (arXiv ’25), KAI Scheduler, Kale (SoCC ’24) Add/remove individual GPUs from running jobs at runtime
2025+ Sub-GPU LithOS (SOSP ’25), Tally (ASPLOS ’25), REEF (OSDI ’22) TPC-level or thread-block-level partitioning within a single GPU

The critical observation: Every single one of these systems assumes either bare metal or all GPUs within one container. None handles per-GPU scheduling across container boundaries with high-bandwidth communication. The scheduling gets finer, but the container boundary stays monolithic.

5.2 What Each System Assumes About Containers

SystemVenueScheduling GranularityContainer ModelHandles Cross-Container GPU Comm?
HiveD OSDI ’20 GPU-group (“cells” with topology affinity) K8s pods — all GPUs in one pod No
Pollux OSDI ’21 Per-job GPU count + batch size Bare metal / one container per job No
Sia SOSP ’23 Per-job GPU count, heterogeneity-aware Bare metal / one container per job No
Tenplex SOSP ’24 Dynamic GPU add/remove with TP+PP+DP resize Bare metal (framework-level state management) No
Singularity Microsoft ’22 Dynamic preemption + migration + resize Internal system (not K8s) No
KAI Scheduler NVIDIA ’25 Per-GPU with topology, fractional, bin-packing K8s pods — all GPUs in one pod No
LithOS SOSP ’25 Sub-GPU (TPC-level partitioning) Bare metal (OS-level GPU management) No
Tally ASPLOS ’25 Thread-block level, LD_PRELOAD interception Container-aware (intercepts across containers) Single-GPU sharing only, not multi-GPU distributed

Tally (ASPLOS ’25) is the closest related work to our approach — it uses LD_PRELOAD to intercept CUDA API calls across container boundaries for GPU sharing with performance isolation (7.2% overhead). But Tally addresses sharing a single GPU between containers, not multi-GPU distributed communication (NCCL allreduce, etc.) across containers. These are fundamentally different problems.

5.3 The Elasticity Problem: Framework Support Exists, Container Support Does Not

GPU workloads have fundamentally different resource needs over time:

  • Inference: Bursty. 10× traffic spikes during peak hours.
  • Training: Long-running but interruptible. Can use more GPUs when available.
  • Mixed workloads: GPUs should flow between training and inference dynamically.

Recent systems have made significant progress on elastic GPU allocation at the framework level:

SystemVenueWhat It DoesRestart Required?Container Assumption
Tenplex SOSP ’24 Changes TP+PP+DP at runtime by re-partitioning model state across GPUs via Parallelizable Tensor Collections (PTCs) No restart, but 122s reconfiguration for 16→8 GPU resize Bare metal. Cannot add/remove containers.
NTP arXiv ’25 When a GPU fails in TP group of 8, redistributes work to remaining 7 with nonuniform tensor parallelism. ~14% more compute per surviving GPU. No restart. Continuous training. Bare metal. Detects failure at process level.
Singularity Microsoft ’22 “All jobs are preemptable, migratable, and dynamically resizable by default.” Transparent to user code. Requires checkpointing for recovery. Internal Microsoft system. Not K8s.
Kale SoCC ’24 Traffic-forecasted GPU count adjustment for online model training. Uses time-series prediction + resource-throughput modeling. Requires reconfiguration. Bare metal.
Lyra EuroSys ’23 Loans idle inference GPUs for elastic training during off-peak. 1.48× speedup in job completion time. Job-level scaling. Cluster-level job scheduler (bare metal).
ElasticFlow ASPLOS ’23 Elastic serverless training with admission control and deadline-aware scaling. 12.95× deadline satisfaction improvement. Deadline-based planning. Serverless platform (bare metal workloads).

The pattern is clear: the framework-level support for elastic GPU scheduling is maturing rapidly (Tenplex can even resize tensor parallelism at runtime). But every one of these systems assumes bare-metal deployment. On Kubernetes, adding or removing a GPU means adding or removing a container — and when you do that, the remaining containers must continue communicating at NVLink bandwidth. No existing system handles this.

5.4 K8s Is Moving Toward Per-Device Allocation

K8s FeatureStatusWhat It Enables
Dynamic Resource Allocation (DRA) GA in K8s 1.32 (2025). NVIDIA donated DRA driver to CNCF (KubeCon 2026). Per-device GPU allocation with topology awareness. Replaces the coarse nvidia.com/gpu: N model. Reduces GPU idle from 20–30% fragmentation via fine-grained allocation.
In-Place Pod Resize Beta in K8s 1.33 (2025) Resize CPU/memory without pod restart. GPU support not yet implemented but is the natural next step.
NVIDIA KAI Scheduler Open-sourced 2025 (Apache 2.0, now CNCF Sandbox) Fractional GPU allocation, topology-aware scheduling, hierarchical fairshare, gang scheduling, bin-packing. Supports reclamation and preemption — but at pod level only.
K8s 1.35 Restart All Containers Alpha in K8s 1.35 (2026) Efficient pod state reset without deletion/recreation. “Especially useful for AI/ML workloads” per K8s docs.

The argument: GPU scheduling has evolved through four generations of increasing granularity, from node-level (HiveD, 2020) to sub-GPU (LithOS, 2025). Elastic systems (Tenplex, NTP, Singularity) can now add/remove individual GPUs from running jobs. K8s is moving toward per-device allocation (DRA, KAI Scheduler). But all of this assumes either bare metal or all GPUs in one container. The moment you containerize at GPU granularity — which K8s DRA enables and NVIDIA is planning for — the communication layer (NCCL) breaks. The framework support exists; the container-level communication support does not. This is the gap.

6. Evidence Pillar 5: Multi-Tenancy on Shared GPU Nodes

6.1 The Economic Pressure

An 8-GPU DGX H100 costs ~$300,000. A 72-GPU GB200 NVL72 rack costs substantially more. Cloud providers cannot afford to give each customer exclusive access to the entire system. Multi-tenancy — multiple customers sharing the same physical hardware — is an economic necessity.

6.2 Multi-Tenancy Requires Container Isolation

When two tenants share a GPU node, their workloads must be in separate containers:

  • Security: Tenant A must not be able to read Tenant B’s GPU memory, intercept their communication, or interfere with their workload.
  • Billing: Each tenant’s GPU usage must be independently measurable.
  • Compliance: Regulated industries require provable isolation boundaries.
  • Fault isolation: Tenant A’s crash must not affect Tenant B.

6.3 But Each Tenant May Need Multiple GPUs with High-Bandwidth Communication

Consider a concrete scenario on a GB200 NVL72 rack (72 GPUs):

  GB200 NVL72 Rack (72 GPUs, one NVLink domain):

  Tenant A: Llama-70B serving (TP=8)  → needs 8 GPUs with NVLink
  Tenant B: Training job              → needs 32 GPUs with NVLink
  Tenant C: Mistral-7B serving (TP=2) → needs 2 GPUs with NVLink
  Tenant D: Batch inference           → needs 16 GPUs with NVLink
  Free: 14 GPUs

  Each tenant's GPUs are in isolated containers (security, billing).
  All 72 GPUs share ONE NVLink domain (1.8 TB/s per GPU).

  Problem: Tenant A's 8 GPUs are in separate containers from Tenant B's 32.
           But WITHIN each tenant, the GPUs also need NVLink communication.
           If each tenant's GPUs are in separate containers (for fault isolation,
           elasticity), then intra-tenant GPU communication also breaks.

The multi-tenancy scenario has two levels:

  1. Inter-tenant isolation: Tenant A’s containers must not communicate with Tenant B’s containers (unless explicitly authorized). This is a security requirement.
  2. Intra-tenant communication: Within a tenant, multiple GPUs (possibly in separate containers for fault isolation/elasticity) must communicate at NVLink bandwidth. This is a performance requirement.

Today, there is no way to achieve both.

7. Synthesis: Why Per-GPU Containers Are Inevitable

The five evidence pillars converge on the same conclusion:

PillarForceWhy It Requires Per-GPU Containers
1. GPU failures Hardware reality 9% annualized GPU failure rate. Per-GPU containers reduce failure blast radius from 8 GPUs to 1 GPU. Critical at scale (100K+ GPUs).
2. Hardware trajectory Architecture trend NVLink domains growing from 8 to 72 to 576+ GPUs. Multiple jobs must share the domain. NVIDIA already building toward “many single-GPU pods.”
3. Utilization crisis Economic pressure 20–30% GPU utilization wastes billions. GPU-level bin-packing requires GPU-level allocation units (containers).
4. Elastic scheduling Workload dynamics Adding/removing GPUs at runtime requires independent GPU lifecycle. Published systems (Tenplex, NTP, Singularity) all assume this.
5. Multi-tenancy Economic + security Shared GPU nodes require tenant isolation. Each tenant’s GPUs must be in separate containers, but still communicate at NVLink bandwidth.

The thesis: Per-GPU containers are not a speculative idea — they are being driven by five independent forces (hardware failures, architecture trends, utilization economics, workload dynamics, and multi-tenancy requirements). NVIDIA is already building toward this future (ComputeDomains, DRA, KAI Scheduler). But their solution is proprietary and limited to newest hardware. The research question is: what challenges does the per-GPU container paradigm create, and how do we solve them in a general, vendor-agnostic way?

8. The Challenges (Research Problem Space)

If per-GPU containers are the future, what breaks? This is where the research lies. We identify the challenges here but do not prescribe solutions — those should emerge from systematic study.

Challenge 1: High-Bandwidth Inter-Container GPU Communication

GPU CCLs (NCCL, RCCL, oneCCL) assume shared OS namespaces for intra-node communication. Container isolation breaks this assumption, causing fallback to TCP/socket (2–10× bandwidth loss). Recovering native interconnect bandwidth (NVLink, Infinity Fabric, Xe Link) across container boundaries is the primary technical challenge.

What we already know: Our exp1–2 showed this is solvable on NVIDIA (cuMem FD passing). But is it solvable on AMD? Intel? Is there a universal mechanism? What is the minimal intervention?

Challenge 2: Topology Virtualization

Each per-GPU container sees only its own GPU. The CCL cannot discover the full interconnect topology (NVLink mesh, PCIe hierarchy, NUMA affinity). Without this, even if communication is restored, the CCL may choose suboptimal algorithms (e.g., ring instead of tree, or wrong ring order).

Challenge 3: Elastic Reconfiguration

When a GPU is added or removed (due to elasticity or failure), the communication group must be reconfigured. In monolithic containers, this requires restarting the entire job. Per-GPU containers should enable online reconfiguration, but the mechanisms for this (NCCL communicator resize, state migration) are poorly understood.

Challenge 4: Security Model

Per-GPU containers with cross-container GPU communication partially re-opens the isolation boundary. What is the security model? Can a malicious container exploit the communication channel to access another container’s GPU memory? How do we provide communication without sacrificing isolation guarantees?

Challenge 5: Scheduling and Placement

The scheduler must understand both physical GPU topology (which GPUs share NVLink) and logical communication groups (which containers need to communicate). Placing communicating containers on GPUs that share NVLink is essential for performance. Existing K8s schedulers (Kueue, Volcano, KAI) have topology awareness at the node level, but not at the per-GPU level within a shared NVLink domain.

Note: Communication recovery (Challenge 1) is the challenge we have the most experience with (exp1–2). But it is only one of five challenges. The research plan should ultimately address as many of these as feasible, with communication as the starting point since we have a head start.

9. Three Concrete Settings to Anchor the Research

The five evidence pillars (Sections 2–6) establish why per-GPU containers are needed. But a strong research project needs a specific workload where we can demonstrate the need end-to-end: the workload works today → a real operational need forces per-GPU containers → something breaks → we study and fix it. Below are three candidates, each grounded in a real system and a real problem.

Setting A: Elastic Expert Parallelism for MoE Serving (vLLM + DeepSeek)

“DeepSeek-V3 serving on 32 GPUs needs to scale down to 16 during off-peak. Today this requires a full restart. Per-GPU containers would make it zero-downtime.”

The workload

Serving DeepSeek-V3/R1 (Mixture-of-Experts, 671B parameters) with Expert Parallelism (EP) on 32+ GPUs. This is the hottest deployment pattern in LLM serving (2025–2026). vLLM, SGLang, and NVIDIA Dynamo all support it.

Why per-GPU containers become necessary

MoE inference demand is highly bursty. During peak, you need 32 GPUs (EP=32) for throughput. During off-peak, 16 GPUs suffice. Keeping 32 GPUs allocated 24/7 wastes ~50% of GPU-hours. You need to dynamically add and remove GPUs from a running serving instance.

What breaks today

  • vLLM (current): Scaling EP up or down requires a full process restart with new configuration. vLLM’s RFC #20323 (“Elastic Expert Parallelism”) explicitly identifies this: “The only viable solution today is to perform a complete restart with a new configuration, which is quite slow and would drop a lot of traffic.”
  • SGLang EEP (2026-03): SGLang’s Elastic Expert Parallelism reduces the scaling interruption from 2–3 minutes to ~10 seconds, but still interrupts service and drops requests during reconfiguration. Tested with DeepSeek V3.2 on 32 GPUs (4 nodes), tolerating up to 16 rank failures.
  • Monolithic container limit: In a single 32-GPU container, you cannot remove 16 GPUs without killing the container. K8s has no mechanism to shrink a container’s GPU allocation at runtime (In-Place Pod Resize does not yet support GPU resources).

How per-GPU containers solve it

  Peak (32 GPUs, EP=32):
  [C0][C1][C2]...[C31]   ← 32 per-GPU containers, each hosts experts
       NVLink mesh

  Scale down to 16 (off-peak):
  [C0][C1][C2]...[C15]   ← K8s removes 16 containers
  Experts redistributed   ← EPLB (vLLM's load balancer) reshuffles
  Remaining 16 continue   ← zero downtime, no dropped requests

  Scale back up to 32 (next peak):
  [C0][C1]...[C15][C16]...[C31]  ← K8s adds 16 new containers
  Experts loaded into new containers
  Standard K8s horizontal pod autoscaler (HPA)

The challenges this creates

  1. Cross-container expert-to-expert communication. EP requires all-to-all communication between GPUs (experts route tokens to each other). With per-GPU containers, this all-to-all must cross container boundaries at NVLink bandwidth.
  2. Dynamic group membership. When GPUs join or leave, the NCCL communicator must be resized or recreated without disrupting ongoing inference.
  3. Expert state migration. When a GPU is removed, its experts must be redistributed to remaining GPUs. When a GPU is added, experts must be loaded.

  • vLLM large-scale serving: DeepSeek @ 2.2k tok/s/H200

Why this setting is strong: (1) It is the most active area in LLM serving right now — DeepSeek MoE is the dominant model architecture. (2) The problem is explicitly identified by vLLM (RFC #20323) as unsolved. (3) The solution maps directly to standard K8s patterns (HPA, rolling updates). (4) It requires solving the cross-container GPU communication problem as a prerequisite.

Setting B: Disaggregated Prefill/Decode Serving (DistServe Model)

“Prefill and decode need separate GPU groups for independent scaling. But KV cache transfer between them needs NVLink on the same node. Container isolation breaks this.”

The workload

LLM serving with prefill/decode disaggregation (DistServe, OSDI ’24; Splitwise, ISCA ’24). Prefill processes the input prompt (compute-bound, needs FLOPS). Decode generates tokens one at a time (memory-bound, needs HBM bandwidth). Separating them onto different GPU groups allows independent scaling and hardware specialization.

Industry adoption (2026): “Almost every production-grade LLM serving framework — NVIDIA Dynamo, llm-d, Ray Serve LLM, SGLang, vLLM, LMCache, MoonCake — runs on disaggregation.” (Hao AI Lab retrospective)

Why per-GPU containers become necessary

Prefill and decode have different scaling needs:

  • Prefill: Scales with input length and request rate. Bursty.
  • Decode: Scales with concurrent sessions. More stable.

To scale them independently, they must be in separate containers (separate pods in K8s, separate scaling groups). You cannot independently scale a sub-process within a monolithic container using K8s primitives.

What breaks today

The critical data transfer between prefill and decode is the KV cache. DistServe’s own design notes:

“DistServe requires the prefill and decoding instance to maintain the same stage on one machine, enabling the use of intra-node NVLink bandwidth for transmission, thus significantly reducing transmission delay.” — DistServe (OSDI ’24)

The dilemma:

OptionIndependent Scaling?KV Cache Transfer Speed
Same container (all GPUs) No NVLink (600 GB/s)
Separate containers, same node Yes TCP fallback (~10 GB/s) — NCCL breaks
Separate containers, different nodes Yes InfiniBand (~50 GB/s) — works but wastes NVLink
Separate containers, same node, NVLink recovered Yes NVLink (600 GB/s)

The challenges this creates

  1. Cross-container KV cache transfer at NVLink bandwidth. The prefill container must send KV cache to the decode container via NVLink, not TCP.
  2. Within each container, TP still needs NVLink. A prefill group of 4 GPUs uses tensor parallelism internally, requiring NVLink within the group. If these 4 GPUs are in per-GPU containers (for fault isolation or fine-grained scheduling), the TP communication also breaks.
  3. Dynamic assignment. GPUs may shift between prefill and decode pools based on load. This requires reconfiguring communication groups.

Evidence and references

  • DistServe: OSDI ’24 + 18-month retrospective
  • Splitwise: ISCA ’24, 2.35× throughput under same cost
  • NVIDIA Dynamo + NIXL: vLLM PD disaggregation docs
  • Gimlet Labs: Multi-vendor prefill-decode disaggregation

Why this setting is strong: (1) Disaggregated serving is already universal in production LLM systems. (2) The NVLink requirement is explicitly stated by DistServe. (3) The tension (independent scaling needs containers, NVLink needs shared namespace) is structural and unavoidable. (4) The workload is clearly defined (KV cache transfer between two GPU groups).

Setting C: GPU-Granular Scheduling on Containerized GPU Fabrics

“GPU scheduling evolved toward per-GPU granularity. Hardware evolved toward rack-scale fabrics. Production evolved toward Kubernetes containers. When all three meet, GPU communication breaks. We study and solve this.”

The research story arc

This setting generalizes preemption into the broader problem of GPU-granular scheduling across container boundaries. The story has three acts:

  1. Act 1: GPU scheduling got finer. Over six years, the community moved from node-level allocation (HiveD, 2020) to per-GPU elastic scheduling (Tenplex, 2024) to sub-GPU partitioning (LithOS, 2025). This evolution enables preemption, elasticity, bin-packing, and fault recovery at GPU granularity. (See Section 5 for the full evolution table.)
  2. Act 2: Hardware and operations demand containers. NVLink domains grew from 8 GPUs (DGX A100) to 72 (NVL72) to 576 (NVL576). You cannot give all 72 GPUs to one job. Multiple jobs must share the fabric. Meanwhile, production deployment requires containers for isolation, billing, security, health monitoring, and rolling updates. K8s has 92% market share in orchestration; AI/ML workloads on K8s at 54% and growing (CNCF survey: 90%+ expect increase). Even NVIDIA runs Slurm inside K8s (Slinky).
  3. Act 3: The container wall breaks GPU-granular scheduling. Every scheduling operation — allocation, preemption, elasticity, migration — that worked on bare metal breaks when GPUs are in separate containers. NCCL falls back to TCP (2–10× bandwidth loss). The scheduler places GPUs on the same NVLink mesh, but the CCL inside the container cannot use it.

What GPU-granular scheduling requires (and what containers break)

Scheduling OperationWhat It NeedsHow Containers Break It
Allocation
(place N GPUs for a job)		 GPUs allocated to the same job must communicate at NVLink bandwidth Per-GPU containers have separate namespaces; NCCL cannot detect co-location
Preemption
(reclaim K GPUs from a job)		 Remaining N−K GPUs must continue running and reform the collective group Removing a container severs NCCL communicator; surviving containers cannot re-establish P2P communication without shared namespaces
Elasticity
(add/remove GPUs at runtime)		 New GPUs must join the existing collective group; departed GPUs must leave cleanly New container cannot join existing NCCL communicator across container boundaries; ncclCommSplit/Shrink not designed for cross-container use
Migration
(move a GPU’s workload to another GPU)		 Checkpoint GPU state, transfer to new GPU, rejoin collective group Cross-container GPU memory transfer requires NVLink; container isolation prevents it
Bin-packing
(consolidate jobs to free nodes)		 After consolidation, GPUs from the same job on the same node must communicate Even after optimal placement, per-GPU containers on the same node cannot use NVLink

The gap confirmed by exhaustive literature search

We surveyed all major GPU scheduling systems from 2020–2026. The finding is stark:

CapabilityBest Existing SystemHandles Containers?
Sub-GPU scheduling (TPC/thread-block level) LithOS (SOSP ’25), Tally (ASPLOS ’25) Tally handles single-GPU sharing across containers; LithOS is bare metal. Neither handles multi-GPU distributed jobs.
Topology-aware per-GPU allocation KAI Scheduler (NVIDIA ’25), HiveD (OSDI ’20) K8s pod-level only. Cannot schedule individual GPUs across container boundaries with NVLink communication.
Elastic GPU add/remove without restart Tenplex (SOSP ’24) Bare metal only. Cannot add/remove containers.
GPU preemption from multi-GPU job NONE FOUND All K8s preemption (KAI, ReclaimIdleResource) is pod-level. No system reclaims individual GPUs from a running multi-GPU job.
Per-GPU preemption for distributed jobs across containers NOTHING EXISTS This is the gap.

Core finding: Distributed training/serving jobs use collective communication (NCCL). Removing one GPU requires all GPUs to coordinate — the collective must be reformed. Most scheduling research assumes single-process or bare-metal deployments where this coordination is straightforward. In per-GPU containers, the coordination must cross container boundaries at NVLink bandwidth. No existing system does this.

Concrete example: preemption

  Shared cluster, 16 GPUs across 2 nodes:

  Node 1: [Training Job A: 8 GPUs, low priority]
  Node 2: [Training Job B: 8 GPUs, low priority]

  High-priority serving job arrives, needs 4 GPUs.

  Today (pod-level preemption, monolithic container):
    Kill Job A entirely (frees 8 GPUs).
    4 GPUs used by new serving job. 4 GPUs idle.
    Job A must checkpoint + full restart.
    Waste: 4 idle GPUs + restart cost.

  With per-GPU containers + cross-container NVLink:
    Scheduler preempts 4 of Job A's per-GPU containers.
    Job A's remaining 4 containers reform NCCL communicator.
    Job A reconfigures to TP=4 (NTP-style), continues training.
    Serving job starts in 4 new per-GPU containers.
    Waste: zero idle GPUs. No full restart.

The challenges this creates

  1. Cross-container collective communication. Per-GPU containers on the same NVLink mesh must communicate at native bandwidth. NCCL’s three gates (hostHash, shmDev, IPC socket) all fail under container isolation. This is the foundational challenge (and what our exp1–2 work addresses for NVIDIA).
  2. Dynamic group membership. When GPUs join or leave (preemption, elasticity, failure), the NCCL communicator must be resized or recreated across container boundaries. ncclCommSplit (NCCL 2.27+) exists but was not designed for cross-container use.
  3. Topology virtualization. Each per-GPU container sees only its own GPU. The CCL cannot discover the full NVLink topology. The scheduler knows the topology, but the CCL inside the container does not.
  4. Rapid reconfiguration. Preemption must be fast (<10s). Full NCCL re-initialization takes seconds. Can we do incremental reconfiguration?
  5. Security model. Cross-container GPU communication partially re-opens the isolation boundary. What is the minimal exposure required?

Evidence and references

  • Scheduling systems: HiveD (OSDI ’20), Pollux (OSDI ’21), Sia (SOSP ’23), CASSINI (NSDI ’24), Crux (SIGCOMM ’24) — all assume bare metal or one container per job
  • Elastic systems: Tenplex (SOSP ’24), Singularity (Microsoft ’22), ElasticFlow (ASPLOS ’23), Lyra (EuroSys ’23), Kale (SoCC ’24) — all bare metal
  • Sub-GPU scheduling: LithOS (SOSP ’25, TPC-level), Tally (ASPLOS ’25, LD_PRELOAD across containers but single-GPU sharing only), REEF (OSDI ’22, microsecond kernel preemption)
  • GPU preemption on K8s: ReclaimIdleResource (CNCF 2026, pod-level), NVIDIA KAI Scheduler (pod-level)
  • Failure tolerance: NTP (arXiv ’25, nonuniform tensor parallelism after GPU failure)

Why this setting is the strongest: (1) It is the broadest — every scheduling operation (allocation, preemption, elasticity, migration, bin-packing) hits the same container wall. (2) It connects to the richest body of prior work (HiveD, Pollux, Sia, Tenplex, KAI — all top venues). (3) The gap is confirmed by exhaustive search: no system does per-GPU scheduling across container boundaries with high-bandwidth communication. (4) The communication recovery we’ve already built (exp1–2, Exp A/A′) is the foundational enabler that makes all of these scheduling operations possible in per-GPU containers. (5) Settings A (elastic EP) and B (disaggregated P/D) are special cases of this general problem.

The gap: The industry is building the pieces (per-GPU allocation via DRA, topology-aware scheduling via KAI, elastic frameworks via Tenplex) but none of them solve the communication problem that arises when GPUs in separate containers need to talk at NVLink bandwidth. NVIDIA’s ComputeDomains solve it for one vendor on one hardware generation. Nobody has studied whether this is solvable in general.

12. Summary

Per-GPU containers are not a solution looking for a problem. They are demanded by five independent, well-evidenced forces: GPU hardware failure rates (9% annualized, one failure every 3 hours at 16K GPUs), dissolving node boundaries (GB200 NVL72: 72 GPUs per NVLink domain), catastrophic GPU utilization (20–30% in production), elastic scheduling requirements (Tenplex, NTP, Singularity), and multi-tenancy economics. NVIDIA is already building toward this future with ComputeDomains and DRA. The research opportunity is to study and solve the challenges this paradigm shift creates — starting with the most fundamental one: recovering high-bandwidth GPU communication across container isolation boundaries.
