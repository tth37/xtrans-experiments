Research Plan v5: Per-GPU Containers
for In-Place Elastic MoE Serving

Refinement of v4: narrowing to the per-GPU-container + vLLM Elastic EP direction
2026-04-20

What this document is. A focused follow-up to research plan v4. v4 proposed three experiments (Tenplex, Oobleck, vLLM Elastic EP) to empirically discover where the container boundary breaks GPU-granular scheduling. We ran v4’s Exp A1 and Exp A3 to completion. The results clarified which direction is worth pursuing further. This v5 plan commits to that direction: per-GPU containers as the substrate for in-place elastic Mixture-of-Experts serving, built on vLLM’s Elastic Expert Parallelism. v5 is not a system design. It lists exploration fields and workaround plans so that a system design can emerge from the investigation.

1. Ultimate Goal

The ultimate goal is unchanged from v4:

Enable true elasticity of LLM serving on shared GPU clusters. “True” elasticity means a serving deployment can acquire or release individual GPUs during operation, without restarting the serving process, without dropping requests, and in a way that the cluster scheduler actually sees — so that freed GPUs become available to other tenants, and new GPUs can be added to meet demand, both at second-to-minute granularity.

Three conditions implicit in this goal:

  • Process survives scaling. No checkpoint-and-restart. The inference endpoint stays up while the resource set changes.
  • Cluster sees the scaling. After scale-down, the freed GPUs must be re-allocatable to other jobs — not just idle inside the container they came from.
  • Serving quality is preserved. Throughput and latency at the new size should be close to what a statically-provisioned deployment of that size would achieve. Brief interruption during the scale event itself is acceptable; a permanent 10× slowdown is not.

2. Motivation

v4 established the general gap between fine-grained GPU scheduling and containerised production. v5 sharpens this into a concrete deployment pattern and argues why the per-GPU container + in-place elastic MoE combination is the specific scenario worth pursuing.

2.1 Why containers at all?

The research question could, in principle, be pursued on bare metal. It is not, for two reasons:

ReasonEvidence
Containers are the production deployment format. Kubernetes has ~92% orchestration market share; AI/ML workloads on K8s are at ~54% and rising. GPU resource accounting in K8s is via nvidia.com/gpu on Pods — a container primitive. AWS EKS, Azure AKS, GCP GKE all ship GPU support through containers. CNCF surveys 2024–2025; v4 Section 1.2.
Containers deliver isolation that shared GPU nodes require. Multi-tenant GPU clusters need fault isolation (9% annualised GPU failure rate observed in Meta Llama 3.1 training on 16K H100s) and security isolation. Both require process / namespace boundaries. The container is the standard mechanism. Meta Llama 3.1 paper; industry deployment norms.

Studying elasticity without containers produces a result that does not transfer to the environments where LLM serving actually runs. The NCCL gates, the cgroup pinning, the namespace isolation — these are production realities, not research-added noise.

2.2 Why multi-GPU containers cannot dynamically adjust GPU count

A single container with multiple GPUs is the dominant deployment pattern today (--gpus all, one Pod per node). This pattern cannot implement the “Cluster sees the scaling” requirement from Section 1, for a reason rooted in how GPU access is granted to a Linux container:

  • NVIDIA Container Toolkit binds devices at launch time. When a container starts with --gpus, the toolkit (nvidia-container-cli) configures the container’s devices cgroup v2 allowlist to permit exactly the requested /dev/nvidia<N> character devices, bind-mounts the NVIDIA driver libraries, and sets NVIDIA_VISIBLE_DEVICES. This configuration is applied by the OCI runtime hook at container creation.
  • Docker has no API to modify device allocations on a running container. docker update can change memory, CPU shares, and a few other runtime limits, but not --device or --gpus. Nor does the OCI runtime spec allow device modification after creation.
  • Kubernetes In-Place Pod Resize (1.27+) does not cover nvidia.com/gpu. In-Place Resize is limited to CPU and memory. Adjusting a Pod’s GPU allocation requires deleting and recreating the Pod.
  • Cgroup v2 device controllers could theoretically be mutated, but this is a privileged operation on the host, not something the container runtime exposes, and it does not update the container’s CUDA visibility (CUDA_VISIBLE_DEVICES and the userspace driver state).

Exp A3 Phase 2 confirmed this experimentally: regardless of what vLLM did internally across a 2→4→2 live rescale, the container’s HostConfig.DeviceRequests.DeviceIDs remained [0, 1, 2, 3] for the entire process lifetime. GPU memory released (nvidia-smi showed ~35 GB → 4 MiB within ~10 s of scale-down), but the container — and therefore the Pod, and therefore the scheduler’s resource accounting — did not.

Conclusion. Multi-GPU containers can host the serving process through a scale event, but they cannot propagate the scaling decision to the cluster scheduler. The freed GPUs are memory-free but container-trapped.

2.3 Two parallelism-adjustment patterns, only one of which needs per-GPU containers

Pattern Examples Does process survive scaling? Multi-GPU container OK?
Stop-restart parallelism adjustment Tenplex (SOSP’24), ByteCheckpoint, Megatron resumable training No. Checkpoint → terminate → restart with new parallelism. Yes. The container can be recreated between runs with a different --gpus set. Scheduler sees clean lifecycle. Exp A1 confirmed this for Tenplex: multi-GPU containers handle stop-restart elasticity without issue.
In-place parallelism adjustment vLLM Elastic EP (v0.17+), NTP (arXiv’25, non-uniform TP), Expert-as-a-Service (arXiv’25) Yes. Communicators are reformed, weights redistributed, and serving continues on the new set of GPUs without process restart. No. The process does not terminate, so the container is never recreated, so the orchestrator-level GPU allocation is never adjusted. Per-GPU containers are required for cluster-level elasticity of in-place systems. Exp A3 Phase 2 confirmed this.

This is the refinement of v4’s argument. Per-GPU containers are not about “finer scheduling in general.” They are the only path to cluster-visible elasticity for in-place systems, because the container allocation is the level at which GPUs are granted and revoked, and in-place systems never give the orchestrator a chance to recreate the container.

2.4 vLLM Elastic EP is the concrete case worth studying

To ground the research in a real system, we need a deployed in-place elastic technique to carry the weight of the study. vLLM’s Elastic Expert Parallelism (shipped in v0.17.0, March 2026) is the natural choice:

  • Production-grade. Part of the dominant OSS LLM serving engine; used by Red Hat, Databricks, Anyscale, and others in production deployments.
  • Genuinely in-place. The POST /scale_elastic_ep endpoint reshapes EP/DP size without restarting the serving process. Exp A3 Phase 1 verified: the process survives 2→4 scale-up (25.8 s) and 4→2 scale-down (4.1 s) with no restart.
  • Mixture-of-Experts is the frontier deployment pattern. DeepSeek-V3/R1, Qwen3-MoE, Llama 4 Maverick are all MoE; EP is the parallelism strategy used to serve them. Elasticity for MoE serving is not a toy use case — it is how future large deployments will scale with traffic.
  • Well-suited to per-GPU containers. Experts are independent computation units that map naturally one-expert-group-per-GPU = one-container. The Expert-as-a-Service paradigm already proposes exactly this disaggregation at the application layer; we study the infrastructure layer that would make it work.
  • We already have a working baseline. Exp A3 Phase 1 gives us 166.6 tok/s native DP=4 performance and verified functional behaviour. Every subsequent result can be compared to this baseline.

The sharpened research question for v5: Can per-GPU containers host vLLM Elastic EP (or a modified variant thereof) with throughput close to native, while enabling the cluster scheduler to actually see the scaling decisions — in a way that can be deployed in a real K8s-style production setting? If yes, how? If not, which specific obstacles are fundamental vs. engineering?

3. What We Have Already Verified (Exp A1 + Exp A3)

v5 does not start from scratch. Two experiments in v4 are complete; their findings constrain v5’s direction. This section summarises what is now known empirically.

3.1 Stop-restart elasticity is solved by multi-GPU containers (Exp A1)

Exp A1 (Tenplex) established that Tenplex’s “elastic” reconfiguration is stop-transform-restart: the training process terminates, state is transformed on the host, and a new process starts with different parallelism. The container can be recreated between runs with a different --gpus set — the scheduler sees clean lifecycle events, no GPU is trapped. This pattern does not need per-GPU containers.

3.2 In-place elastic EP works on native hardware (Exp A3 Phase 1)

Exp A3 Phase 1 confirmed:

  • vLLM Elastic EP v0.19 works end-to-end with Qwen3-30B-A3B on 4× A100-40GB.
  • Throughput scales near-linearly: 87.4 tok/s at DP=2 → 166.6 tok/s at DP=4 (1.91×, 95% efficiency).
  • Scale-up 2→4 in 25.8 s. Scale-down 4→2 in 4.1 s.
  • Service interruption during scaling: ~3.5 s 503 window.
  • After scale-down, GPU memory drops to 4 MiB within ~10 s — CUDA contexts are cleanly released at the process level.

3.3 Multi-GPU container traps the freed GPUs (Exp A3 Phase 2)

Inside a single container with --gpus '"device=0,1,2,3"', the same 2→4→2 cycle works internally (throughput 66.7 → 154.1 → 69.2 tok/s, 7–24% container overhead), but HostConfig.DeviceRequests.DeviceIDs stays at [0,1,2,3] for the entire container lifetime, even after vLLM has internally scaled down and released GPU 2 and 3 memory. An orchestrator sees four GPU requests held indefinitely. This is the motivation argument (Section 2.2) in experimental form.

3.4 Per-GPU containers eliminate the trap but expose new problems (Exp A3 Phase 3)

With four containers (one per GPU), a Docker bridge network, and a Ray cluster spanning the containers, vllm serve at DP=4 runs successfully after one workaround. Each container’s DeviceIDs is just ['N'] — the trap is eliminated. But three new problems emerged:

ProblemWhat we saw
vLLM’s DP-master assumption Default flags fail with ValueError: Not enough resources to allocate 4 DP ranks on DP master node, possible to fit 1. Workaround: --data-parallel-size-local 1.
NCCL falls back to TCP sockets All three NCCL same-node gates (hostHash, shmDev, abstract UDS) fail. Every channel logs via NET/Socket/0. Throughput drops to 127.8 tok/s (−23% vs native), even though GPU 0 and GPU 1 have physical NVLink NV12 between them.
/scale_elastic_ep bricks the service The known EPLB num_redundant ≥ 0 bug combined with 4-container coordination leaves partial Ray actor teardown and unrecoverable state. Process cannot be restarted without tearing down all containers.

The first two problems define the primary technical challenges for v5. The third is a vLLM-level bug orthogonal to our research, but one we must manage operationally.

4. Exploration Fields

Each subsection describes an exploration field: what it is, why it matters, what we have already observed, and a workaround plan — the concrete next experiments and investigations that would move understanding forward. We deliberately do not propose a final system design. The design should fall out of the exploration.

4.1 Recovering NCCL bandwidth across per-GPU containers

4.1.1 Challenge

NCCL’s transport-selection logic decides at communicator init time whether two ranks are “on the same node.” Three independent gates are checked; all three fail in our per-GPU container setup:

GateCheckWhy it fails in Phase 3
hostHash hash(gethostname() + /proc/sys/kernel/random/boot_id) Each container gets a distinct UTS namespace and hostname (ep-rank-0, ep-rank-1, …).
shmDev stat("/dev/shm").st_dev Each container mounts its own /dev/shm tmpfs by default.
IPC socket Abstract UDS for cuMem FD passing (used to exchange CUDA IPC handles). Abstract UDS are network-namespace-scoped; each container has its own netns.

Any single failure is enough to downgrade NCCL to the NET (sockets) transport. On our hardware, this converts a 600 GB/s NVLink NV12 link into a ~10 GB/s loopback TCP link and costs 23% throughput on Qwen3-30B-A3B at DP=4.

4.1.2 Workaround plan

  1. Apply the LD_PRELOAD shim from Exp A (common/shim/, 261 lines of C99) to each per-GPU container. The shim intercepts gethostname(), stat(), bind(), and sendmsg() to make all three gates pass. Exp A’s validation showed 100% NVLink bandwidth recovery (156.8 GB/s AllReduce, 262.4 GB/s raw P2P) for the NCCL benchmark. It has not been validated with vLLM.
  2. Rerun Exp A3 Phase 3 with the shim injected. Measure: (a) NCCL transport selection (expect NET/IB, P2P/IPC, or similar NVLink-capable transport instead of NET/Socket/0); (b) throughput at DP=4 (target: close to Phase 2’s 154.1 tok/s or Phase 1’s 166.6 tok/s); (c) scale-up weight transfer time (in Phase 1 this was ~50–200 ms per new engine; in Phase 3 without shim it would be seconds via TCP).
  3. Characterise which gates are actually load-bearing. The shim spoofs all three; in practice NCCL may select NVLink if any one of them is made to pass. Stripping the shim down to the minimal intercept set tells us which gate is the operative barrier, and by extension what the minimal container-runtime changes would look like (instead of LD_PRELOAD).
  4. Check whether vLLM’s additional communication paths (weight transport, EPLB reshuffle coordination) also benefit. NCCL gates affect all NCCL calls, but vLLM also uses Gloo (CPU collectives) and a TCP store for stateless-group coordination; those are already TCP-based and unaffected. Verify that only the NCCL data-path is the bottleneck.

This is the highest-priority exploration because Phase 3 already demonstrated the loss is large and the shim is known to recover bandwidth in simpler benchmarks. If it does not work with vLLM, that in itself is a finding and redirects the research.

4.2 Coordinating container lifecycle with vLLM’s scaling state machine

4.2.1 Challenge

vLLM today expresses scaling as an HTTP call that internally orchestrates Ray actor creation/destruction. The caller’s assumption is “the Ray cluster stays, vLLM manipulates actors inside it.” The per-GPU-container design wants the inverse: the container lifecycle is the primitive. Scale-down is “stop container ep-rank-3”; scale-up is “start a new container.”

The two views are not equivalent:

  • Under the current HTTP model, vLLM destroys a Ray actor but the underlying Ray worker (and its container) keeps running, just idle. Scheduler still sees the GPU allocated. We saw this in Phase 3: after attempted scale-down, containers remained Up even though vLLM had torn down engines.
  • Under the container-lifecycle model, the external action is docker stop ep-rank-3. Ray node detaches abruptly. vLLM’s existing elastic EP code does not currently handle “a Ray node just went away unexpectedly” — it orchestrates its own actor teardown first.

4.2.2 Workaround plan

  1. Map vLLM’s scaling state machine onto container lifecycle events. Re-read the scale-down path in vllm/distributed/elastic_ep/elastic_state.py: which steps can be driven by an external actor that ALSO controls container lifecycle? Which steps strictly require vLLM to be in charge?
  2. Experiment with a thin external coordinator. The loop: (a) call /scale_elastic_ep to 2; (b) after the API completes, docker stop the two departing containers. Measure: does this produce a clean cluster-visible scale-down? Does it work repeatedly? Can we do the inverse order (stop first, then call API)?
  3. Study whether Ray’s graceful node-removal can integrate with vLLM. Ray supports node shutdown with drain semantics (ray.shutdown() on a worker). If vLLM’s reconfiguration code reacts to Ray node events the same way as it reacts to actor failures, we might be able to use Ray events as the trigger, with vLLM re-forming communicators from the surviving set.
  4. Prototype a “container stop” that behaves like a controlled actor exit. Send a signal to the Ray worker to detach cleanly, wait for vLLM to finish its reconfiguration, then terminate the container. Measure service interruption compared to HTTP-API scale-down.

The output of this exploration is an understanding of which scaling operations can be expressed as container lifecycle events without modifying vLLM, which require small patches, and which require larger design changes.

4.3 Auditing vLLM’s hidden multi-GPU-master assumptions

4.3.1 Challenge

Exp A3 Phase 3 surfaced one such assumption: the CLI rejected --data-parallel-size 4 with “Not enough resources to allocate 4 DP ranks on DP master node, possible to fit 1 DP ranks.” The workaround (--data-parallel-size-local 1) got us past it, but this hints at a broader pattern: vLLM’s elastic EP was designed and tested against single-node multi-GPU deployments. How many other design-time assumptions about topology are baked in where they will surface only in per-GPU container deployments?

Examples of assumptions that might (or might not) exist:

  • Does scale-up assume the new Ray actor can be created on the same node as an existing one for weight transport? (We know weight transport uses GPU P2P; in a per-GPU setup the “same node” pairing doesn’t exist.)
  • Does the KV cache memory synchronisation (SYNC_KV_CACHE_MEMORY_SIZE state) assume all engines have similar topology? Our heterogeneous SXM4+PCIe-A100 setup might expose NUMA/PCIe differences that were invisible with all GPUs in one container.
  • Does the EPLB load-balancer assume collocated ranks for efficient telemetry collection?
  • Does the stateless-group TCP store coordinator scale to 4, 16, 64 rank-containers the same way it scales within a single process?

4.3.2 Workaround plan

  1. Systematic source audit. Read the Ray-related code paths in vLLM’s elastic EP for assumptions about rank co-location, file-system visibility (shared model dir), or host-level resource expectations. Produce a short “per-GPU container deployment checklist”: which flags / env vars must be set, which features are incompatible.
  2. Stress test at larger DP. 4 containers is too small to expose coordinator scaling. Try DP=8 or DP=16 on hardware that can host it (another machine, cloud instance) and see what breaks at larger rank counts.
  3. Document the delta between in-container config and per-container config. An operator deploying vLLM on K8s needs to know exactly which settings differ from a single-node deployment. The audit produces this delta.

4.4 Understanding EPLB robustness and the cold-start scale-down path

4.4.1 Challenge

vLLM’s EPLB rebalance code has a logical invariant (num_physical ≥ num_logical) that is violated when a cold-started DP=N deployment with num_redundant_experts=0 tries to scale down. The assertion fires and the serving process does not recover. We hit this in Phase 1 (documented workaround: start at DP≤N/2, scale up first, then scale down freely). In Phase 3 the same bug bricked the multi-container cluster because recovery requires coordinated tear-down across 4 containers.

For any production deployment, two questions arise: (a) is this a vLLM bug that will be fixed upstream, and if so when? (b) what are the safe-deployment rules until it is fixed?

4.4.2 Workaround plan

  1. Trace the EPLB rebalance code end to end. Identify exactly which code path derives the new num_physical from the old one during scale-down, and under what inputs the invariant holds. This is already partially mapped in the analysis report (num_replicas = num_replicas // ep_group.size() * num_gpus).
  2. Experiment with --eplb-config.num_redundant_experts > 0. Redundancy allows scale-down to consume redundant copies rather than drop logical experts. Verify scale-down 4→2 with e.g. num_redundant_experts=4 succeeds from a cold DP=4 start. Measure: memory cost of redundancy, throughput cost, whether the bug is actually side-stepped.
  3. Check upstream issue tracker. If this is known, find the fix PR. If not, file a minimal repro and propose a fix (we have the full stack trace and a tight repro from Exp A3 Phase 1). An upstream fix would remove one constraint on our deployment; if it is not forthcoming, our research must explicitly document the workaround.
  4. Test recovery from a broken scale event. When the assertion fires, what is the minimal set of actions to restore service? (Restart containers? Restart only the master? Wipe Ray state?) This determines the operator-side failure-handling logic for any future system.

4.5 Characterising cross-container scale-up weight transport

4.5.1 Challenge

Scale-up in vLLM Elastic EP streams non-expert model parameters (~28 GB for Qwen3-30B-A3B) from existing engines to new engines over the NCCL channel (see Analysis Report Section 6.2). In native and multi-GPU-container deployments, this uses NVLink P2P and takes sub-second in theory and single-digit seconds in practice. In per-GPU containers without the shim, this transfer uses TCP sockets — potentially tens of seconds per new rank.

We have not yet isolated the weight-transfer phase in Phase 3 timing. The 39 s Phase-2 scale-up includes Ray actor spawn, weight transfer, EPLB reshuffle, and group switching; only a small fraction is likely pure transfer time.

4.5.2 Workaround plan

  1. Instrument the scale-up state machine. Add timing probes at each state transition (WAIT_NEW_CORE_ENGINES_INIT, CREATE_STANDBY_GROUPS, TRANSFER_EXPERT_MAPPING, TRANSFER_WEIGHTS, …) either by log parsing (existing INFO lines include timestamps) or by a small vLLM patch. Report time spent in each phase for all three phases of Exp A3.
  2. Measure weight transfer with and without the shim. After Section 4.1 workarounds are in place, re-time scale-up in Phase 3. Expected: without shim, transfer phase dominates scale-up time and explains much of the overhead; with shim, transfer approaches Phase 2 timing.
  3. Consider alternative transfer paths if NCCL-over-TCP is unworkable. The mounted model volume is visible to every container — a new rank could load non-expert weights from disk instead of receiving them over NCCL. This trades transfer time for I/O time and requires a vLLM patch, but it is worth understanding as a fallback if NCCL-over-shim does not recover enough bandwidth.

5. Exploration Sequence and Decision Points

The exploration fields above are not independent. A suggested order, with explicit decision points:

Iteration Focus What we learn Decision it unblocks
1 4.1 NCCL bandwidth recovery (shim + vLLM) Does the Exp A shim make NCCL pick NVLink across per-GPU containers in a vLLM workload? What fraction of the 23% loss is recovered? If yes: per-GPU containers become viable from a performance standpoint; proceed. If no: investigate why vLLM’s NCCL usage differs from the benchmark, possibly deeper modifications.
2 4.5 Scale-up weight transport (instrumented) How much does scale-up slow down in per-GPU containers? Can the shim recover it? Determines whether dynamic scale-up is fast enough for second-latency scheduling, or only fit for minute-scale rebalancing.
3 4.2 Container lifecycle as scaling primitive Can scaling be driven from the outside, with vLLM just reacting? What is the minimal coordination required? Decides whether the system design should wrap vLLM behind an operator that manages containers, or modify vLLM to accept lifecycle-event triggers directly.
4 4.3 Assumption audit + 4.4 EPLB robustness What deployment rules and config deltas are required; which vLLM bugs block us; what production guardrails are needed. Completes the catalogue of known-obstacles; everything remaining is the actual research-system problem.

Stopping condition. If Iteration 1 (NCCL + shim + vLLM) does not recover most of the 23% loss, the per-GPU container direction is in trouble and we reassess. If Iteration 2 shows scale-up time dominated by something neither the shim nor alternative transport can help (e.g., a fundamental Ray actor cold-start latency), we may need to rethink the scaling granularity. If Iteration 3 shows container lifecycle can not be coordinated with vLLM without invasive patches, the system design implications narrow substantially.

6. Non-Goals for This Phase

To keep v5 focused, the following are deliberately out of scope until the exploration in Section 4 produces results:

  • No new system design. We do not propose a Kubernetes operator architecture, a new serving engine, or a modified NCCL replacement in this plan. The exploration is supposed to reveal what the design should be. Writing a design first would bias the experiments toward confirming it.
  • No new NCCL shim. Exp A’s existing shim is the candidate; if it does not suffice, we investigate why, but we do not build a replacement.
  • No new model. Qwen3-30B-A3B is the reference model throughout. Changing models would muddle the comparisons with v4’s A1/A3 baselines.
  • No multi-node cluster. Single node (node192, 4× A100-40GB) is sufficient to study the per-GPU container question because NCCL gates are about the container boundary, not the physical host. Multi-node introduces orthogonal concerns (network topology, cross-host NVLink domains) that should be tackled separately.
  • No production-readiness claims. If the exploration succeeds, the next phase plans a productisation path; that is not in v5.
