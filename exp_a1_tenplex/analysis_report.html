1. Objective

Tenplex (SOSP ’24) is the state-of-the-art open-source system for elastic GPU scaling. It reconfigures tensor, pipeline, and data parallelism dimensions between training stages, preserving training progress across GPU allocation changes. This experiment reproduces Tenplex’s elastic scheduling on our hardware to answer two questions: (1) what is the actual mechanism — is reconfiguration in-place or stop-and-restart? and (2) do “freed” GPUs become available to the cluster after a scale-down event?

Result: Tenplex uses multi-GPU Docker containers internally, and when it scales from 4 GPUs to 2 GPUs, the container retains --gpus device=0,1,2,3 — GPUs 2 and 3 sit idle inside the container but remain unavailable to the cluster. Framework-level elasticity does not translate to cluster-level elasticity under the multi-GPU container model.

2. Background

2.1 Tenplex’s Elastic Scaling — Not In-Place

Tenplex enables elastic GPU scaling for distributed training via Parallelizable Tensor Collections (PTCs). The Tenplex paper describes this as reconfiguring TP×PP×DP parallelism “without restart,” which might suggest in-place elasticity — dynamically adding or removing GPU processes to a running training job. This is not what Tenplex does.

Source code analysis reveals that Tenplex’s reconfiguration is a stop–transform–restart cycle: the training process is fully terminated, model state is transformed on the host to match the new parallelism layout, and a new training process starts from the transformed state. The “without restart” claim refers to the fact that training progress is not lost (the iteration counter continues), not that the process remains running. In practice, Tenplex is a sophisticated checkpoint/restore system with two key optimizations over naive checkpoint/restore:

  1. PTC state transformation: Instead of saving a full checkpoint and reloading it with a different parallelism config (which requires resharding tensors at load time), Tenplex runs tenplex-state-transformer to directly convert the tensor layout from source parallelism to target parallelism. The paper reports ~122s for a 16→8 GPU resize.
  2. Orchestrated scheduling: tenplex-run automates the full lifecycle — stopping training, running state transformation, restarting with new parallelism — driven by a scaling schedule.

Neither optimization changes the fundamental architecture: every reconfiguration involves a full process stop and restart. There is no mechanism to add or remove GPU processes from a running training job.

2.2 Tenplex’s Architecture — A Key Discovery

Prior to running this experiment, we assumed Tenplex executed training as bare-metal processes. Source code analysis revealed that Tenplex uses Docker containers internally. The tenplex-run CLI is a Go orchestrator that:

  1. SSHes to each host in the cluster
  2. Runs docker run --network host --gpus device=0,1,2,3 ... to create training containers
  3. Executes torchrun with Megatron-LM inside each container
  4. Manages the scaling schedule by stopping and recreating containers at each scaling point

This means Tenplex already operates within a multi-GPU container model — the very model that production Kubernetes deployments use. The question becomes: does this model support cluster-level elasticity, or does it trap GPU resources?

2.3 Container Lifecycle During Reconfiguration

Source code analysis of tenplex-run/runop/runop.go:ScalingTraining() reveals that every parallelism change follows this sequence:

  1. Training receives a stop signal via an HTTP endpoint (/stop)
  2. Training process exits; the container auto-removes (docker run --rm)
  3. tenplex-state-transformer repartitions PTC state on the host (when Tenplex mode is enabled); or checkpoint is simply saved to disk (when running without Tenplex mode)
  4. A new container starts with docker run --gpus device=... and the updated parallelism config
  5. The new container loads the transformed state (or checkpoint) and resumes training

The following diagram illustrates the two modes: 

Full Tenplex mode:
  train (container A) → stop → PTC state transform on host → start (container B) → resume

Without Tenplex (--no-tenplex, what we ran):
  train (container A) → stop + save checkpoint → start (container B) → load checkpoint → resume

In both modes, the container is fully destroyed and recreated. There is no mechanism to change a running container’s GPU assignment — the --gpus flag is fixed at container creation time, a fundamental Docker constraint.

5.1 Container Lifecycle

Docker container events were captured via docker events --filter 'name=trainer'. Each scaling event follows an identical pattern: the old container dies, is destroyed (--rm), and a new container is created and started.

The container recreation gap — from old container destruction to new container creation — is ~1 second in both transitions. Docker overhead for container lifecycle is negligible. The real startup cost is inside the container: CUDA kernel compilation and model initialization take ~2–3 minutes on the first run of each stage.

5.2 Training Throughput

Stage GPUs Parallelism Iterations Mean Iter Time Min / Max
Stage 0 4 TP=2, DP=2 ~290 947.6 ms 938.2 / 1009.8 ms
Stage 1 2 TP=2, DP=1 ~430 738.7 ms 698.7 / 1775.1 ms

Stage 1 (2 GPUs) has a lower per-iteration time (738.7 ms vs 947.6 ms) because eliminating data parallelism removes the AllReduce gradient synchronization. However, with half the GPUs, the effective training throughput in tokens/s is lower: each iteration still processes a global batch of 32 samples, but with 2 GPUs each handling micro-batch = 4 × 4 accumulation steps, versus 4 GPUs with no accumulation.

Checkpoint continuity was maintained: Stage 1 loaded the checkpoint saved at iteration 299 by Stage 0, and continued training from iteration 300. Stage 2 loaded the checkpoint at iteration 721 from Stage 1.

5.3 GPU Assignment — The Trapping Problem

This is the central finding. We extracted the exact docker run commands from each stage’s log:

Stage Active GPUs --gpus Flag --nproc_per_node Idle GPUs
Stage 0 0, 1, 2, 3 device=0,1,2,3 4 None
Stage 1 0, 1 device=0,1,2,3 2 GPU 2, GPU 3
Stage 2 0, 1, 2, 3 device=0,1,2,3 4 None

In Stage 1, Tenplex scales the training workload from 4 GPUs to 2, but the Docker container still holds all 4 GPUs via --gpus device=0,1,2,3. GPUs 2 and 3 are completely idle — no training process is assigned to them — yet they remain allocated to the container and are invisible to the cluster scheduler.

This occurs because Tenplex’s createWorkers() function (tenplex-run/job/job.go) computes GPU assignments from the gpu-per-host and gpu-per-container flags, which are global constants — not adjusted per scaling point. The number of processes (nproc_per_node) changes with parallelism, but the container’s device allocation does not.

5.4 Stage 2 Failure — OOM on Scale-Up

Stage 2 attempted to scale back to 4 GPUs but crashed with a CUDA out-of-memory error on GPU 2: 

torch.cuda.OutOfMemoryError: CUDA out of memory.
Tried to allocate 64.00 MiB. GPU 2 has a total capacity of 39.49 GiB
of which 51.12 MiB is free.

GPU 2 had 39.49 GiB total but only 51 MiB free at allocation time, despite the previous container being destroyed (--rm). This suggests that during the ~1s gap between containers, another process on the node claimed GPU 2’s memory. This highlights a real-world risk in multi-tenant environments: GPUs released during scale-down can be seized by other workloads, making scale-up unreliable.

6. Analysis

6.1 Why GPUs Are Trapped

The GPU trapping problem emerges from the interaction of three design decisions, each individually reasonable:

  1. Docker’s GPU allocation is immutable. The --gpus flag is set at container creation and cannot be changed at runtime. Docker provides no API to release a subset of GPUs from a running container. This is a fundamental constraint of the container runtime.
  2. Tenplex uses a fixed container GPU configuration. The gpu-per-container parameter is a global constant (4 in our setup). When Tenplex scales to 2 GPUs, it reduces nproc_per_node from 4 to 2 but does not change the container’s device allocation. This is a Tenplex design choice — the container is treated as a resource envelope, not a precise allocation.
  3. Kubernetes reinforces the trap. In production, the NVIDIA device plugin allocates GPUs to pods at scheduling time. Even if the training process frees GPUs internally, the pod’s cgroup still holds them. K8s does not support partial GPU deallocation from a running pod. The in-place pod resize feature (KEP-1287) explicitly excludes GPU resources.

The result is a disconnect between framework-level elasticity (Tenplex can resize) and cluster-level elasticity (freed GPUs are not reclaimable). From the cluster scheduler’s perspective, all 4 GPUs remain “in use” for the entire duration of the training job, regardless of how many the job actually needs at any given moment.

6.2 Quantifying the Waste

In our experiment, Stage 1 ran for 606 seconds with 2 GPUs idle. This represents:

  • 2 GPU × 606 s = 1,212 GPU-seconds of waste (GPUs powered on, memory allocated, but doing no work)
  • As a fraction of total GPU-time across all stages: 1,212 / (4 × 1,530) = 19.8% of GPU resources wasted

In Tenplex’s paper benchmarks, which run on 16–64 GPUs with more frequent scaling events from the Microsoft Philly trace, the waste fraction would be proportionally larger during scale-down periods.

6.3 The Reconfiguration Overhead Breakdown

Each scaling event incurs the following costs:

ComponentTimeNotes
Container destroy + create ~1 s Docker overhead is negligible
CUDA kernel compilation ~30 s JIT compilation of fused kernels (first run per container)
Model initialization ~90 s Weight allocation, optimizer state, data loader setup
Checkpoint load <5 s Loading 350M-param checkpoint from local NVMe
Total per reconfiguration ~2–3 min GPUs idle during entire period

The dominant cost is not Docker or state transformation — it is the cold start of a new PyTorch process inside the container. This cost would persist even with per-GPU containers, as each new container must compile CUDA kernels and initialize the model.

Limitation: This experiment used --no-tenplex mode, which replaces PTC state transformation with standard checkpoint save/load. The full Tenplex mode adds a state transformation step (reported as ~122s in the paper for 16→8 GPUs) but eliminates the need for checkpoint save/load. The container lifecycle and GPU trapping behavior are identical in both modes.

6.4 Does Tenplex Motivate Per-GPU Containers?

The GPU trapping observed in Section 5.3 is not a fundamental limitation of multi-GPU containers. It is an engineering detail in Tenplex’s implementation: the gpu-per-container parameter is hardcoded rather than adjusted per scaling point. A straightforward fix — creating the Stage 1 container with --gpus device=0,1 instead of device=0,1,2,3 — would release GPUs 2 and 3 to the cluster. This is purely engineering work, not a research contribution.

More fundamentally, Tenplex’s stop-transform-restart architecture means that multi-GPU containers can support elastic scheduling perfectly well. The sequence is: stop the old container → transform state → start a new container with a different GPU count. The container boundary adds no friction because the process is already stopped. The optimization opportunities lie in checkpoint/restore efficiency and cold-start reduction, not in container granularity.

Per-GPU containers are motivated by in-place elasticity, not stop-restart elasticity. If training processes remain running while GPUs are added or removed — without stopping and restarting the entire job — then Docker’s immutable --gpus flag becomes a genuine barrier. A running container cannot release or acquire GPUs. Per-GPU containers solve this: stopping a container immediately frees its GPU, while surviving containers continue training uninterrupted.

The In-Place Elasticity Landscape

Tenplex is not in-place elastic. How close are other systems?

System Venue Elasticity Type GPU Count Changes? Motivates Per-GPU?
Tenplex SOSP ’24 Stop-restart Yes No — multi-GPU containers suffice
Gyges arXiv ’25 In-place (inference) No (TP1↔TP4, fixed GPU count) Partial — in-place but no true elasticity
ElasWave arXiv ’25 In-place (training) Yes (online resharding) Yes — dynamic communicator, sub-second recovery
Elastic Horovod — In-place (DP only) Yes (workers join/leave) Partial — DP only, no TP/PP
LoongServe SOSP ’24 In-place (inference, SP) No (sequence parallelism only) Partial — different parallelism axis
TorchElastic — Stop-restart Yes No — full worker group restart

ElasWave is the strongest candidate for motivating per-GPU containers: it performs online elastic recovery from in-memory state with a dynamic communicator that edits communication groups in-place, achieving sub-second recovery time. Its training processes keep running while GPUs are added or removed. Under a multi-GPU container model, this capability would be blocked by Docker’s immutable GPU assignment. Under a per-GPU container model, adding a GPU means starting one new container; removing a GPU means stopping one container — while all other containers continue training.

Gyges demonstrates that in-place parallelism transformation is feasible for inference (TP1↔TP4 with zero downtime), but does not change the GPU count. It motivates the mechanism but not the full elasticity story.

Key insight from Exp A1: Stop-restart elastic systems like Tenplex do not motivate per-GPU containers — multi-GPU containers handle them adequately. To motivate per-GPU containers, we need in-place elastic scheduling where training processes survive GPU allocation changes. ElasWave (online resharding, dynamic communicators) is the strongest open-source candidate for this direction.

7. Conclusions

  1. Tenplex is not in-place elastic — it is stop-transform-restart: every parallelism reconfiguration fully terminates the training process, destroys the container, and creates a new one. Tenplex’s contribution over naive checkpoint/restore is the PTC state transformation (converting tensor layouts between parallelism configs), not in-place process management.
  2. GPU trapping is an implementation detail, not a fundamental limitation: when scaling from 4 to 2 GPUs, Tenplex’s container retains all 4 GPU devices (19.8% waste in our schedule). However, since Tenplex stops and recreates the container anyway, the fix is trivial: create the new container with --gpus device=0,1. Multi-GPU containers can support stop-restart elasticity without GPU trapping.
  3. Stop-restart elasticity does not motivate per-GPU containers: since the training process is fully stopped at each scaling event, changing the container’s GPU allocation is straightforward. The optimization opportunities are in checkpoint/restore speed and cold-start reduction, not container granularity.
  4. Per-GPU containers are motivated by in-place elasticity: if training processes survive GPU allocation changes (as in ElasWave’s online resharding or Gyges’s in-place parallelism transformation), Docker’s immutable --gpus flag becomes a genuine barrier that per-GPU containers resolve.
  5. Container recreation is fast (~1s), but cold start is not: the real reconfiguration overhead is ~2–3 minutes of CUDA compilation and model initialization per scaling event, during which all assigned GPUs are idle.
  6. Scale-up is fragile in multi-tenant environments: the 1s gap between container destruction and recreation is enough for other processes to claim GPU memory, causing OOM failures when scaling back up.

Next steps: Tenplex’s stop-restart model does not motivate per-GPU containers. To find systems that do, we should investigate in-place elastic scheduling systems — where training processes survive GPU allocation changes. Candidates: (1) ElasWave (online resharding with dynamic communicators, sub-second recovery), (2) Gyges (in-place TP transformation for inference, fixed GPU count), and (3) Elastic Horovod (in-place DP worker join/leave). An in-place elastic system running in per-GPU containers — where scaling is expressed as container lifecycle — would be the strongest demonstration of the per-GPU container value proposition. See updated research plan (Section 4.2) for revised candidate analysis.
