for In-Place Elastic MoE Serving

1. Objective

Exp A3 Section 5.7 reported a −24% output-throughput gap between the native and multi-GPU-container (MGC) regimes for vLLM Elastic EP at DP=4 (native 171.23 tok/s → MGC 130.62 tok/s), along with a 4.6× TTFT blow-up (387 ms → 1777 ms). Those numbers are surprising: same host, same GPUs, same driver, same NCCL — the only difference is that the serve process runs inside a Docker container rather than directly on the host. The stated goal of Exp C is to identify the root cause(s), with particular focus on the disproportionate TTFT blow-up, and to either provide a concrete configuration fix or a principled finding about containerization cost.

The investigation was organized around five falsifiable hypotheses: H1 (cold TTFT / compile), H2 (NCCL init overhead), H3 (--shm-size sensitivity), H4 (--ipc=host as load-bearing), and H5 (bench-variance as explanation of the headline). The result splits into two findings — one about the headline itself and one about the container regime.

Result: The −24% "gap" is a measurement artefact, not a container overhead. When both regimes are given enough warm-up to reach steady state (2 benches for native, 2–5 for MGC), DP=4 plateau TPOT is 89.66 ms native vs 87.67 ms MGC (n=13 vs n=18 post-plateau measurements, 0.77σ apart — not statistically distinguishable). The real container overhead is a bounded one-time cost: first TTFT after scale-up is ~1400 ms inside the container vs ~540 ms native, collapsing to ~200 ms in both regimes by bench 2.

2. Background

Exp A3 established that vLLM Elastic EP works identically inside a multi-GPU container: DP=2→4→2 scale events succeed, expert reshuffle closes cleanly, and NCCL picks the same-node transport (per §5.5.1 / §5.5.2). The gap it observed was not a correctness or functionality problem — it was a throughput shortfall at DP=4.

The headline evolved: initial measurement (§5.5.1) reported −7.5% at DP=4; a same-session re-run (§5.7) reported −23.7%. Both used the cycle command, which runs one DP=4 bench at a single point in time (bench 1 at DP=4, right after scale-up). That single-shot cold measurement is the pivot that Exp C ended up probing.

All numbers here use the exact same model, bench shape, and CLI flags as Exp A3 §5.7: Qwen3-30B-A3B, 32 prompts at concurrency 16 at DP=4, --random-input-len 128 --random-output-len 128, vLLM 0.19.0 with --enforce-eager (no torch.compile, no CUDA graph capture). Only the bench count differs: Exp A3's cycle runs one DP=4 bench; Exp C runs 2 warm-up + 5–10 measurement benches back-to-back so TPOT actually reaches a plateau before being measured.

3.3 Protocol

Two distinct protocols, used for different purposes:

  • Cycle protocol (inherited from Exp A3): start → bench DP=2 → scale 2→4 → bench DP=4 → scale 4→2 → bench DP=2 → stop. This is what §5.7 used. Exp C's H5 investigation ran three cycles per regime to measure run-to-run variance of this protocol.
  • Plateau protocol (new in Exp C): start → scale to DP=4 → 2 back-to-back warm-up benches (discarded) → 5–10 consecutive measurement benches → stop. Each bench is n=32 prompts at concurrency 16 with random 128/128 input/output lengths. Used to establish steady-state TPOT rather than cold-bench-1 TPOT.

All runs were protected by pre-flight GPU-idle checks (require_gpus_free from exp_a3_vllm_ep/scripts/common.sh); each run's directory contains run_meta.txt with pre- and post-run nvidia-smi snapshots confirming no external GPU activity during the measurement window. node192 is a shared host; several runs during the investigation were aborted or deferred when an external user's workload appeared on a GPU mid-session.

Two findings already visible from this variance pass:

  1. The −24% headline is itself a noisy sample. The session-2 gap reproduced at −14%, not −24%. Native DP=4 mean (148.91) sits 2.5σ below §5.7's single anchor (171.23).
  2. Cycle-protocol CV is 6–10%. Too noisy to reliably measure a single-digit-% container overhead, but clearly real if the overhead were 20–25%.

The 2.33σ separation made it look like the gap was "real but smaller than claimed." Sections 5.2–5.3 show why that interpretation was wrong too.

5.2 H1: Cold TTFT blow-up is real and is one-time

One cycle + two extra benches per regime at DP=4: a warm-up bench at out=128, then a long-output bench at out=1024. If H1's "TTFT dominates the gap" claim were correct, the out=1024 bench should amortise the TTFT overhead and shrink the gap.

But the throughput gap does not shrink with output length: the warm out=128 gap is −14.0% (155.12 vs 133.43); the warm out=1024 gap is −17.3% (87.06 vs 71.98). That initially looked like a persistent steady-state gap with a TPOT mechanism: warm TPOT was native 100.93 vs MGC 117.71, a +16.6% container penalty. Section 5.3 shows that interpretation was wrong.

5.4 H2: NCCL init duration is identical in both regimes

Scale-up 2→4 timings extracted from serve.log and container.log of the native_n3 and mgc_n10 runs:

The NCCL-communicator init itself is identical in wall-clock between regimes. The 14 s container penalty on scale-up is entirely in Ray's placement-group / actor-spawn / broadcast phase — the post-NCCL synchronization. H2's "NCCL init is slower inside the container" prediction is not supported.

This is consistent with Exp A3 §5.5.2's prior observation of "Ray actor spawn slower in container (+51%)" at scale-up. The 14 s penalty is real but doesn't affect steady-state throughput; it is an operational cost borne once per scale event.

6. Analysis

6.1 Why the cycle protocol produces a phantom gap

The cycle command bench sequence at DP=4 is a single 32-prompt bench immediately after the scale 4 call completes. In the plateau framework, this is bench 1 at DP=4. Both regimes are still far from plateau at bench 1:

  • Native bench 1 TPOT: ~110 ms (plateau ~90 ms — ~22% above)
  • MGC bench 1 TPOT: ~110–120 ms (plateau ~88 ms — ~25–35% above)

Cross-session variance on that single-shot cold read is high because the warm-up curve depends on prior host state: an immediately preceding run leaves GPU clocks high and CPU caches warm, producing a quicker warm-up; a long-quiet host gives a slower warm-up. In the §5.7 session, native happened to land close to plateau at bench 1 (171.23 tok/s, TPOT 91.1 ms — near-plateau) and MGC happened to be farther from plateau (130.62 tok/s, TPOT 109.4 ms — distinctly cold-ish). Subtracting the two produces a −24% "gap" that is really two warm-up-position readings, not two plateau readings.

The §5.5.1 headline (−7.5% at DP=4) is consistent with the same mechanism: that session happened to catch both regimes more similarly positioned on their warm-up curves.

6.2 The real container costs, and what they aren't

Containerization does impose costs on vLLM Elastic EP in this configuration — they're just not per-token costs:

CostMagnitudePerAmortises over
Container cold-start wall-clock +44 s vs native (~110 s vs ~66 s) container launch serving lifetime (unbounded)
Scale-up 2→4 duration +14 s (~41 s vs ~27 s) scale event post-scale steady-state traffic
First-bench (post-scale) TTFT +~850 ms (~1400 ms vs ~540 ms) first request after cold/scale bench 2 and later requests
Per-token TPOT −2 ms (MGC faster) / 0.77σ — no effect — —

None of the container-side costs are unbounded; all of them amortise away within seconds of steady-state serving. For long-lived production workloads, the container imposes no measurable steady-state throughput penalty relative to running on the host directly.

6.3 Why MGC is slightly faster per token at plateau

The 2 ms / 2.2% TPOT advantage is not statistically significant (0.77σ), but it is directionally consistent across both replicates. Two plausible (and non-mutually-exclusive) explanations:

  1. Sized /dev/shm with guaranteed lower bound. --shm-size=16g pre-allocates a dedicated shm region backed by the container's own tmpfs. Native's /dev/shm is shared with every other process on the host, subject to external eviction pressure. Under light host load the difference is nil, but it's the kind of configuration that could be a slight positive under contention.
  2. Ray plasma's object_store_memory sizing happens to select a better path at 16 GB. The shm=64m variant plateaus 5 ms higher (93.51 vs 87.67), suggesting 64 MB pushes Ray toward a slower fallback path. 16 GB is comfortably above that threshold; the host's 252 GB /dev/shm would plausibly route the same way native does.

The more interesting finding from the shm sweep is warm-up speed vs plateau height trade-off: smaller shm plateaus faster (single-bench) but a few ms higher; larger shm takes 2–5 benches but plateaus lower. This is a second-order effect (<7%) compared to the headline clarification, but it is a real configuration knob for operators who care about cold-start latency.

6.4 Source of the long MGC warm-up trajectory

The 2–5 bench window during which MGC TPOT descends from ~110 ms (bench 1) to ~88 ms (plateau) is container-specific — native plateaus after one warm-up. Candidate causes (not individually pinned down in this investigation, but narrowed):

  • Ruled out: NCCL communicator init (identical duration in both regimes); MoE-config lookup (the Config file not found at .../E=64,N=768,device_name=NVIDIA_A100-SXM4-40GB.json warning fires identically in both — both use the default MoE config); torch.compile / CUDA graph capture (both disabled by --enforce-eager).
  • Likely candidates: Triton JIT kernel cache state (fresh container ⇒ empty cache ⇒ first forwards JIT-compile MoE dispatch kernels); page-cache warmup of bind-mounted weight files; Ray plasma initialization (specifically the "lazily format a large shm region" hypothesis introduced in the shm=16g-vs-64m analysis).

Distinguishing between these would require a dedicated investigation (profile a single vLLM request end-to-end under py-spy / perf, compare between regimes). Given that the cost is bounded and the steady-state gap is zero, the incremental value of pinning this down is low; leaving it as a follow-up.

6.5 Methodology limitations

Limitations: (1) All data is from a single shared host (node192); warm-up behaviour may be host-specific (CPU/GPU idle-clock ramping, tmpfs backing store, kernel version). (2) n=3 cycles per regime for H5 variance calibration and n=13/n=18 plateau measurements per regime is enough to show a ~2 ms difference is not significant, but not enough to confidently claim MGC is actually faster. The "no measurable gap" conclusion is robust; the sign of the difference is not. (3) Only DP=4 out=128 was studied in depth. Longer-output workloads (out=1024) were sampled once per regime and showed a similar proportional gap (−17%) pre-plateau, but no 10-bench plateau measurement was taken at that shape. (4) Only multi-GPU container was studied. The per-GPU-container regime (Exp A3 §5.6) has fundamentally different mechanics (NCCL falls to NET/Socket/0 at container boundaries) and this warm-up finding does not modify the −39.6% gap there.

7. Conclusions

  1. The −24% DP=4 throughput gap in §5.7 is a measurement artefact. When both regimes are run long enough to reach steady state, native plateaus at 89.66 ms TPOT (n=13) and MGC plateaus at 87.67 ms TPOT (n=18) — 0.77σ apart, statistically indistinguishable.
  2. The cold TTFT blow-up is real but one-time. First TTFT after scale-up is ~1400 ms in MGC vs ~540 ms native (2.6×), collapsing to ~200 ms in both regimes by bench 2. Amortised over any realistic serving session, it disappears.
  3. NCCL communicator init is identical between regimes (19 s per scale event). The 14 s container scale-up penalty is in Ray's placement-group / broadcast phase, not NCCL. H2 is closed as minor and mis-labelled.
  4. --shm-size trades warm-up speed against plateau height. shm=64m plateaus in one bench but 5 ms higher; shm=16g plateaus in 2–5 benches but 2 ms lower than native. Useful operational knob; negligible relative effect.
  5. --ipc=host vs --ipc=private: deferred. Since there is no steady-state gap to explain, the incremental value of discriminating H4 is low. The variant script is in place for anyone who wants to finish the measurement.
  6. The cycle benchmark protocol is systematically unreliable for measuring cross-regime steady-state differences at the ~10% level. Its single bench at DP=4 bench-position 1 is ~20–30% above plateau and the offset-to-plateau depends on pre-run host state. A plateau-seeking protocol gives trustworthy numbers; the Exp C variant harness has the primitives.

Next steps: (a) Integrate a bench_to_steady helper into exp_a3_vllm_ep/scripts/common.sh that runs benches until rolling TPOT stabilises within ε, then returns the stabilised measurement. Replaces single-shot cycle-bench for cross-regime comparison. (b) Update Exp A3 §5.5 / §5.7 with the corrected cross-regime steady-state numbers and a note that the previously reported gap was a methodology artefact; keep the cold-TTFT / scale-up-duration observations as they are. (c) If a mechanism-level explanation of the long MGC warm-up is wanted, a single-request py-spy / perf profile of bench 1 vs bench 5 in MGC would likely isolate it (expected candidates: Triton JIT, page-cache faults on weight files, Ray plasma init). (d) The per-GPU-container regime (Exp A3 §5.6) is not addressed by this finding; its −39.6% gap is NCCL transport selection, not warm-up.
