1. Objective

Determine whether a minimal LD_PRELOAD shim can recover full NVLink peer-to-peer bandwidth between per-GPU Docker containers, replacing the ad-hoc environment variable workarounds discovered in Experiment 2 (NCCL_CUMEM_ENABLE, NCCL_HOSTID, shared /dev/shm, --network=host). The experiment maps every system call NCCL uses for container detection, builds the thinnest possible interception layer, and validates it across NCCL versions.

Result: A 261-line C shim (libxtrans_shim.so) intercepts three NCCL container gates and recovers 99.2–100.2% of baseline NVLink bandwidth across all collective operations, with bridge networking, no shared /dev/shm, and no NCCL-specific environment variables. The shim is compatible with NCCL 2.19–2.21 (cuMem VMM transport).

2. Background

NCCL checks three conditions (“gates”) during communicator initialization to determine if two processes share a physical node. If any gate fails, NCCL falls back from P2P/NVLink to network sockets, reducing bandwidth by 20–50×. In per-GPU containers with separate namespaces, all three gates fail:

  1. hostHash gate: NCCL computes hash(gethostname() + /proc/sys/kernel/random/boot_id). Separate UTS namespaces produce different hostnames.
  2. shmDev gate: NCCL calls stat("/dev/shm") and compares st_dev values. Separate mount namespaces produce different device IDs.
  3. IPC socket gate: NCCL’s cuMem proxy uses abstract Unix domain sockets (@tmp/nccl-socket-<rank>-<hash>) for SCM_RIGHTS file descriptor passing. Abstract sockets are scoped to the network namespace and invisible across containers.

Experiment 2 recovered bandwidth by sharing the network namespace (--network=host), mounting a common tmpfs (/dev/shm), and setting NCCL_CUMEM_ENABLE=1. This experiment replaces all three workarounds with a single LD_PRELOAD library.

3.3 Methodology

Each benchmark runs 2 containers (one GPU each) coordinated by Docker Compose. The PyTorch entry point calls dist.init_process_group(backend='nccl', init_method='env://'). Each collective operation is measured at 6 message sizes (1 KB – 100 MB) with 100 iterations after warmup. Timing uses CUDA events with 5% trimmed mean.

Key findings from A1:

  • NCCL uses unconnected SOCK_DGRAM abstract UDS with pattern @tmp/nccl-socket-<rank>-<hash>.
  • cuMem file descriptors are passed via sendmsg with SCM_RIGHTS (96 FD passes for 24 channels × 2 directions × 2 operations).
  • In the isolated config, NCCL creates only 1 NCCL socket and 0 FD passes — it abandons P2P immediately when the peer socket is unreachable.
  • A separate @cuda-uvmfd-<ipc_ns>-<pid> socket is created by the CUDA driver for internal UVM management. This is per-process and does not require cross-container visibility.

NCCL 2.18 uses the legacy cudaIpcMemHandle path (P2P/IPC) which requires shared IPC namespace (--ipc=host). The shim successfully passes the hostHash and shmDev gates, but the underlying IPC mechanism cannot share GPU memory handles across containers without a shared IPC namespace. Starting with NCCL 2.19, the cuMem VMM path (P2P/CUMEM) is used by default on A100, which passes file descriptors over UDS — compatible with the shim’s socket redirection.

Limitation: NCCL 2.25, 2.27, and 2.28 were not yet available in public PyTorch wheels or NGC containers at the time of testing (April 2026). These should be tested when released. The interception surface (POSIX syscalls) is stable, so breakage is unlikely.

6. Analysis

6.1 Shim Mechanism

The shim (libxtrans_shim.so, 261 lines of C99) intercepts five libc functions via dlsym(RTLD_NEXT, ...):

FunctionGateActionWhen Called
gethostname() hostHash Returns configured hostname (XTRANS_HOSTNAME) Once during getHostHash() init
stat() / __xstat() shmDev Replaces st_dev for /dev/shm with configured value Once during fillInfo()
bind() IPC socket Redirects abstract UDS @tmp/nccl-socket-* to filesystem path in XTRANS_UDS_DIR Per proxy socket (∼48 calls)
sendmsg() IPC socket Redirects abstract UDS destination in msg_name to filesystem path Per IPC message (∼200 calls)

All intercepted functions are called only during NCCL communicator initialization. Zero data-path overhead: once P2P channels are established, all GPU–GPU communication flows over NVLink hardware without any shim involvement.

6.2 cuMem VMM as the Enabling Mechanism

The shim approach works because NCCL 2.19+ uses the cuMem Virtual Memory Management API for cross-process GPU memory sharing. cuMem exports GPU memory as POSIX file descriptors, which can be passed between processes via sendmsg with SCM_RIGHTS on any Unix domain socket — abstract or filesystem-based. The shim redirects NCCL’s abstract sockets to filesystem sockets in a shared volume, making the existing FD-passing mechanism work across network namespaces.

NCCL 2.18 uses the older cudaIpcMemHandle mechanism, which produces opaque blobs requiring a shared IPC namespace. No amount of syscall interception can bridge this — it is a fundamental limitation of the legacy IPC API.

6.3 Task A4: Security Analysis

Threat Model

The shim operates in a cooperative, single-tenant environment where all containers on a node belong to the same workload (e.g., distributed training). The security question is: does the shim weaken container isolation compared to the alternatives?

Comparison with --ipc=host

Property--ipc=hostExp2 WorkaroundXTrans Shim
IPC namespace Shared (full access to host SysV IPC, POSIX semaphores) Separate Separate
Network namespace Host or shared Host (--network=host) Separate (bridge)
/dev/shm Host shared memory Shared tmpfs volume Per-container (isolated)

NCCL env vars required Varies NCCL_CUMEM_ENABLE=1 None

Attack Surface Analysis

The shim’s security posture is strictly better than both --ipc=host and the exp2 workaround:

  1. No shared IPC namespace: Unlike --ipc=host, containers cannot access each other’s SysV shared memory, semaphores, or message queues. This prevents a compromised container from reading arbitrary host process memory.
  2. No shared network namespace: Unlike exp2’s --network=host, containers have separate network stacks. A compromised container cannot sniff traffic on the host network or bind to host ports.
  3. Scoped GPU memory access: cuMem FDs are exchanged only through the /shared_uds volume. A container can only access GPU memory that another container in the same workload explicitly exports. There is no mechanism to enumerate or access GPU allocations from other tenants.
  4. No escalation path: The shim uses LD_PRELOAD which operates entirely in user space. It does not require CAP_SYS_PTRACE, --privileged, or any elevated capabilities.

Residual Risks

  • Shared UDS volume: Any container with the volume mounted can create sockets in /shared_uds. A malicious container could impersonate an NCCL proxy socket and intercept FD-passing messages. Mitigation: In production, the volume should be restricted to containers in the same pod/job via orchestrator policy (K8s pod-level volumes, Compose service groups).
  • GPU topology discovery: NVIDIA_VISIBLE_DEVICES=all is required for NVML topology discovery. This exposes GPU metadata (names, UUIDs) but not compute access (restricted by CUDA_VISIBLE_DEVICES). A compromised container could enumerate GPUs on the node.
  • Hostname spoofing: The shim returns a fixed hostname, making containers appear to be on the same node. This is by design. In a multi-tenant environment, the orchestrator must ensure only same-workload containers share the shim configuration.

Minimal Privilege Summary

The shim requires only:

  • LD_PRELOAD environment variable (standard, no capability)
  • A shared tmpfs volume for UDS sockets (~100 MB)
  • NVIDIA_VISIBLE_DEVICES=all for NVML topology discovery
  • GPU device access via NVIDIA Container Toolkit (standard)

7. Conclusions

  1. Three gates identified: NCCL’s container detection relies on gethostname() (hostHash), stat("/dev/shm") (shmDev), and abstract Unix domain sockets (IPC proxy) — all interceptable via LD_PRELOAD.
  2. Full bandwidth recovery: The shim recovers 99.2–100.2% of baseline NVLink bandwidth across all collective operations (all_reduce, all_gather, reduce_scatter, broadcast, P2P) with bridge networking and no NCCL-specific environment variables.
  3. Zero data-path overhead: All interception occurs during communicator initialization; the GPU–GPU data path is unmodified NVLink hardware.
  4. cuMem VMM boundary at NCCL 2.19: The shim works with NCCL 2.19+ which uses the cuMem VMM transport (P2P/CUMEM). NCCL 2.18 uses legacy cudaIpcMemHandle which requires shared IPC namespace.
  5. Strictly better security than alternatives: The shim preserves separate IPC, network, and mount namespaces, whereas --ipc=host and --network=host each break a namespace boundary.

Next steps: (1) Validate the DMA-BUF file descriptor path as a vendor-agnostic IPC primitive on NVIDIA (Exp A′). (2) Apply the same gate taxonomy to AMD RCCL and test the three AMD IPC paths (Exp B). (3) Test against NCCL 2.25+ when available.
