1. Architecture Map of an Elastic Scale Event

An elastic scale event is a distributed protocol with six cooperating subsystems. Understanding the names, roles, and boundaries up front makes the later sections easier to follow.

Subsystem Canonical file Role during a scale event
StatelessGroupCoordinator distributed/stateless_coordinator.py Replaces torch.distributed._world. Holds the three transport channels (NCCL / Gloo / TCP-store) for a single group. Survives scale events as an instance per group; the old instance is destroyed and a new instance takes its place.
ElasticEPScalingState distributed/elastic_ep/elastic_state.py Per-engine state machine driving each rank through the scale event. Four flavours (see §3) depending on whether the rank is existing/new and the event is up/down.
ElasticEPScalingExecutor distributed/elastic_ep/elastic_execute.py Concrete operations invoked by the state machine: creating standby groups, transferring weights, swapping MoE configs, expanding EPLB state.
Standby group registry distributed/elastic_ep/standby_state.py Process-global slots (_STANDBY_WORLD, _STANDBY_DP, _STANDBY_EP, _STANDBY_EPLB) that hold the not-yet-active group coordinators until the atomic switch.
Weight transfer engine distributed/weight_transfer/ Pluggable backend (NCCL broadcast or CUDA IPC) that streams non-expert model parameters from old ranks to new ranks during scale-up.
EPLB state / rebalance distributed/eplb/ Tracks logical-vs-physical expert mapping, accumulates expert load counters, and re-solves placement when topology changes (see §6).

Two client-facing entry points drive all of this from outside the engine:

  • POST /scale_elastic_ep {"new_data_parallel_size": N} → HTTP handler in entrypoints/serve/elastic_ep/api_router.py.
  • AsyncLLM._scale_up_elastic_ep() / _scale_down_elastic_ep() in v1/engine/core_client.py (lines 1542–1695). These send ReconfigureDistributedRequest messages to every engine, spawn / retire Ray actors as needed, then await a barrier on the stateless coordinator.

2. Stateless Group Coordinator — the TCP-Store-Backed Group

The foundation that makes every other piece possible. Classical PyTorch process groups are registered in a global torch.distributed._world._world and identified by an opaque ProcessGroup pointer; destroying and recreating one invalidates ranks that held references to it. vLLM’s StatelessGroupCoordinator replaces this with a self-contained object whose lifetime can be managed per-scale-event.

2.1 Structure of a single coordinator

A StatelessGroupCoordinator instance (stateless_coordinator.py:61–194) owns three independent process-group abstractions over the same set of ranks:

ChannelBackendUsed for
device_group NCCL GPU collectives — weight-transfer broadcast, all-reduce for sync, any MoE collective that runs on this group.
cpu_group Gloo CPU-side host collectives (metadata, barriers that must not block on CUDA streams).
tcp_store_group PyTorch c10d::TCPStore Keyed object exchange, barriers, broadcast of small Python objects. Wraps StatelessProcessGroup (distributed/utils.py:167–194).

All three channels use three independent TCP ports (_PORTS_FMT = "!3I", stateless_coordinator.py:28) that rank 0 binds locally and publishes into the shared coord_store (a parent TCPStore that every rank already knows how to reach via data_parallel_master_ip:coord_store_port). Non-zero ranks block on _fetch_group_ports() (stateless_coordinator.py:112) until rank 0 publishes. This is why the coordinator scheme extends across machines: every rank only needs one globally-known TCP endpoint (the coord store) to bootstrap every group it will ever join.

2.2 Public operations

The coordinator exposes roughly the same API as a PyTorch ProcessGroup (stateless_coordinator.py:195–366): size(), broadcast(), broadcast_object_list(), broadcast_tensor_dict(), send_object()/recv_object(), send_tensor_dict()/recv_tensor_dict(), barrier(), gather(). Tensor-dict transfer is hybrid: metadata travels over the TCP-store (cheap, no CUDA dependency) while tensor payloads route through device_communicator / cpu_group.

2.3 Why this matters

  • Creation is rank-local. Any subset of the world can form a new group as long as each member can talk to the coord store — no “the launcher must restart” barrier. This is what makes both scale-up (new ranks join) and scale-down (departing ranks leave) possible in-place.
  • Destruction is local too. The old group is torn down on an all-ranks barrier at elastic_state.py:510 via stateless_destroy_torch_distributed_process_group(old_dp_group), not by removing a global registry entry. Other groups (EPLB, EP, WORLD) are unaffected.
  • Coexistence is free. Two coordinators can live on the same ranks simultaneously, because their membership is held by the coordinator object itself, not by a singleton. This is the basis of standby groups (§4).

3. The Scaling State Machine (per engine)

elastic_state.py defines four separate IntEnum state machines, one per role a rank can play during a scale event. A rank picks exactly one on entry and walks it to completion. The state enum names below are verbatim from the source.

3.1 The four role state-machines

3.1.1 Scale-up, existing engine (ScaleUpExistingEngineState, elastic_state.py:33–42)

0  WAIT_NEW_CORE_ENGINES_INIT        wait for new Ray actors to come online
1  CREATE_STANDBY_GROUPS             allocate new NCCL groups next to active
2  TRANSFER_EXPERT_MAPPING           broadcast physical-to-logical map to newcomers
3  WAIT_NEW_CORE_ENGINES_WEIGHTS_INIT wait for new ranks' weight buffers ready
4  TRANSFER_WEIGHTS                  P2P send non-expert weights (batch_isend_irecv)
5  SYNC_KV_CACHE_MEMORY_SIZE         max-reduce of per-rank KV reservation
6  SWITCH_AND_PREPARE                atomic group swap; rebuild MoE configs
7  EPLB_RESHUFFLE                    rearrange experts on the new topology
8  COMPLETE                          terminal

3.1.2 Scale-up, new engine (ScaleUpNewEngineState, elastic_state.py:45–49)

0  PRE_KV_INIT        receive expert map + weights; init KV cache slot
1  PREPARE            participate in all_reduce to sync engine state
2  EPLB_RESHUFFLE     reshuffle with the rest of the cluster
3  COMPLETE           terminal

3.1.3 Scale-down, remaining engine (ScaleDownRemainingEngineState, elastic_state.py:52–56)

0  PREPARE            increment barrier; confirm all survivors present
1  EPLB_RESHUFFLE     reshuffle experts including those inherited from leavers
2  SWITCH_AND_PREPARE create standby (smaller) group and atomically swap
3  COMPLETE           terminal

3.1.4 Scale-down, removing engine (ScaleDownRemovingEngineState, elastic_state.py:59–62)

0  PREPARE            join the staged barrier so survivors know we are done
1  EPLB_RESHUFFLE     emit expert weights for the remaining ranks to inherit
2  COMPLETE           exit cleanly (Ray actor will be reclaimed)

3.2 The two-stage barrier

Transitions between these states are gated by a custom barrier (_staged_barrier, elastic_state.py:182–225) rather than torch.distributed.barrier():

  1. Stage 1 (TCP-store). Each rank writes a per-state arrival key to dp_store, then polls for peers’ keys with a 5 s timeout (elastic_state.py:152–225). A rank that times out yields the state and re-enters next iteration — the engine loop is not allowed to deadlock on a reconfiguration.
  2. Stage 2 (NCCL). Once all ranks have registered on the store, a standard torch.distributed.barrier(group=dp_group) forces GPU-side synchronisation (elastic_state.py:215).

Stage 1 exists because during a scale event some engines may still be draining a forward pass while others have already reached the barrier; the two-stage design lets everyone queue on the cheap TCP store and only enter the GPU barrier when all participants are ready.

3.3 Atomic switch, in code

The handoff the analysis-report calls “atomic switch” consists of these statements executed in order on every rank (elastic_state.py:505–535):

# 1. Tear down the old group (collective)
stateless_destroy_torch_distributed_process_group(old_dp_group)

# 2. Adopt the new group as the active one
self.engine_core.dp_group        = new_dp_group
self.engine_core.dp_rank         = new_dp_rank
self.engine_core.dp_store        = new_dp_store

# 3. Agree with peers on shared engine scalars
tensor = torch.tensor(
    [engines_running, current_wave, step_counter],
    dtype=torch.int32, device="cpu")
torch.distributed.all_reduce(
    tensor, op=torch.distributed.ReduceOp.MAX, group=new_dp_group)

# 4. Rank 0 signals the client side
_eep_send_engine_core_notification(
    EEPNotificationType.RECONFIGURE_FINISHED)

Step 1 is the collective that must succeed on every old-group member — a departing container that died early would deadlock surviving ranks here. This is exactly the “container as actor exit” question raised in v5 §4.2.

4. Standby Groups — How Two Communicators Share a GPU

A standby group is a StatelessGroupCoordinator created while the active group is still serving. It lives in four module-level slots (standby_state.py:14–18):

_STANDBY_WORLD: StatelessGroupCoordinator | None = None
_STANDBY_DP:    StatelessGroupCoordinator | None = None
_STANDBY_EP:    StatelessGroupCoordinator | None = None
_STANDBY_EPLB:  StatelessGroupCoordinator | None = None

create_standby_groups() (standby_state.py:37–100) is called from ElasticEPScalingExecutor.create_standby_groups() (elastic_execute.py:175–197) during the CREATE_STANDBY_GROUPS state. It:

  1. Re-uses the cached parent coord store (get_cached_tcp_store_client(master_ip, coord_store_port), standby_state.py:59) so the standby groups share a name-space with the active ones.
  2. Computes the new membership tensors for WORLD, DP, EP, EPLB using an all-ranks reshape (standby_state.py:75–99): 
    all_ranks = torch.arange(new_world_size_across_dp).reshape(
    -1, new_dp_size, pp_size, tp_size)
standby_dp_ranks = (all_ranks
                    .transpose(1, 3)
                    .reshape(-1, new_dp_size)
                    .unbind(0))
  3. Instantiates a new StatelessGroupCoordinator per group. Each call negotiates three fresh TCP ports under keys like "world_0", "dp_0", "ep_0" on the shared coord store.

Coexistence constraints observed in the code:

  • Both groups target the same torch.device(f"cuda:{local_rank}") (stateless_coordinator.py:155). Two ncclComm_t handles exist per GPU during the window.
  • No communicator pooling — each coordinator’s device_communicator manages its own NCCL send/recv buffers. Peak device memory during the transition is therefore higher than either endpoint configuration. The code does not explicitly account for this; in practice it works because NCCL communicator state is tens of MB per GPU.
  • Kernels are serialised per CUDA stream, so old-group collectives cannot truly run concurrently with standby-group collectives on the same GPU. The overlap claimed in analysis-report §6.1 materialises as async queueing, not parallelism.

pop_standby_groups() (standby_state.py:102–123) is the “commit” operation — it returns the four standby coordinators and clears the module-level slots in one shot. The caller (switch_and_prepare(), elastic_execute.py:290) feeds the result to _replace_active_groups() (parallel_state.py:1205–1212), which rebinds the global _WORLD, _DP, _EP, _EPLB symbols in a single critical section.

5. Weight Transfer Subsystem

Populating new ranks with model weights is the largest data movement in a scale-up event. vLLM’s weight-transfer subsystem is a standalone, pluggable library (distributed/weight_transfer/) that the elastic path uses but that also has independent consumers (RLHF, training).

5.1 Backend registry

The factory registers two backends (weight_transfer/factory.py:112–122):

NameClassTransportElastic EP uses it?
nccl NCCLWeightTransferEngine (nccl_engine.py:97) ncclBroadcast over a fresh PyNcclCommunicator Yes (in-cluster scale-up)
ipc IPCWeightTransferEngine (ipc_engine.py:111) CUDA IPC handles exchanged out-of-band (Ray RPC or HTTP POST) No — used by external training frameworks to push updates in

The elastic-EP scale-up path calls batch_transfer_weights() (elastic_execute.py:48–86) directly, which uses the standby DP group’s batch_isend_irecv() rather than the WeightTransferEngine abstraction — the engine factory is used when weights come from outside the cluster. The same NCCL channel and the same container-gate considerations apply either way.

5.2 PackedTensor — why small sends don’t dominate

A 30 B-parameter model has thousands of named tensors. Naively issuing one ncclBroadcast per tensor would pay the per-collective latency (tens of microseconds) thousands of times. PackedTensor (weight_transfer/packed_tensor.py) solves this by linearising many tensors into a uint8 view and double-buffering 1 GB chunks:

DEFAULT_PACKED_BUFFER_SIZE_BYTES = 1 << 30      # 1 GiB per buffer
DEFAULT_PACKED_NUM_BUFFERS       = 2            # double-buffered

The producer packs tensors into the current buffer as t.view(torch.uint8).view(-1), broadcasts it once via group.broadcast() (packed_tensor.py:76), then rotates to the next buffer while receivers unpack. The receiver reads the ordered shapes[] / dtype_names[] metadata to split the uint8 buffer back into the original tensors (packed_tensor.py:112–140).

This is why analysis-report §6.2’s claim that non-expert weight transfer is “seconds in practice” holds only when NCCL selects a fast transport: bytes-in-flight is already maximised by packing, so the remaining bottleneck is transport bandwidth. A TCP socket at ~10 GB/s takes ~3 s per 28 GB rank; NVLink at ~600 GB/s would take ~50 ms. The 60× gap is exactly what v5 §4.5 wants to measure.

5.3 Which parameters move and which don’t

The elastic-EP batch_transfer_weights explicitly filters out expert weights from the state-dict iteration (elastic_execute.py:59–71):

# Build set of parameter IDs belonging to experts
expert_weights_set = { id(w)
                       for layer in moe_layers
                       for w in layer.expert_weights }

for name, param in model.named_parameters():
    if id(param) in expert_weights_set:
        continue          # skip experts — they come from EPLB reshuffle
    if "expert_map" in name:
        continue          # skip routing metadata
    ...
    send/recv(param)

The rationale is sound: a newcomer rank will not end up with the same expert slice as any existing rank, so copying one peer’s experts would be wrong. Instead, expert weights either come from (a) shared-filesystem loading (cheap — each new rank reads its own slice), or (b) an EPLB reshuffle that moves experts from overloaded old ranks to new ones. The existence of enable_ep_weight_filter (config/parallel.py) skips non-local expert weights at load time, making (a) cheap in wall-clock.

5.4 Sender/receiver pairing

For a scale-up from M to N>M ranks, each old rank pairs with ⌈(N−M)/M⌉ or ⌊(N−M)/M⌋ new ranks (elastic_execute.py:216–232):

num_new          = new_dp_size - old_dp_size
num_dst_per_src  = num_new // old_dp_size
remainder        = num_new  % old_dp_size

if dp_rank < remainder:
    recv_begin  = dp_rank * (num_dst_per_src + 1)
    recv_end    = recv_begin + (num_dst_per_src + 1)
else:
    recv_begin  = (remainder * (num_dst_per_src + 1)
                  + (dp_rank - remainder) * num_dst_per_src)
    recv_end    = recv_begin + num_dst_per_src

For the 2→4 case that dominates analysis-report phase-1 and phase-3 measurements: 2 new ranks, 2 old ranks, each old sends to one new in parallel, no stragglers. The receiver mirror-calculates its sender (elastic_execute.py:501–535).

6. EPLB: Logical vs Physical Experts, and the Cold-DP=N Bug

EPLB is the single most complex subsystem involved in elasticity. It owns three tensors per MoE layer that encode the logical/physical expert mapping, plus load-accumulation counters, plus the rebalancing policy. The elastic-EP code piggybacks on it heavily: every scale event triggers a rearrange().

6.1 The three mapping tensors

TensorShapeMeaningSource
physical_to_logical_map (num_moe_layers, num_physical_experts) Value at [L, p] is the logical expert ID that physical slot p hosts. eplb_state.py:94–109
logical_to_physical_map (num_moe_layers, num_logical_experts, num_redundant_experts+1) Value at [L, l, k] is the physical slot hosting the k-th replica of logical expert l, or −1 if there is no such replica. eplb_state.py:110–133
logical_replica_count (num_moe_layers, num_logical_experts) Number of non-−1 entries in logical_to_physical_map[L, l, :]. eplb_state.py:134–148

The maximum replication factor for any logical expert is hard-coded at 1023 (MAX_EXPERT_REDUNDANCY = 1023, eplb_state.py:397), which sets the last dimension of logical_to_physical_map when num_redundant_experts is smaller.

6.2 Load accumulation

Per forward pass, each MoE layer writes routing hits into expert_load_pass (eplb_state.py:150–156, int32 GPU tensor of shape (num_moe_layers, num_physical_experts)). A sliding window of these is retained in expert_load_window (eplb_state.py:157–170, shape (window_size, num_moe_layers, num_physical_experts)). Recording is gated by should_record_tensor (eplb_state.py:605–616) — counters are only updated when the next rearrangement is within window_size steps, to avoid wasting GPU cycles on load counts that will be discarded.

6.3 The rebalance policy

The default policy is DefaultEplbPolicy (eplb/policy/default.py). Its rebalance_experts() is a three-step hierarchical packing:

  1. Pack groups to nodes. Logical experts are grouped (by model-defined group structure), and groups are distributed across num_nodes via balanced_packing() to equalise per-node token load.
  2. Construct redundant experts within each node. Greedy packing replicates hot experts to fill up to num_physical_experts / num_nodes slots per node.
  3. Pack physical experts to GPUs. Normalise load by replica count (tokens_per_phy = tokens_per_mlog / mlogcnt) then balanced-pack across GPUs within a node, respecting node boundaries.

The output is a fresh physical_to_logical_map which the surrounding plumbing translates into the per-rank expert-weight movements described in §6.5.

6.4 The num_replicas // ep_group.size() * num_gpus formula

This is the formula at the centre of the cold-DP=N scale-down bug (analysis-report §6.3). It lives in EplbState.rearrange() (eplb_state.py:771–786):

if rank_mapping is not None and len(rank_mapping) == ep_group.size():
    # NOTE(yongji): scale down...
    coordinator = get_ep_group()
    tcp_store_group = coordinator.tcp_store_group
    num_gpus  = sum(new_rank != -1 for new_rank in rank_mapping.values())
    num_nodes = _node_count_with_rank_mapping(tcp_store_group, rank_mapping)
    num_replicas = (num_replicas // ep_group.size() * num_gpus)

Unpacking: if the current config has num_replicas = 128 physical slots spread over ep_group.size() = 4 ranks (32 per rank), a scale-down to num_gpus = 2 ranks sets num_replicas = 128/4 * 2 = 64. For Qwen3-30B-A3B with num_logical_experts = 128 and num_redundant_experts = 0, the initial num_replicas equals 128, and a 4→2 scale-down computes num_replicas = 64 < num_logical = 128. The later invariant check fires, and the assertion aborts the process.

v5 §4.4 connection. The formula is per-rank-proportional, not preserve-total-replicas. This means scale-down always reduces num_replicas proportionally, and with zero redundancy that reduction is fatal. The documented workaround (start at DP≤N/2, scale up first) works because the scale-up path runs the rebalance policy on a larger num_replicas, building replicas that survive the next scale-down.

6.5 Expert weight movement during rebalance

The data movement is done by move_to_buffer() (eplb/rebalance_execute.py:144–319) in four phases on every rank:

  1. Local moves (rebalance_execute.py:230–242): copy within-rank expert weights to an intermediate buffer. Uses torch.cuda.stream() for async overlap.
  2. Post sends (rebalance_execute.py:244–275): for each expert we own that another rank now needs, communicator.add_send(w[src], dst).
  3. Post recvs (rebalance_execute.py:277–305): for each expert we’re inheriting, communicator.add_recv(buf[dst], src).
  4. Execute (rebalance_execute.py:308): communicator.execute() flushes the batch atomically.

The communicator can be any of three backends (eplb/eplb_communicator.py): TorchDistNcclEplbCommunicator (NCCL batched isend/irecv), TorchDistGlooStagedEplbCommunicator (H2D→gloo P2P→D2H, host-staged), or PyNcclEplbCommunicator (raw ncclSend/ncclRecv with manual group_start/group_end). Elastic EP forces the PyNccl backend (elected at eplb_communicator.py:254–267), because it needs to target the stateless group rather than the torch.distributed world.

6.6 Rank mapping for departing ranks

On scale-down, survivors need to know which old ranks are disappearing so they can inherit those ranks’ experts. The encoding is a dict[int, int] (old rank → new rank) where departing ranks map to −1 (rebalance_execute.py:645–677). The rearrange code walks this dict, expands departing ranks’ expert slots into −1 placeholders in the working tensor, and lets the policy re-solve placement over the reduced num_gpus.

7. Runtime All-to-All Dispatch — the Hot Path

Orthogonal to scale events, every MoE forward does an All-to-All. This is the throughput-determining collective. vLLM registers eight backends in device_communicators/cuda_communicator.py:118–172 and implements them in device_communicators/all2all.py.

7.1 AgRsAll2AllManager walk-through

The backend used in Exp A3 is allgather_reducescatter (all2all.py:150–249):

def dispatch(hidden_states, topk_weights, topk_ids):
    # Concatenate inputs along dim 0
    # Query per-rank token counts from DP metadata
    chunk_sizes = dp_metadata.get_chunk_sizes_across_dp_rank()
    # Variable-size all-gather — each rank holds the global batch
    gathered = self.dist_group.all_gatherv(
        cat_input, chunk_sizes)
    return split_gathered(gathered)           # → (h, w, ids)

def combine(hidden_states):
    # Variable-size reduce-scatter: expert outputs summed and
    # scattered back to token owners
    return self.dist_group.reduce_scatterv(
        hidden_states, chunk_sizes)

Under the hood, all_gatherv / reduce_scatterv on a StatelessGroupCoordinator are grouped NCCL collectives between ncclGroupStart()/ncclGroupEnd() (pynccl.py:295–309). Every forward pass of every MoE layer touches this path.

7.2 Why the EP group replacement is invisible to all2all managers

A subtle but important design choice: the All2AllManagerBase stores a reference to the current EP communicator at construction time (base_device_communicator.py:43–44), not a symbolic reference to “the EP group”. When _replace_active_groups() swaps _EP, the all2all manager still holds a reference to the old group object.

The code gets away with this because the manager’s operations are all method calls through the group reference (self.dist_group.all_gatherv(…)), and the group object itself is either a torch ProcessGroup or a StatelessGroupCoordinator — no shared state between the two. Once the active EP group is replaced, a new All2AllManager is constructed against the new group in switch_and_prepare() (elastic_execute.py:409–414). The old manager is simply dereferenced.

Only NixlEP is built for elastic scaling. It stores a class-level singleton buffer and calls buffer.connect_ranks() / buffer.disconnect_ranks() (all2all.py:486–500) to incrementally adjust the peer set rather than rebuild. For the backends we use, the group-swap-and-rebuild pattern is the norm.

8. NCCL Communicator Lifecycle (PyNccl)

8.1 Construction

PyNcclCommunicator (pynccl.py:58–147) is a thin Python wrapper over libnccl.so. Construction is:

  1. Load libnccl.so via NCCLLibrary (pynccl_wrapper.py:324–344). Cached per library path at class level.
  2. Rank 0 calls ncclGetUniqueId() (pynccl_wrapper.py:398–401).
  3. Rank 0 broadcasts the unique ID. For stateless groups (pynccl.py:127), this is group.broadcast_obj(unique_id, src=0) routed through the coord-store TCP group — not torch.distributed. This is why container separation does not block NCCL init.
  4. All ranks call ncclCommInitRank(world_size, unique_id, rank) (pynccl_wrapper.py:412–421).
  5. A warmup all_reduce on a 1-element tensor sanity-checks the comm (pynccl.py:141–146).

8.2 Are multiple comms per GPU supported?

Yes. PyNcclCommunicator is not a singleton, and nothing in its construction prevents multiple instances on the same cuda:N device. This is what makes standby groups possible. Each instance owns its own ncclComm_t; there is no communicator pool or shared buffer across instances.

8.3 ncclCommSplit is not used

NCCL 2.13+ supports ncclCommSplit(), which can cheaply derive a sub-communicator from a parent. A grep across the vLLM tree shows zero usages. Every new group is built via the full ncclGetUniqueId + broadcast + ncclCommInitRank sequence, which is the slowest option but the most robust to arbitrary rank-set changes.

8.4 Destruction is collective and explicit

PyNcclCommunicator.destroy() (pynccl.py:148–153) calls ncclCommDestroy(comm). This is collective — all ranks must participate or deadlock. The code-base explicitly avoids relying on Python GC for this (pynccl_wrapper.py:287–290); destruction happens only from explicit code paths:

  • Scale-up: old DP group destroyed at elastic_state.py:510 after standby becomes active.
  • Scale-down: same as above; departing ranks join the destroy collective before their state-machines transition to COMPLETE.
  • Weight transfer engines: temporary comms are destroyed at the end of shutdown() (weight_transfer/nccl_engine.py:274–339).

This is the direct answer to v5 §4.2’s container-lifecycle question: a container that dies abruptly before the destroy collective fires will deadlock survivors. Graceful exit must cooperate with the state machine.

10. Container-Boundary Matrix — Where the Deep Dive Meets v5

Every subsystem above makes specific assumptions about shared state between participating ranks. Annotated against v5 §4’s per-GPU container scenario:

Subsystem Shared-state assumptions Per-GPU container outcome
Stateless coordinator (stateless_coordinator.py) Only TCP reachability to coord_store_port. No shmem, no hostname, no PID. Passes. Already used cross-container in phase 3.
PyNccl unique-ID broadcast Uses the coord store for stateless groups, so no torch.distributed rendezvous required. Passes.
NCCL data transport (ncclBroadcast, ncclAllGather, ncclReduceScatter, ncclSend/ncclRecv) NCCL's three same-node gates must pass: hostHash, shmDev, abstract-UDS. Fails without shim. All three gates fail → NCCL picks NET/Socket (measured: −23% throughput, analysis-report §5). v5 §4.1 workaround plan: inject common/shim/.
Standby group creation (§4) Same as stateless coordinator + NCCL data transport. Passes architecturally, slow without shim.
Weight transfer (§5) Same as NCCL data transport, plus expert-weight read from shared model dir for newcomers. Slow without shim (NCCL over TCP). Shared model dir bind-mount is already standard per phase-3 setup. v5 §4.5 investigates.
EPLB rebalance P2P (PyNcclEplbCommunicator) NCCL send/recv between ranks. Same as NCCL data transport.
Runtime all-to-all (§7) NCCL collectives on the EP group. Same as NCCL data transport — this is the throughput-dominant hot path.
ncclCommDestroy collective at atomic switch All ranks must participate. A rank that crashed or its container stopped without cooperating will hang survivors. Design gap for v5 §4.2. “docker stop ep-rank-3” is not safe as-is; survivors wait indefinitely. Need a cooperative exit hook before container termination.
EPLB invariant (num_physical ≥ num_logical) Per-rank proportional scaling formula (§6.4) is independent of container layout. Orthogonal to containers. Bug still fires if hit from a cold DP=N start with num_redundant_experts=0. v5 §4.4.
Ray actor lifecycle engine_manager.scale_up_elastic_ep assumes Ray can place new actors; remove_run_refs_for_scale_down releases actor refs but leaves workers running. Ray worker != container. Under the per-GPU model we need the worker’s container to stop, which Ray does not drive. v5 §4.2 & 4.3.

11. Open Questions Raised by the Deep Dive

Several design details deserve empirical confirmation before the next phase of v5 exploration:

  1. Does the shim restore ncclSend/ncclRecv bandwidth, not just ncclAllReduce? Exp A validated the shim on AllReduce and raw P2P bandwidth. vLLM’s weight transfer is ncclBroadcast over a fresh comm; runtime all-to-all is grouped ncclAllGather+ncclReduceScatter; EPLB rebalance is grouped ncclSend/ncclRecv. A micro-benchmark of each primitive with the shim loaded would directly derisk v5 §4.1.
  2. What is the real cost of the double communicator during the standby window? The code does not audit peak GPU memory during the transition. With a 30 B model already at 85% GPU util, an extra NCCL comm context is probably fine, but a measurement would confirm it.
  3. Is there any path to ncclCommSplit for scale-down? Scale-down from N to N-k ranks is exactly the case that ncclCommSplit was designed for. A patch to vLLM that detects “pure subset” transitions and uses ncclCommSplit could cut group-reform time from seconds to milliseconds — with no container interaction.
  4. Can the destroy collective be made tolerant of a dead peer? Exactly the v5 §4.2 question. NCCL 2.22 added ncclCommAbort and ncclCommFinalize, which allow non-blocking destruction. The vLLM path does not use them; adopting them would soften the “container must exit cooperatively” hard constraint.
  5. Could expert-weight transport bypass NCCL entirely? The shared model volume is mounted in every per-GPU container by construction (phase 3 deployment). New ranks could load their experts directly from disk — trading NCCL bandwidth for page-cache I/O. Exp A3’s 28 GB at ~3 GB/s NVMe = ~10 s; at ~10 GB/s cached = ~3 s. Either beats TCP-fallback NCCL. v5 §4.5’s fallback idea.
