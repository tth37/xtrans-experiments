1. Objective

Characterize vLLM’s Elastic Expert Parallelism (Elastic EP) mechanism — the first production-grade system for in-place live rescaling of GPU resources during MoE inference — as a baseline for studying what happens when this mechanism meets container boundaries. Specifically: confirm that live rescaling (2↔4 EP workers) works on our hardware, measure rescaling time and throughput impact, and identify where containerization creates friction.

Result. Live rescaling works in all three regimes on a 2→4→2 cycle. At DP=4 steady state (plateau TPOT, lower is better): native 93.26 ± 6.32 ms, multi-GPU container 86.92 ± 0.94 ms, per-GPU containers 87.64 ± 0.86 ms — MGC and per-GPU differ by 0.8%, statistically indistinguishable. The two container boundaries still have distinct structural consequences: multi-GPU container traps the orchestrator view (HostConfig.DeviceRequests.DeviceIDs immutable at [0,1,2,3] for the container’s lifetime, even when only 2 GPUs are resident); per-GPU containers free the trap (each owns one GPU) but force NCCL onto NET/Socket/0 because all three same-node gates fail across container boundaries. The NCCL transport change is visible at the log level and motivates the LD_PRELOAD shim in v5 §4.1, but the per-token EP all-to-all payloads at this DP / request shape are small enough that the 10–25× microbenchmark bandwidth gap does not propagate 1:1 to serving.

2. Background

2.1 From Exp A1: Stop-Restart Systems Don’t Need Per-GPU Containers

Exp A1 (Tenplex) revealed that “elastic GPU scheduling” in Tenplex is actually stop-transform-restart: the training process fully terminates, state is transformed on the host, and a new process starts with a different GPU count. Under the multi-GPU container model, this pattern works fine — you stop the container, recreate it with a different --gpus flag, and restart. The GPUs are freed between container lifecycles. Per-GPU containers add complexity without benefit.

2.2 In-Place Systems Are Fundamentally Different

vLLM’s Elastic EP is in-place: the serving process stays running while GPUs are added or removed. The system reinitializes NCCL communicators, transfers expert weights, and reshuffles expert placement — all without restarting. This creates a fundamentally different interaction with containers:

  • Multi-GPU container (prediction): Scale-down works internally, but the freed GPUs remain trapped in the container. Docker’s immutable --gpus flag prevents the cluster from reclaiming them. Scale-up is impossible beyond the --gpus set — Docker cannot hot-add GPUs to a running container. The in-place benefit is negated.
  • Per-GPU container (prediction): Each EP worker lives in its own container. Scale-down = stop containers, GPUs immediately free. Scale-up = start new containers, new workers join. The container lifecycle directly expresses the scaling semantics.

2.3 Why MoE + Expert Parallelism

Mixture-of-Experts models are uniquely suited for per-GPU container deployment. Each expert is a self-contained computation unit that maps naturally to one GPU and one container. Unlike tensor parallelism (which shards a single tensor across GPUs, requiring tight coupling), expert parallelism routes tokens to independent expert modules via All-to-All communication. The “Expert-as-a-Service” paradigm (arXiv ’25) already proposes disaggregating experts into independent services — precisely what per-GPU containers provide at the infrastructure level.

2.4 vLLM Elastic EP: Technical Overview

Elastic EP shipped in vLLM v0.17.0 (March 2026). When GPU resources change, vLLM brings up or tears down engine cores, reinitializes EP/DP communicators via stateless NCCL groups, transfers model weights GPU-to-GPU, and rebalances expert loads via EPLB algorithms. The serving process survives the scaling event without restart. For implementation-level mechanics see mechanism_deep_dive.html.

4.2 Scaling plan: the 2→4→2 cycle

All three regimes run the same cycle: DP=2 (bench) → POST /scale_elastic_ep {new_dp: 4} → DP=4 (bench) → POST /scale_elastic_ep {new_dp: 2} → DP=2 (bench).

Why not start cold at DP=4 and scale down. A cold DP=N start with the default num_redundant_experts=0 has num_physical = num_logical. vLLM’s rearrange() uses a per-rank-proportional scaling formula (num_phy_new = num_phy_old // old_ep × new_ep) which would halve num_physical on a 4→2 scale-down and violate the num_physical ≥ num_logical invariant, hitting assert num_redundant ≥ 0 in vllm/distributed/eplb/policy/default.py. Starting at DP=2 and scaling up first conjures redundancy (num_phy = 128 // 2 × 4 = 256) that the later scale-down consumes, so the cycle closes cleanly at R=0. This matches vLLM’s own CI test (tests/distributed/test_elastic_ep.py::test_elastic_ep_scaling). See mechanism_deep_dive.html §6 for the full derivation and state table.

4.3 Measurement methodology: plateau-seeking benches

Each bench-point in the cycle is a plateau-seeking run (common/harness/bench_to_steady.py, invoked by each regime’s cycle_steady subcommand): loop vllm bench serve at a fixed shape until the last 3 TPOTs are within 3% of each other (max−min range), then record 2 extra samples for statistical power. Defaults: warmup_min=2, window=3, eps_pct=3%, max_benches=15, extra_samples=2. A run that fails to converge within 15 benches is flagged and its plateau window is reported as the last 5 benches (with the caveat that warmup-curve drift may still be present). The same shape is applied identically to all three regimes, so cross-regime comparison is valid by construction.

Bench shapes per DP size:

  • DP=2: --num-prompts 16 --max-concurrency 8 --random-input-len 128 --random-output-len 128
  • DP=4: --num-prompts 32 --max-concurrency 16 --random-input-len 128 --random-output-len 128

The DP=4 shape is 2× the DP=2 shape in both request count and concurrency, keeping per-engine load constant across the cycle (concurrency / dp = 4 requests-in-flight per DP engine throughout).

Required vLLM flags for elastic EP (same in all three regimes):

--enable-expert-parallelUse EP instead of TP for MoE layers
--enable-elastic-epEnable stateless NCCL groups for dynamic scaling
--enable-eplbExpert load balancing (mandatory with elastic EP)
--data-parallel-backend rayOnly Ray supports elastic scaling

Additional flags, constraints and the full invocation are in Appendix A.

The DP=4 plateau is the load-bearing comparison: all three regimes converged tightly (σ ≤ 6.32 ms) on the same bench shape under the same harness. Multi-GPU container and per-GPU containers are within 0.8% of each other; both are nominally ~6% faster than native, but that gap sits inside the native plateau’s own ±6.32 ms σ — we treat it as noise rather than a signal without a mechanism explanation.

The DP=2 plateaus do not converge within the 15-bench ceiling for either container regime. Their σ (10–37 ms on a mean near 90 ms) reflects the 6–10% cycle-protocol variance floor documented in exp_c_container_overhead/analysis_report.html §5.8 and is not a regime signal.

Near-linear scale-up. Plateau tok/s at DP=4 (170.55) vs DP=2 initial (78.05) gives a 2.19× speedup on 2× the GPU count. The DP=4 bench shape is also 2× the DP=2 shape in concurrency, so this is a per-engine-load-matched comparison.

Reconciling with NCCL microbenchmarks. The NCCL microbenchmarks in common/benchmarks/ (and the archived Exp A gate-taxonomy work) measure a 10–25× raw-bandwidth hit when NCCL falls back to NET/Socket/0: 156 GB/s NVLink AllReduce vs ~11 GB/s TCP. A naive reading would expect the same penalty at the serving layer. The plateau data argues that isn’t what happens here: the residual steady-state gap between MGC (NVLink-enabled) and per-GPU containers (TCP-fallback) at DP=4 is <1%. The plausible mechanism is that vLLM’s EP all-to-all dispatches per-token expert activations — small per-message payloads — where NCCL bandwidth is not the binding constraint at this DP and request shape. This doesn’t contradict the microbenchmark; it says the microbenchmark penalty doesn’t propagate 1:1 to serving under these conditions. The v5 §4.1 shim track retains motivation (larger DP, larger batch, or larger token-context may shift the balance toward bandwidth-bound) but cannot claim a 23% recovery headroom on this workload.

Container overhead on scale-up (MGC +51% vs native) is consistent with Ray actor startup being slower inside a container — each DPMoEEngineCoreActor spawn imports Python and imports vLLM modules from a container filesystem rather than the host page cache. Scale-down is barely affected (MGC +15%) because it does not spawn actors, only tears them down. Per-GPU-containers scale-up/down timings track the MGC regime within the harness latency budget — the extra container boundaries do not add a detectable step.

GPU memory reclamation. Within ~10 s of scale-down 4→2, GPU 2 and GPU 3 drop from ~35 GB used to 4 MiB (idle) in all three regimes. The freed GPUs are physically reclaimable by the OS. Whether they are administratively reclaimable depends on the orchestrator view — which is a regime-specific question, treated in §5.4.

5.3 Service-interruption window during scale

During a scale event, the vLLM middleware rejects incoming requests with HTTP 503 until the new engine configuration is live. We probed the window with a completion request every 0.5 s during the native round-2 scale-down 4→2:

The 503 window (3.53 s) is shorter than the full scaling operation (9.64 s): the middleware flag clears as soon as the remaining engines complete reconfiguration, while outgoing-engine teardown and GPU memory release continue asynchronously. The middleware is the same (vllm/entrypoints/serve/elastic_ep/middleware.py) in all three regimes, so the qualitative window behaviour is shared; a scale-specific re-measurement across regimes is not expected to change the number meaningfully.

No request migration. The 8 probes that received 503 were rejected outright — not queued, retried, or migrated. A production deployment would need client-side retry with backoff during the 503 window.

5.4 Orchestrator visibility of GPU allocation

At every scale event in the multi-GPU-container regime, we snapshotted two independent views of the 4-GPU state: nvidia-smi from the host (what memory is actually resident on each GPU), and docker inspect HostConfig.DeviceRequests (what GPU allocation Docker still assigns to the container):

Moment (MGC regime) Host nvidia-smi (MiB) Docker DeviceIDs
Initial (DP=2) 37803, 37803, 4, 4 [0, 1, 2, 3]
Post-scale-up to DP=4 (+12 s) 36461, 36461, 35637, 35797 [0, 1, 2, 3]
Post-scale-down to DP=2 (+12 s) 37449, 37445, 4, 4 [0, 1, 2, 3]
Final (container still running, workload idle) 0, 0, 0, 0 [0, 1, 2, 3]

Multi-GPU container: the orchestrator trap. HostConfig.DeviceRequests.DeviceIDs is immutable at [0, 1, 2, 3] for the entire container lifetime — regardless of whether vLLM is actively using 2 GPUs or 4 GPUs inside. The host physically reclaims GPU 2–3 memory within ~10 s of scale-down (same as native), but Docker still accounts these GPUs as allocated to this container. An orchestrator (Kubernetes, Slurm, Ray cluster manager) sees the container holding all four nvidia.com/gpu requests for as long as the container runs. Scaling the container’s GPU allocation down requires killing the container — which terminates the in-place serving process, defeating the in-place benefit.

The scale-up inverse trap. Scale-up in the MGC regime works here only because the container was started with all 4 GPUs visible. A container started with --gpus '"device=0,1"' could not scale up to DP=4 because Ray cannot spawn new actors on GPUs the container’s device-cgroup does not permit. This follows directly from Docker’s cgroup semantics; we did not run the failing configuration explicitly.

The per-GPU-containers regime makes both traps disappear by construction. Each container has a single-GPU DeviceRequests.DeviceIDs:

ep-rank-0 DeviceIDs: ['0']
ep-rank-1 DeviceIDs: ['1']
ep-rank-2 DeviceIDs: ['2']
ep-rank-3 DeviceIDs: ['3']

Stopping ep-rank-2 and ep-rank-3 frees GPUs 2 and 3 at both levels: vLLM releases the CUDA context (memory to 4 MiB, same as native/MGC), and the orchestrator sees the container gone (no more nvidia.com/gpu requests from it). This is the scheduling-level elasticity multi-GPU containers cannot provide.

Why this is different from Exp A1 (Tenplex). Tenplex stops the training process to reshape parallelism — if the container is tied to Tenplex’s lifecycle, the orchestrator can recreate the container with a different --gpus set when Tenplex restarts. The trap window is bounded by the restart interval. For vLLM Elastic EP the process stays running indefinitely — there is no restart during which the container can be recreated — so the trap lasts for the entire serving lifetime.

5.5 NCCL transport at the container boundary

We extracted the NCCL-level transport selection from the vLLM serve log (NCCL_DEBUG=INFO). Observations:

Regime NCCL transport selected Evidence (log excerpt)
Native NVLink / CUDA IPC / SHM (same-node fast path) All three same-node gates pass; no NET/Socket entry for inter-rank channels.
Multi-GPU container NVLink / CUDA IPC / SHM (same-node fast path) Single container ⇒ single hostname, single /dev/shm, single netns. All three gates pass by construction.
Per-GPU containers NET/Socket/0 — TCP sockets over the Docker bridge network Every inter-rank channel log line ends with via NET/Socket/0 and isAllDirectP2p 0 directMode 0.

Per-GPU log excerpt:

[0] NCCL INFO Assigned NET plugin Socket to comm
[0] NCCL INFO Check P2P Type isAllDirectP2p 0 directMode 0
[0] NCCL INFO Channel 00/0 : 1[0] -> 2[0] [send] via NET/Socket/0
[0] NCCL INFO Channel 01/0 : 0[0] -> 1[0] [receive] via NET/Socket/0

NCCL picks the NET transport when any of its three same-node gates fails: the gethostname()-based hostHash, stat("/dev/shm").st_dev, or the abstract UDS used by cuMem for CUDA-IPC FD exchange. In the per-GPU regime all three fail by construction (separate hostnames, separate tmpfs, separate netns). This is the observation that motivated the common/shim/ LD_PRELOAD shim in the archived Exp A work; gate-taxonomy details and the shim’s interception points are in archived/exp_a_nccl_gates/. Its planned application to vLLM per-GPU containers is v5 §4.1. As §5.1 showed, applying the shim to vLLM is not expected to recover a 23% penalty on this specific workload — the penalty never materialised at plateau — but it remains the correct intervention for heavier bandwidth regimes.

5.6 Per-GPU-container-specific friction

Three concrete problems appear only in the per-GPU containers regime and do not surface in native or MGC. Each was addressable, either with a vLLM flag (5.6.1), by constraining the scaling pattern (5.6.2), or with a local patch (5.6.3).

5.6.1 The --data-parallel-size-local 1 requirement

Launching with the same flags that worked for native and MGC (--data-parallel-size 4 --data-parallel-backend ray) in the per-GPU regime fails immediately:

ValueError: Not enough resources to allocate 4 DP ranks on DP master
node ep-rank-0, possible to fit 1 DP ranks.

vLLM’s Ray placement logic assumes all DP ranks fit on the DP master node — i.e., it expects the master to be a multi-GPU host. The single-GPU container model violates this assumption. Passing --data-parallel-size-local 1 tells vLLM “this node contributes 1 DP rank; find the others elsewhere in the Ray cluster”. Placement groups then form correctly, one actor per container, and the cluster comes up.

This is a design-time assumption in the CLI, not a code bug: the API was built around the common case of a single multi-GPU node. A true K8s per-GPU-Pod deployment would need every operator to set -dpl 1, and elastic scaling (/scale_elastic_ep) would need to keep the assumption honoured across scale events — a nuance that does not apply to MGC.

5.6.2 /scale_elastic_ep from cold DP=4 is unrecoverable

If the cluster comes up at DP=4 (rather than DP=2 scaled up first), a scale-down 4→2 hits the same EPLB num_redundant ≥ 0 assertion that §4.2 describes. In native that failure bricks a single serving process; in per-GPU containers it spans four containers coordinating over TCP, and the recovery story is correspondingly worse — the process enters a permanent 503 state, Ray actors are partially torn down, all 4 GPUs end up at 4 MiB, and restarting vllm serve in ep-rank-0 alone is insufficient (leftover Ray actors on the other containers cause conflicts). Full recovery requires tearing down and recreating all four containers.

The practical implication is that the 2→4→2 cycle pattern is mandatory for per-GPU containers under R=0; there is no safe way to start cold at DP=4.

5.6.3 Placement-group patch (patches/0001)

Cold DP=2 startup in a 4-Ray-node cluster trips AssertionError: Created 4 DP placement groups, expected 2 in RayDPClient.make_dp_placement_groups before model load. The inner bundle loop has an early exit when len(placement_groups) == dp_size, but the outer node loop does not — it keeps visiting remaining nodes and over-creates placement groups. The scale-up sibling add_dp_placement_groups in the same file already has the matching outer-loop early exit; the cold-start path simply missed it. patches/0001_placement_group_early_exit.patch (applied by Dockerfile.per_gpu_containers to produce xtrans-vllm-ep-patched:v0.19.0) adds the outer-loop break to the cold path. With this patch the full 2→4→2 cycle runs at num_redundant_experts=0, matching the native and MGC convention. Safety across the supported VLLM_RAY_DP_PACK_STRATEGY values (strict, fill, span) is documented inline in the patch header.

6. Conclusions

  1. Live rescaling works in all three regimes on the 2→4→2 cycle. DP=4 plateau tok/s is 2.19× DP=2 initial tok/s under load-matched bench shapes, consistent with near-linear scaling.
  2. Containerisation imposes no measurable steady-state TPOT penalty at DP=4. Plateau gap MGC vs native = −6.8% (within native’s σ); per-GPU vs MGC = +0.8% (within both plateaus’ σ). The per-GPU regime is not materially slower than MGC under matched methodology for this workload.
  3. Multi-GPU container — orchestrator trap is real and permanent. Scale-down releases GPU memory on the host (GPU 2–3 drop to 4 MiB within ~10 s) but HostConfig.DeviceRequests.DeviceIDs stays [0,1,2,3] for the container’s lifetime. In Kubernetes the Pod holds 4 nvidia.com/gpu for its whole life regardless of actual use — memory-free but container-trapped.
  4. Per-GPU containers solve the trap at the orchestrator level: each container owns exactly one GPU, so stopping a container releases the GPU at both layers (vLLM CUDA-context release and Docker DeviceRequests removal).
  5. Per-GPU container boundary forces NCCL onto NET/Socket/0. All three NCCL same-node gates (hostname, /dev/shm, abstract UDS / netns) fail, so NCCL picks the socket transport for every inter-rank channel. This is visible at the log level and consistent with the microbenchmark model in archived Exp A.
  6. The NCCL transport change does not translate to measurable serving slowdown at this DP and request shape. The microbenchmark’s 10–25× raw-bandwidth gap shrinks to <1% on the DP=4 serving plateau because EP all-to-all payloads are per-token-small. Larger DP, larger batch, or larger token-context may shift the balance toward bandwidth-bound; the v5 §4.1 shim track retains motivation for those regimes but cannot be credited with a 23% recovery on this workload.
  7. Tenplex vs vLLM distinction holds experimentally. Stop-restart systems can live in multi-GPU containers because the process terminates — the orchestrator gets a window to recreate the container with different --gpus. vLLM’s serving process stays running indefinitely — no such window — so multi-GPU containers cannot deliver cluster-level elasticity for in-place systems.
  8. Elastic EP has structural fragility at R=0. Scale-down from a cold DP=N start always violates num_physical ≥ num_logical. The 2→4→2 cycle works by conjuring redundancy at scale-up and consuming it at scale-down (mechanism_deep_dive.html §6); production deployments must either follow that cycle or run with num_redundant_experts > 0 from the start.
  9. Stateless NCCL groups are architecturally container-ready but performance depends on gate-passing. The groups form over TCP (no shared memory required), Ray cluster spans containers cleanly, engine actors get placed. What changes between regimes is the NCCL data-path transport, not the coordination layer.

Next steps. (a) Apply the common/shim/ LD_PRELOAD shim to the per-GPU-container setup and re-measure at larger DP and larger request shapes to see where the bandwidth gap begins to bind on serving. (b) Test scaling-via-container-lifecycle end-to-end — requires an orchestrator or vLLM-aware controller that coordinates container stop/start with elastic-EP reconfiguration. (c) Consider whether num_redundant_experts > 0 cold starts work reliably for scale-down without the num_redundant ≥ 0 crash.
