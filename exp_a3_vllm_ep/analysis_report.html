1. Objective

Characterize vLLM’s Elastic Expert Parallelism (Elastic EP) mechanism — the first production-grade system for in-place live rescaling of GPU resources during MoE inference — as a baseline for studying what happens when this mechanism meets container boundaries. Specifically: confirm that live rescaling (4→2→4 EP workers) works on our hardware, measure rescaling time and throughput impact, and understand the internal mechanics deeply enough to predict where containerization will create friction.

Result: Live rescaling works on all three deployment regimes with progressively more friction. Native: 166.6 tok/s at DP=4, 1.91× linear scaling. Multi-GPU container: 154.1 tok/s (−7.5%), but Docker’s HostConfig.DeviceRequests.DeviceIDs is immutable at [0,1,2,3] for the container’s lifetime — freed GPUs are memory-free but container-trapped. Per-GPU containers: 127.8 tok/s (−23%), because NCCL’s same-node gates all fail and transport falls back to TCP sockets — validating the motivation for the LD_PRELOAD shim from Exp A. Per-GPU containers do solve the orchestrator trap (each container owns 1 GPU), but pay a real throughput cost until NCCL gates are spoofed.

2. Background

2.1 From Exp A1: Stop-Restart Systems Don’t Need Per-GPU Containers

Exp A1 (Tenplex) revealed that “elastic GPU scheduling” in Tenplex is actually stop-transform-restart: the training process fully terminates, state is transformed on the host, and a new process starts with a different GPU count. Under the multi-GPU container model, this pattern works fine — you stop the container, recreate it with a different --gpus flag, and restart. The GPUs are freed between container lifecycles. Per-GPU containers add complexity without benefit.

2.2 In-Place Systems Are Fundamentally Different

vLLM’s Elastic EP is in-place: the serving process stays running while GPUs are added or removed. The system reinitializes NCCL communicators, transfers expert weights, and reshuffles expert placement — all without restarting. This creates a fundamentally different interaction with containers:

  • Multi-GPU container (prediction): Scale-down works internally, but the freed GPUs remain trapped in the container. Docker’s immutable --gpus flag prevents the cluster from reclaiming them. Scale-up is impossible — Docker cannot hot-add GPUs to a running container. The in-place benefit is negated.
  • Per-GPU container (prediction): Each EP worker lives in its own container. Scale-down = stop containers, GPUs immediately free. Scale-up = start new containers, new workers join. The container lifecycle directly expresses the scaling semantics.

2.3 Why MoE + Expert Parallelism

Mixture-of-Experts models are uniquely suited for per-GPU container deployment. Each expert is a self-contained computation unit that maps naturally to one GPU and one container. Unlike tensor parallelism (which shards a single tensor across GPUs, requiring tight coupling), expert parallelism routes tokens to independent expert modules via All-to-All communication. The “Expert-as-a-Service” paradigm (arXiv ’25) already proposes disaggregating experts into independent services — precisely what per-GPU containers provide at the infrastructure level.

2.4 vLLM Elastic EP: Technical Overview

Elastic EP shipped in vLLM v0.17.0 (March 2026). When GPU resources change, vLLM brings up or tears down engine cores, reinitializes EP/DP communicators via stateless NCCL groups, transfers model weights GPU-to-GPU, and rebalances expert loads via EPLB algorithms. The serving process survives the scaling event without restart.

5. Results

The native regime ran in the following sequence: start at DP=2, benchmark, scale up to DP=4, benchmark, scale down to DP=2, benchmark. A second round (2→4, then 4→2 with a 503-window probe) was added to test repeatability and capture the service-interruption window.

Note on starting config: Our initial attempt started at DP=4 and tried to scale down to DP=2. This crashed with AssertionError: num_redundant ≥ 0 in the default EPLB policy. The bug: during scale-down, the rebalance code computes num_phy = num_phy // ep_group.size() * num_gpus, which reduces physical experts below logical count when num_redundant_experts=0. Starting at DP=2 and scaling up first creates physical redundancy (num_phy > num_log), after which scale-down back to DP=2 works because redundancy is consumed rather than turning negative. This matches vLLM’s own CI test (test_elastic_ep.py::test_elastic_ep_scaling).

5.1 Native Execution Plan (Revised)

Our initial plan was to start at DP=4 and scale down to DP=2. That crashed with AssertionError: num_redundant ≥ 0 in the EPLB rebalance policy:

File ".../vllm/distributed/eplb/policy/default.py", line 93, in replicate_experts
    assert num_redundant >= 0
AssertionError

The cause is in eplb_state.py:rearrange():

num_replicas = num_replicas // ep_group.size() * num_gpus  # scale-down

With num_redundant_experts=0, starting at DP=4 means num_phy = num_log = 128. Scaling down to DP=2 computes num_phy = 128 // 4 * 2 = 64, but num_log = 128, so num_redundant = -64. The assertion fires. Scale-down-first from a zero-redundancy start is fundamentally broken: it would have to silently drop experts.

Working plan: start at DP=2, scale up first. Scale-up creates physical redundancy (num_phy = 128 // 2 * 4 = 256 > 128), after which scale-down back to DP=2 consumes the redundancy and the assertion holds. This matches vLLM’s own CI test (tests/distributed/test_elastic_ep.py::test_elastic_ep_scaling).

Final native-regime sequence: DP=2 (bench) → scale-up → DP=4 (bench) → scale-down → DP=2 (bench) → scale-up again (round 2) → scale-down again (round 2, with 503-window probing).

5.5.3 The trap: host-side GPU memory and Docker’s view

At every scale event we snapshotted two independent views of the 4-GPU state: (1) nvidia-smi from the host (what memory is actually resident on each GPU), and (2) docker inspect HostConfig.DeviceRequests (what GPU allocation Docker still assigns to the container). Results:

Post-scale-up (+12 s) 36461, 36461, 35637, 35797 [0, 1, 2, 3]

Core finding of the multi-GPU-container regime: HostConfig.DeviceRequests.DeviceIDs is immutable at [0, 1, 2, 3] for the entire container lifetime — regardless of whether vLLM is actively using 2 GPUs or 4 GPUs inside. The host does see GPU 2–3 memory drop to 4 MiB within ~10 s of scale-down (the memory physically releases, same as native), but Docker still accounts these GPUs as allocated to this container. An orchestrator (Kubernetes, Slurm, Ray cluster manager) sees the container holding all four nvidia.com/gpu requests for as long as the container runs. Scaling the container's GPU allocation down requires killing the container — which terminates the in-place serving process.

Why this is fundamentally different from Exp A1 (Tenplex): Tenplex stops the training process to reshape parallelism. If the container is tied to Tenplex's lifecycle, the orchestrator can recreate the container with a different --gpus set when Tenplex restarts — the GPU-trap window is bounded by the restart interval. For vLLM Elastic EP, the process stays running indefinitely. There is no restart during which the container can be recreated with a different GPU count. The trap lasts for the entire serving lifetime.

5.5.4 Scale-up and the inverse problem

Scale-up in the multi-GPU-container regime works here because the container was started with all 4 GPUs visible. But this is only the lucky case: if the container had been started with --gpus '"device=0,1"' (only 2 GPUs visible), vLLM’s scale-up to DP=4 would fail because Ray cannot spawn new actors on GPUs that the container’s device-cgroup does not permit. The inverse trap: a container cannot acquire GPUs that were not in its --gpus set at launch time. This was noted as Step 4 in the research plan’s multi-GPU-container design and logically follows; we did not run this configuration explicitly because the predicted failure is a direct consequence of Docker’s cgroup semantics rather than any vLLM-specific behaviour.

5.6 Per-GPU Containers Regime

The per-GPU-container regime puts each EP rank in its own container: 4 Docker containers, each launched with --gpus '"device=N"' where N ∈ {0, 1, 2, 3}, a custom Docker bridge network xtrans-per-gpu joining them, each container running a Ray worker (the head lives in ep-rank-0). The API server (vllm serve) is exec’d inside ep-rank-0 and its Ray-backed engine actors are placed one per container.

The expected finding, stated in the research plan, was friction: per-GPU containers would expose problems that multi-GPU containers conceal. We hit three distinct problems, each qualifying as a research finding in its own right.

5.6.2 Finding #1 — vLLM's DP-master-node assumption breaks

Launching with --data-parallel-size 4 --data-parallel-backend ray (the settings that worked in the native and multi-GPU container regimes) produces an immediate failure:

ValueError: Not enough resources to allocate 4 DP ranks on DP master node
ep-rank-0, possible to fit 1 DP ranks.

vLLM’s Ray placement logic assumes all DP ranks fit on the DP master node — i.e., it expects the master to be a multi-GPU host. The single-GPU container model violates this assumption. The workaround is to also pass --data-parallel-size-local 1, which tells vLLM “this node contributes 1 DP rank; find the others elsewhere in the Ray cluster.” After this flag, placement groups are created correctly, one actor is spawned per container, and the cluster comes up.

This is a design-time assumption, not a code bug: the CLI was built around the common case of a single multi-GPU node. In a true K8s per-GPU Pod deployment, every operator would need to know to set -dpl 1. Elastic scaling (/scale_elastic_ep) would also need to keep this assumption honoured across scale events, a nuance that does not apply to multi-GPU containers.

5.6.3 Finding #2 — NCCL falls back to TCP sockets

With the cluster serving, we extracted the NCCL-level transport selection from debug logs (NCCL_DEBUG=INFO). Every single inter-rank channel shows the same transport:

[0] NCCL INFO Assigned NET plugin Socket to comm
[0] NCCL INFO Check P2P Type isAllDirectP2p 0 directMode 0
[0] NCCL INFO Channel 00/0 : 1[0] -> 2[0] [send] via NET/Socket/0
[0] NCCL INFO Channel 01/0 : 0[0] -> 1[0] [receive] via NET/Socket/0
[0] NCCL INFO Connected all rings, use ring PXN 0 GDR 0

Reading the flags:

  • Socket NET plugin: NCCL selected its TCP-sockets transport (not NVLink, not SHM, not NVLink-P2P, not InfiniBand).
  • isAllDirectP2p 0 directMode 0: CUDA P2P check failed — NCCL couldn’t establish direct GPU-to-GPU memory access between sibling containers.
  • PXN 0 GDR 0: no PCIe cross-NUMA bridging (PXN), no GPUDirect RDMA (GDR).
  • via NET/Socket/0: every send/receive channel traverses TCP over the Docker bridge network.

On our hardware, GPU 0 ↔ GPU 1 has NVLink NV12 (~600 GB/s bidirectional). In the per-GPU-container regime, that NVLink is bypassed; the link effectively becomes loopback TCP between two containers on the same host bridge, capped around 10–15 GB/s. The throughput consequence:

Config Transport Throughput (tok/s) vs. native
Native DP=4 NVLink / CUDA IPC / SHM 166.6 100%
Multi-GPU container DP=4 NVLink / CUDA IPC / SHM 154.1 −7.5%
Per-GPU containers DP=4 TCP sockets 127.8 −23.3%

The 23% loss is smaller than one might expect given the raw bandwidth gap (~40×) — this is because EP dispatch is not bandwidth-bound for our 128-expert / 4096-token / 4096-hidden configuration. A larger context or higher expert count would amplify the penalty. Multi-node deployments (where TCP is the only option anyway) would not see this degradation, but single-node deployments with per-GPU containers are paying a real cost.

Why this happens is exactly the problem characterised in Exp A (archived). NCCL decides whether two ranks are on the same node by three gates:

  1. hostHash collision: hash(gethostname() + /proc/sys/kernel/random/boot_id). Each container has its own hostname (ep-rank-0 through ep-rank-3), so the hostHashes differ.
  2. shmDev match: stat("/dev/shm").st_dev. Different containers have different tmpfs mounts for /dev/shm, so device IDs differ. (--ipc=host partially mitigates this but doesn’t fully unify the view.)
  3. IPC socket exchange: abstract UDS used by cuMem to exchange FDs for CUDA IPC. Abstract UDS are network-namespace-scoped, so separate network namespaces prevent the exchange.

Any one of these gates failing causes NCCL to conclude “these are different nodes” and pick the NET (sockets) transport. In our setup, all three gates fail.

This validates the Exp A motivation. The LD_PRELOAD shim at common/shim/ was built to spoof the three gates and let NCCL pick NVLink-P2P between containers. The per-GPU-container result here is the uncorrected baseline. With the shim applied, we should see cross-container throughput close to the multi-GPU-container regime (or native). Applying the shim to vLLM is a natural next experiment.

5.6.4 Finding #3 — Elastic scaling API doesn't recover the process

Triggering /scale_elastic_ep 4→2 in this configuration hit the same EPLB num_redundant ≥ 0 assertion that we documented in the native regime when scaling down from a cold DP=4 start. We already knew this was an EPLB bug independent of containerisation; what’s new in the per-GPU-container regime is that vLLM cannot recover from the failed scaling event in this topology. The process enters a permanent 503 state, Ray actors are partially torn down, and all 4 GPUs end up at 4 MiB (completely idle). The serving service is effectively dead and must be restarted from scratch.

In the native regime the same bug also bricked the serving process, but the failure was contained to a single process. In per-GPU containers the failure spans four containers coordinating over TCP, and the recovery story is correspondingly more complicated: Ray workers are still up (waiting for instructions), but the vLLM engine actors that lived on them are gone. Restarting vllm serve in ep-rank-0 alone is not sufficient; the leftover Ray actors and stale connections on the workers cause conflicts.

5.6.5 The intended benefit — GPU reclamation at the container boundary

Despite the friction, per-GPU containers do deliver the predicted benefit. Each container’s HostConfig.DeviceRequests.DeviceIDs contains only its own single GPU:

ep-rank-0 DeviceIDs: ['0']
ep-rank-1 DeviceIDs: ['1']
ep-rank-2 DeviceIDs: ['2']
ep-rank-3 DeviceIDs: ['3']

Compare to the multi-GPU-container regime, where one container claimed ['0','1','2','3'] for its entire lifetime. Stopping ep-rank-2 and ep-rank-3 in the per-GPU-container regime would free GPUs 2 and 3 at both levels: vLLM releases the CUDA context (memory to 4 MiB, as in the multi-GPU-container regime) and the orchestrator sees the container gone (no more nvidia.com/gpu requests). This is exactly the scheduling-level elasticity that the multi-GPU container cannot provide.

The DP=2 post-down figure (96.35) is higher than DP=2 initial (73.07) — warmup effect: TTFT drops from 1096 ms (cold) to 165 ms (warm) once EPLB has rebalanced and CUDA graphs are primed.
  • Across any row: the multi-GPU-container regime pays ~24% for containerisation overhead at DP=4 (shm-size caps, Ray-inside-container init), even though it keeps the same NCCL same-node transport as native. Per-GPU containers pay another ~20% on top of the multi-GPU container because NCCL falls to NET/Socket/0. The compounding is visible in TTFT: 387 ms native → 1777 ms multi-GPU container → 6405 ms per-GPU containers.
  • The DP=2 cells for the per-GPU-container regime are now populated: the original n/a entries encoded the observation that vLLM’s elastic-EP implementation did not straightforwardly support dynamic rank counts in a per-GPU-container cluster under default flags. The single placement-group patch is enough — 66.63 tok/s cold DP=2 and 69.64 tok/s post-scale-down — using the same num_redundant_experts=0 convention as native. The per-GPU-container regime’s ~9–28% DP=2 throughput shortfall vs native comes from the NCCL NET/Socket/0 fallback (v5 §4.1 shim track), not from the patch.

6. Analysis: How In-Place Elastic EP Actually Works

Elastic EP has to solve three hard problems that, individually, have no standard solution in PyTorch or NCCL: resizing a collective communication group without tearing down the process, redistributing experts across a changed rank topology while keeping the model correct, and moving model state between old and new ranks fast enough that the service interruption is short. This section explains the techniques vLLM uses for each, independent of the specific code.

6.1 Technique 1: Stateless NCCL groups — making collectives resizable

NCCL communicators are traditionally static: a group of ranks agrees on membership at init time (via ncclGetUniqueId on rank 0 and broadcast), and that membership is baked into every collective call. To change the group — add a rank, remove a rank, reshape topology — the standard approach is to destroy everything and call ncclCommInitRank again. But in vLLM’s case, the serving process is using the collective continuously, so tearing the communicator down tears down serving.

The technique vLLM uses is to decouple group formation from PyTorch’s global WORLD state:

AspectClassical NCCL/PyTorch groupStateless group (vLLM’s technique)
Membership state Registered in global torch.distributed._world Held only by participating ranks; no global registry
Rendezvous Out-of-band (env vars, launcher) In-band, over a TCP “coordinator store”
Port negotiation Fixed at launch time Rank 0 pre-binds sockets, publishes ports via the coord store
Coexistence One communicator per logical group Old and new communicators can exist simultaneously

Three independent channels are built for each group: (i) a device channel (NCCL) for GPU collectives, (ii) a CPU channel (Gloo) for host-side collectives, (iii) a TCP-store channel for object/metadata exchange. A coordinator store — a small TCP key-value server — is the only piece of global infrastructure; everything else is negotiated in-band. This is why the scheme extends beyond a single machine: there is no assumption of shared memory, shared hostname, or shared PID namespace.

The resize technique is then simple in principle (even if the synchronization is subtle): for scale-up, build a standby new group next to the running old group, drive all participants through a handoff barrier, then destroy the old group. Because both can coexist while the handoff completes, the GPU-side collective work can overlap group setup with useful traffic rather than stalling.

Implication for containers: stateless groups require only TCP connectivity — no shared memory, no shared IPC namespace, no shared gethostname(). This is the enabler of the per-GPU-container regime: each EP rank can in principle live in a separate container (or separate node) and still join a single collective, provided NCCL’s own transport selection cooperates.

6.2 Technique 2: Weight transport during scale-up

When new DP ranks are added, they start with no model weights loaded. There are two ways to populate them: re-load from disk or copy from an existing rank over GPU interconnect. For a 30 B parameter MoE model, disk-loading is slow (tens of seconds per rank) and bottlenecks on the storage subsystem. P2P copy over NVLink (or PCIe if NVLink is unavailable) is the faster path.

vLLM’s approach mixes the two:

  • Non-expert weights — transferred over GPU interconnect. Embeddings, attention projections, LM head, layer norms. These are ~28 GB for Qwen3-30B-A3B and are identical across all ranks, so one existing rank can stream them directly to each new rank using NCCL point-to-point sends.
  • Expert weights — each rank loads its own slice from disk. Expert weights are sharded across ranks, so the new ranks can’t just copy from an existing rank — they need different expert subsets. Loading from shared model storage is the natural choice (and vLLM has enable_ep_weight_filter to skip non-local expert weights during load, reading just the rank-local experts).

The scheduling technique is load-balanced sender-receiver pairing. Adding N ranks from a base of M ranks, each old rank sends the shared weights to ⌈N/M⌉ or ⌊N/M⌋ new ranks. For our 2→4 scale-up: 2 new ranks, 2 old ranks, each old rank sends to one new rank in parallel — the transfer overlaps perfectly.

Why this matters for per-GPU containers. The weight transfer uses the same stateless NCCL channel that EP dispatch uses. In a per-GPU container deployment, the transfer crosses container boundaries; if NCCL falls back to TCP instead of NVLink, transfer time blows up by 10–100× (roughly 600 GB/s → ~10 GB/s). This is where the NCCL gate analysis from Exp A (hostHash, /dev/shm stat, abstract UDS for cuMem FD exchange) re-enters the picture.

6.3 Technique 3: Expert reshuffling (EPLB) under changing topology

MoE experts are not uniform: real workloads activate some experts much more than others. A good static placement (expert i on rank i mod N) becomes a bad placement when traffic is skewed — one rank becomes the bottleneck for every forward pass. EPLB (Expert Parallel Load Balancer) addresses this by observing a sliding window of expert-activation counts and periodically recomputing placement.

Under static topology, EPLB has a clear job: minimize the load variance across ranks. Under changing topology, EPLB must do something stranger: handle the case where the rank count itself has changed, and decide which experts end up on which of the new ranks with minimum migration cost.

Distinction between logical and physical experts

This is the conceptual lever that makes the problem tractable. A model has num_logical experts (128 for Qwen3-30B-A3B) — these are fixed by the model architecture. But at run-time vLLM can maintain any number num_physical ≥ num_logical of physical replicas, where heavily-loaded logical experts get multiple physical copies to spread the load. The per-rank expert count is num_physical / EP.

Two invariants govern the reshuffle:

  • Every logical expert must be somewhere. Losing a logical expert would silently degrade model correctness.
  • num_physical ≥ num_logical. You can have redundancy; you cannot have under-coverage.

What reshuffling does in each scale direction

ScaleTopology changeReshuffle technique
Scale-up (more ranks) More physical slots available Replicate hot experts onto new ranks; cold experts stay put. New num_physical can exceed old, increasing redundancy.
Scale-down (fewer ranks) Fewer physical slots — some experts must move Construct a rank-mapping: departing ranks (> new_ep_size) map to −1 (“gone”); their experts are reassigned to surviving ranks. If num_redundant_experts was zero, this step must not lose any logical experts.

The bug we hit at DP=N→N/2 with zero redundancy. vLLM’s scale-down rebalance computes new num_physical = (old_num_physical / old_ep) × new_ep — a per-rank-proportional scaling. With zero redundancy at the start (num_physical = num_logical), halving EP halves num_physical, which falls below num_logical — the invariant num_physical ≥ num_logical is violated and the rebalance hits an assertion failure. The workaround — start at DP≤N/2 and scale up first — works because scaling up increases num_physical (adds physical replicas of existing logical experts), building the redundancy that later scale-down can consume.

6.4 How the three techniques compose during a scale-up

The three techniques above operate in a particular order to accomplish a single scale event. The essential dependency chain:

  1. Stateless group creation: a NEW standby group is negotiated over
     the coordinator store. Old group keeps serving.

        (requires: coordinator store reachable from new ranks)

  2. Weight transport: non-expert weights streamed from old ranks
     to new ranks over the new group's NCCL channel. Expert weights
     loaded from disk by new ranks in parallel.

        (requires: GPU P2P / NCCL working between old and new ranks)

  3. EPLB reshuffle: given observed load + new rank count, compute
     new logical-to-physical mapping. Experts may be replicated or
     migrated.

        (requires: agreement on physical expert count, rank topology)

  4. Atomic switch: all ranks barrier, destroy old group, activate new.
     Serving resumes on the new topology.

        (requires: all-ranks synchronization, two-stage barrier)

The middleware 503 window brackets steps 1–4: new requests are rejected from the moment the scale event is triggered until the switch completes. In-flight requests must complete on their current engine or fail; there is no request-migration mechanism.

6.5 Expert dispatch at run-time: AllGatherV + ReduceScatterV

Orthogonal to scale events, every MoE forward pass does an All-to-All to dispatch tokens to the experts that compute on them, and another to combine the results back. vLLM supports several backends for this; we used allgather_reducescatter, which implements All-to-All as a pair of standard collectives rather than a native All-to-All kernel.

DISPATCH (AllGatherV):              COMBINE (ReduceScatterV):

All ranks gather all tokens so      All ranks scatter per-expert outputs
each holds the global batch         back to the rank that owns each token.

  R0:[own tokens] ─┐                 R0 gets back R0's tokens with
  R1:[own tokens] ─┤─▶ everyone      expert outputs summed across ranks.
  R2:[own tokens] ─┤    holds all
  R3:[own tokens] ─┘    tokens

Per MoE layer, the communication volume is roughly 2 × N_tokens × hidden_dim × 2 bytes (dispatch + combine, BF16). For the default 4096-token context at 4096 hidden dim, that’s ~128 MB per layer. With 48 MoE layers in Qwen3-30B-A3B, a full forward has ~6 GB of EP communication. This is why interconnect topology dominates scaling efficiency: the 1.91× we saw on our heterogeneous node (NVLink between GPU0–1, PCIe between GPU2–3, QPI between groups) is already close to optimal; homogeneous NVLink should give something closer to 2.0×.

6.6 Why this is relevant to the container question

Each technique in this section makes a different assumption about the execution environment:

TechniqueAssumptionContainer-friendly?
Stateless groups TCP connectivity to coord store and peer NCCL ports Yes — no shared memory, no global WORLD
Weight transport (P2P) NCCL chooses a fast transport between sender and receiver GPU Only if NCCL gates pass; otherwise TCP fallback (slow)
EPLB reshuffle All ranks agree on expert counts and observe load Yes — no new environmental assumptions
Run-time All-to-All NCCL collective performance dominates serving throughput Only if NCCL gates pass; otherwise huge throughput loss

The per-GPU-container regime stresses the second and fourth items: per-GPU containers have separate hostnames, separate /dev/shm, separate network namespaces. Whether NCCL still picks NVLink / NVLink-P2P between sibling containers — or falls back to TCP — is what determines whether the architectural container-readiness of stateless groups actually translates into throughput parity.

7. Conclusions

  1. Live rescaling works across all three topologies. Scale-up 2→4 in 25.8 s native / 39.2 s multi-GPU container / not tested cleanly in per-GPU container because of the EPLB bug from a cold DP=4 start. Throughput scales near-linearly on native (1.91× on 2× GPUs, 95% efficiency).
  2. Progressive overhead as containerisation increases: native 166.6 tok/s → multi-GPU container 154.1 tok/s (−7.5%) → per-GPU container 127.8 tok/s (−23%). The first step is small container overhead; the second step is NCCL transport falling to sockets.
  3. Multi-GPU container — orchestrator trap confirmed. Scale-down releases GPU memory on the host (GPU 2–3 drop to 4 MiB within ~10 s) but HostConfig.DeviceRequests.DeviceIDs remains [0,1,2,3] for the container’s entire lifetime. In K8s the Pod holds 4 nvidia.com/gpu for its whole life regardless of actual use. Memory-free but container-trapped.
  4. Per-GPU containers — trap solved, NCCL performance dies. Each container owns exactly one GPU, so stopping a container truly releases the GPU at the orchestrator level. But NCCL’s three same-node gates all fail (different hostnames, different /dev/shm, different netns), so NCCL picks TCP sockets for every inter-rank channel — 23% throughput loss vs. native.
  5. The NCCL shim from Exp A is exactly what the per-GPU-container regime needs. Spoofing hostname, /dev/shm stat, and abstract UDS bind/sendmsg should let NCCL pick NVLink-P2P across containers and recover the ~23% loss. The per-GPU regime is the uncorrected baseline for that measurement.
  6. vLLM's DP-master-node assumption is incompatible with per-GPU containers by default. --data-parallel-size-local 1 fixes it but is not the default. The CLI was designed around multi-GPU master nodes — a hidden multi-GPU-container assumption inside the API surface itself.
  7. Tenplex vs vLLM distinction holds experimentally. Stop-restart systems (Tenplex) can live in multi-GPU containers because the process terminates — the orchestrator gets a window to recreate the container with different --gpus. vLLM’s serving process stays running indefinitely — no such window — so multi-GPU containers cannot deliver cluster-level elasticity for in-place systems.
  8. Elastic EP remains fragile. Scale-down from a cold DP=N start fails the EPLB num_redundant ≥ 0 invariant; the same bug was the one unrecoverable failure in the per-GPU-container regime. Production deployments must start at DP≤N/2 and scale up first OR set num_redundant_experts > 0.
  9. Stateless NCCL groups are architecturally container-ready but depend on NCCL gate-passing for performance. The groups themselves form (only TCP connectivity required), Ray cluster spans containers cleanly, engine actors get placed. What fails is not the coordination layer but the data path.

Next steps: (a) Apply the LD_PRELOAD shim (common/shim/) to the per-GPU container setup and re-measure. Expected: same Ray / vLLM behaviour (the per-GPU-container regime already works), but NCCL should see the gates pass and pick NVLink-P2P, recovering most of the 23% loss. (b) Investigate whether vLLM’s num_redundant_experts > 0 path works reliably for scale-down without hitting the num_redundant ≥ 0 assertion. (c) Test scaling-via-container-lifecycle cleanly — probably requires an orchestrator or a vLLM-aware controller that coordinates container stop with elastic-EP reconfiguration.
