1. Objective

Characterize vLLM’s Elastic Expert Parallelism (Elastic EP) mechanism — the first production-grade system for in-place live rescaling of GPU resources during MoE inference — as a baseline for studying what happens when this mechanism meets container boundaries. Specifically: confirm that live rescaling (4→2→4 EP workers) works on our hardware, measure rescaling time and throughput impact, and understand the internal mechanics deeply enough to predict where containerization will create friction.

Result (preliminary): Source-level investigation reveals that Elastic EP uses a sophisticated state machine with stateless NCCL groups, GPU-to-GPU P2P weight transfer, and EPLB expert reshuffling. The architecture is fundamentally amenable to per-GPU containers because stateless groups use TCP-based coordination with no shared-memory assumptions. Phase 1 performance numbers pending.

2. Background

2.1 From Exp A1: Stop-Restart Systems Don’t Need Per-GPU Containers

Exp A1 (Tenplex) revealed that “elastic GPU scheduling” in Tenplex is actually stop-transform-restart: the training process fully terminates, state is transformed on the host, and a new process starts with a different GPU count. Under the multi-GPU container model, this pattern works fine — you stop the container, recreate it with a different --gpus flag, and restart. The GPUs are freed between container lifecycles. Per-GPU containers add complexity without benefit.

2.2 In-Place Systems Are Fundamentally Different

vLLM’s Elastic EP is in-place: the serving process stays running while GPUs are added or removed. The system reinitializes NCCL communicators, transfers expert weights, and reshuffles expert placement — all without restarting. This creates a fundamentally different interaction with containers:

  • Multi-GPU container (Phase 2 prediction): Scale-down works internally, but the freed GPUs remain trapped in the container. Docker’s immutable --gpus flag prevents the cluster from reclaiming them. Scale-up is impossible — Docker cannot hot-add GPUs to a running container. The in-place benefit is negated.
  • Per-GPU container (Phase 3 prediction): Each EP worker lives in its own container. Scale-down = stop containers, GPUs immediately free. Scale-up = start new containers, new workers join. The container lifecycle directly expresses the scaling semantics.

2.3 Why MoE + Expert Parallelism

Mixture-of-Experts models are uniquely suited for per-GPU container deployment. Each expert is a self-contained computation unit that maps naturally to one GPU and one container. Unlike tensor parallelism (which shards a single tensor across GPUs, requiring tight coupling), expert parallelism routes tokens to independent expert modules via All-to-All communication. The “Expert-as-a-Service” paradigm (arXiv ’25) already proposes disaggregating experts into independent services — precisely what per-GPU containers provide at the infrastructure level.

2.4 vLLM Elastic EP: Technical Overview

Elastic EP shipped in vLLM v0.17.0 (March 2026). When GPU resources change, vLLM brings up or tears down engine cores, reinitializes EP/DP communicators via stateless NCCL groups, transfers model weights GPU-to-GPU, and rebalances expert loads via EPLB algorithms. The serving process survives the scaling event without restart.

6. Analysis: Elastic EP Internal Mechanics

This section documents the detailed internal mechanics of vLLM’s Elastic EP, derived from source-level investigation of vLLM v0.19.0. Understanding these mechanics is essential for predicting how containerization will affect scaling behavior in Phases 2 and 3.

6.1 Scaling API and Request Handling

Elastic EP exposes a single HTTP endpoint for triggering rescaling:

POST /scale_elastic_ep
Content-Type: application/json

{"new_data_parallel_size": 2, "drain_timeout": 120}

When this endpoint is called, the system enters a full service interruption:

  1. A ScalingMiddleware sets a global flag and returns 503 Service Unavailable to all incoming requests.
  2. If VLLM_ELASTIC_EP_DRAIN_REQUESTS is set, the system waits up to drain_timeout seconds for in-flight requests to complete (polling every 1 second).
  3. The actual reconfiguration begins (described below).
  4. When complete, the middleware flag is cleared and serving resumes.

No request migration: Requests already queued on an engine that will be shut down are not migrated to surviving engines. They either complete before shutdown or fail. There is no continuity guarantee for in-flight requests during scaling.

6.2 Scale-Down State Machine

Scale-down is the simpler operation. The client-side coordinator (AsyncMPClient._scale_down_elastic_ep) sends a ReconfigureDistributedRequest to each engine:

  • Engines with dp_rank < new_size receive KEEP_CURRENT_RANK.
  • Engines with dp_rank ≥ new_size receive SHUTDOWN_CURRENT_RANK.

The client immediately truncates its engine lists (stopping request routing to removed engines), then waits for the setup switch to complete. The Ray placement groups for removed engines are cleaned up via ray.util.remove_placement_group().

On the engine side, remaining engines run a simplified state machine:

  1. PREPARE: Barrier synchronization across remaining engines.
  2. EPLB_RESHUFFLE: Experts from removed ranks are redistributed to remaining ranks via a rank-mapping algorithm (ranks ≥ new_ep_size map to −1, meaning “removed”).
  3. SWITCH_AND_PREPARE: Create standby groups, destroy old groups, switch to new NCCL communicators.
  4. COMPLETE: Resume serving.

6.3 Scale-Up State Machine (Detail)

Scale-up is significantly more complex. Two parallel state machines run — one for existing engines, one for new engines — synchronized via two-stage barriers.

Existing Engine States

#StateAction
0 WAIT_NEW_CORE_ENGINES_INIT Block until new Ray actors are initialized and send NEW_CORE_ENGINES_INIT_READY notification.
1 CREATE_STANDBY_GROUPS Two-stage barrier. Create standby stateless NCCL groups (DP, EP, EPLB) for the new configuration. Old groups remain active. Both old and new groups coexist temporarily.
2 TRANSFER_EXPERT_MAPPING Rank 0 broadcasts physical_to_logical expert mapping to all existing engines. New engines learn which experts live where.
3 WAIT_NEW_CORE_ENGINES_WEIGHTS_INIT Block until new engines load model with dummy weights and are ready to receive real weights.
4 TRANSFER_WEIGHTS GPU-to-GPU P2P transfer of non-expert model parameters from existing engines to new engines via batch_isend_irecv. Expert weights are NOT transferred (they are loaded from disk). Sender-receiver pairing is load-balanced across existing engines.
5 SYNC_KV_CACHE_MEMORY_SIZE Synchronize KV cache memory budget across all DP ranks (new engines may have different available memory).
6 SWITCH_AND_PREPARE Atomic switch: destroy old DP group, activate standby group. All-reduce engine state (running flag, wave counter, step counter). Recompile MoE modules for new EP size.
7 EPLB_RESHUFFLE Redistribute experts across new EP group using EPLB algorithm. All engines (old + new) participate.
8 COMPLETE Resume serving with new DP size.

New Engine States

#StateAction
0 PRE_KV_INIT Load model with dummy weights. Receive expert mapping from rank 0. Initialize KV cache. Receive real weights via P2P from sender engine.
1 PREPARE All-reduce engine state with existing engines.
2 EPLB_RESHUFFLE Participate in global expert redistribution.
3 COMPLETE Join serving.

Two-Stage Barrier Synchronization

The state machine uses a two-stage barrier to handle asynchronous arrival of reconfiguration signals:

  1. Stage 1 (TCP store): 5-second timeout. If some engines are still in model forward when others reach the barrier, the timeout fires and those engines execute one more forward step before retrying.
  2. Stage 2 (PyTorch distributed barrier): Synchronous, no timeout. Only reached after all engines pass Stage 1.

This mechanism ensures correctness even when engines are at different points in the forward loop when the reconfiguration request arrives.

6.4 Stateless NCCL Groups

The key innovation enabling elastic scaling is the use of stateless NCCL groups instead of PyTorch’s standard global process groups. This has direct implications for containerization.

How Stateless Groups Differ

AspectStandard GroupsStateless Groups
Initialization Requires global WORLD group TCP store coordination only
State Registered in torch.distributed._world No global state; isolated per group
Lifecycle Implicit; coupled to WORLD Explicit create/destroy; independent
Port allocation Fixed at init time Pre-bound sockets; published via coord store
Multiple groups All within same WORLD Can coexist (old + standby)
Cross-container Requires shared namespace Only needs TCP connectivity

Each StatelessGroupCoordinator creates three independent communication channels:

  1. Device group (NCCL): GPU-to-GPU collective operations (AllReduce, AllGatherV, ReduceScatterV for All-to-All).
  2. CPU group (Gloo): CPU-side collective operations.
  3. TCP store group: Object broadcast and metadata exchange.

Rank 0 of each group allocates listening sockets and publishes port numbers to a coordinator TCP store. Other ranks fetch ports from the store. This eliminates TOCTOU race conditions and enables truly independent group formation.

Key insight for Phase 3: Because stateless groups require only TCP connectivity (no shared memory, no shared PID namespace, no global WORLD group), they can theoretically work across container boundaries with no modification. Each EP worker in a separate container needs only network access to the coordinator store and to other workers’ NCCL ports. This is the architectural basis for per-GPU container EP.

6.5 Expert Weight Transfer

During scale-up, non-expert model parameters (embeddings, attention layers, LM head) are transferred from existing engines to new engines via GPU-to-GPU P2P:

  • Method: torch.distributed.P2POp with isend/irecv, batched via batch_isend_irecv.
  • What transfers: All non-expert parameters (~27–28 GB for Qwen3-30B-A3B). Expert weights are loaded from disk by new engines, not transferred.
  • Sender-receiver pairing: Load-balanced. If adding N new workers and M old workers exist, each old worker sends to ⌈N/M⌉ or ⌊N/M⌋ new workers.

Estimated transfer time: For ~28 GB at NVLink bandwidth (∼600 GB/s), the raw P2P transfer takes ~47 ms. Actual time is dominated by NCCL collective overhead and barrier synchronization, expected to be 50–200 ms total per new engine.

6.6 All-to-All Communication Pattern

Expert parallelism uses AllGatherV + ReduceScatterV for expert dispatch and combine (the allgather_reducescatter backend):

DISPATCH (AllGatherV):          COMBINE (ReduceScatterV):

Each rank sends its tokens      Each rank sends expert outputs
to ALL other EP ranks.          back to originating ranks.

  Rank0 ──┬──▶ Rank0            Expert0..31 on Rank0 ──┬──▶ Rank0
  Rank1 ──┤──▶ Rank0            Expert32..63 on Rank1 ──┤──▶ Rank0
  Rank2 ──┤──▶ Rank0            Expert64..95 on Rank2 ──┤──▶ Rank0
  Rank3 ──┘──▶ Rank0            Expert96..127 on Rank3──┘──▶ Rank0

Communication volume per MoE layer: N_tokens × hidden_dim × 2 bytes × 2 (dispatch + combine). For 4096 tokens with hidden_dim=4096 in BF16: ~128 MB per layer.

Cross-container implication: In Phase 3 (per-GPU containers), this All-to-All communication must cross container boundaries for every MoE layer. On our hardware, GPU0↔GPU1 uses NVLink (fast), but GPU0↔GPU3 crosses QPI (slow). The NCCL shim (common/shim/) may be needed to prevent NCCL from falling back to TCP when it detects separate hostnames.

7. Conclusions (Preliminary)

  1. Elastic EP is architecturally container-ready: Stateless NCCL groups require only TCP connectivity, not shared memory or process namespaces. Per-GPU containers should work without modifying vLLM.
  2. Scaling causes full service interruption: All requests receive 503 during the entire scaling window. There is no request migration or gradual transition.
  3. Scale-up is complex: An 8-state state machine coordinates new engine creation, standby group formation, weight transfer, KV cache synchronization, group switching, and EPLB reshuffling.
  4. Scale-down is simpler: Engines at removed ranks receive a shutdown signal; remaining engines reshuffle experts and reinitialize communicators.
  5. Weight transfer is GPU-to-GPU P2P: ~28 GB for Qwen3-30B-A3B, estimated 50–200 ms per new engine at NVLink bandwidth.
  6. Phase 1 throughput and rescaling time: TBD.

Next steps: (1) Execute Phase 1 when GPUs are available: baseline throughput, live rescaling 4→2→4, measure rescaling duration and 503 window. (2) Phase 2: repeat in a multi-GPU container to confirm GPU trapping. (3) Phase 3: per-GPU containers with cross-container NCCL All-to-All.
