1. Objective

Exp A3 Phase 3 established that per-GPU containers eliminate the multi-GPU container’s “GPU-trap” problem at the architectural level — each container holds exactly one GPU, so releasing a container truly releases that GPU to the scheduler. But the same phase surfaced three concrete bugs in vLLM Elastic EP that made the scaling API unusable in that topology. Exp B’s objective is to turn Phase 3’s architectural-only elasticity story into a functionally working elastic cycle by patching the vLLM install inside a derived Docker image — without touching the upstream submodule (3rdparty/vllm/), without modifying the research plan’s model or hardware, and without requiring the NCCL bandwidth-recovery shim (v5 §4.1 is an orthogonal track).

Result: Two local patches (0001: EPLB scale-down precondition, 0002: placement-group outer-loop early exit) turn Phase 3 into a working elastic cycle with no pre-allocated redundancy, matching Exp A3 Phase 1 native's 2 → 4 → 2 pattern. Scale-up 49.93 s, scale-down 12.92 s, DP=2 post-scale-down throughput 67.53 tok/s (vs 62.21 tok/s at cold DP=2 initial), and 2 of 4 GPUs are fully released to 4 MiB (container-scoped reclamation) while all 4 containers remain Up. Patch 0001’s --eplb-config.num_redundant_experts redirect is a diagnostic fallback for operators that must cold-start above their minimum DP, not a workaround the primary cycle needs.

2. Background

Research plan v5 argues that per-GPU containers are the only path to cluster-visible elasticity for in-place systems like vLLM Elastic EP — multi-GPU containers keep the freed GPUs “memory-free but container-trapped” because Docker’s HostConfig.DeviceRequests.DeviceIDs is immutable for the container’s lifetime. Exp A3 Phase 3 built the per-GPU-container topology (four containers on a Docker bridge network, Ray cluster spanning them) and confirmed the architectural premise. It then uncovered three failure sites in vLLM 0.19.0 that Exp B addresses (one each for startup, cold-start, and scaling).

2.1 Finding #1 — DP-master-node resource assumption

Launching with --data-parallel-size 4 --data-parallel-backend ray fails immediately with ValueError: Not enough resources to allocate 4 DP ranks on DP master node, possible to fit 1. vLLM’s Ray placement logic assumes the DP master is a multi-GPU host; in a per-GPU setup it has a single GPU. The workaround is --data-parallel-size-local 1, which tells vLLM “this node contributes 1 rank, find the others elsewhere in the Ray cluster.” This is a design-time CLI assumption, not a bug; no patch is needed, but every deployment must pass the flag. Already documented in Exp A3 §5.6.2.

2.2 Finding #2 — Placement-group count mismatch for DP < Nnodes

Even with --data-parallel-size-local 1 set, starting with --data-parallel-size 2 against a 4-Ray-node cluster asserts:

AssertionError: Created 4 DP placement groups, expected 2

The cold-start function RayDPClient.make_dp_placement_groups walks every Ray node allocating placement groups. An inner break stops the per-node bundle loop once len(placement_groups) == dp_size, but the outer loop over nodes has no early exit, so it keeps visiting remaining nodes and over-creating PGs. The same file’s scale-up sibling add_dp_placement_groups already has the outer break (if num_pg_created >= num_pg_to_create: break) — the cold-start path simply missed it.

2.3 Finding #3 — EPLB num_redundant ≥ 0 on scale-down

Starting at DP=4 works, but POST /scale_elastic_ep {new_data_parallel_size: 2} crashes inside a Ray worker with:

assert num_redundant >= 0
AssertionError

Triggered by policy/default.py:93 in replicate_experts, which requires num_phy >= num_log. The feeder code at eplb_state.py computes num_replicas = num_replicas // ep_group.size() * num_gpus: a proportional shrink that preserves per-GPU slot count and reduces total. When num_redundant_experts=0 (the default), the total already equals num_logical, so any proportional shrink drops below it — for DP=4 → 2 with Qwen3’s 128 logical experts, 128//4×2 = 64 < 128, and the assertion fires. The serving process dies; in the multi-container topology the failure cascades across four Ray actors and the cluster is unrecoverable without a full teardown.

The container-scoped GPU-allocation invariant is preserved: each container’s DeviceIDs is a single element list ['N'], so stopping ep-rank-2 and ep-rank-3 externally releases GPUs 2 and 3 to the host scheduler with no further vLLM work. The headline invariant (two GPUs freed, scheduler can see it) holds either way; which two is an operational detail that a Track 3 coordinator would resolve by reading vLLM’s rank layout post-scale before issuing docker stop.

6. Analysis

6.1 What patch 0001 does

Patch 0001 inserts a 7-line precondition check at the top of eplb_state.rearrange()’s scale-down branch. The check detects the invariant violation before the buggy formula fires:

new_num_replicas = num_replicas // ep_group.size() * num_gpus
if new_num_replicas < model.num_logical_experts:
    raise ValueError(
        f"Scale-down from ep_size={ep_group.size()} to {num_gpus} "
        f"requires num_redundant_experts ≥ {model.num_logical_experts} "
        f"(current: {current_redundant}). Pass "
        f"--eplb-config.num_redundant_experts={model.num_logical_experts} "
        f"at vllm serve startup."
    )
num_replicas = new_num_replicas

This is a refuse-with-actionable-message, not a repair: when num_redundant_experts=0 the patch still cannot scale down (a proper repair would require growing per-GPU expert density across the cluster, see §6.3). What it does is replace a cryptic Ray-worker AssertionError at policy/default.py:93 with a ValueError that names the exact CLI flag the operator needs to set. Combined with --eplb-config.num_redundant_experts=128, this is a complete and reproducible scale-down path on the default vLLM deployment — which we validated end-to-end in Config C.

The math behind the required value: for scale-down from DP=N to DP=M to keep new_num_phy ≥ num_log, the operator needs num_redundant_experts ≥ num_log × (N − M) / M. For N=4, M=2, num_log=128 that is 128 redundant experts. The patch suggests num_logical_experts as a conservative upper bound that satisfies any supported scale-down pair; a TODO comment in the patch notes that the tight per-pair minimum could be computed inline.

6.2 What patch 0002 does

Patch 0002 adds two lines after the inner bundle-allocation loop inside RayDPClient.make_dp_placement_groups:

            if len(placement_groups) == dp_size:
                break
+           if len(placement_groups) ≥ dp_size:
+               break

        if len(placement_groups) < dp_size:
            raise ValueError(...)

The inner break (already present) exits the bundle-allocation loop for one Ray node; the new outer break exits the node loop. Safe across all three VLLM_RAY_DP_PACK_STRATEGY values (strict, fill, span): in the first two, each iteration synchronously emits placement groups with no accumulated state; in span, placement groups are emitted only after collected_bundles reaches world_size, at which point collected_bundles = []. An outer break can therefore never leave a partial bundle remainder behind.

The correctness of the fix is additionally supported by the observation that the file’s scale-up sibling add_dp_placement_groups already has the identical pattern (if num_pg_created ≥ num_pg_to_create: break). The cold-start path simply forgot the outer break; we are aligning with upstream intent, not working around it.

6.3 Why the grow-density repair is a structural refactor

The obvious “real fix” for Finding #3 is to clamp num_replicas to max(num_logical_experts, ...) so per-GPU expert density grows on scale-down: at DP=4→2 the two surviving GPUs would each hold 64 physical experts instead of 32. This was attempted (see the original patch 0001 iteration) and clears the first assertion — but immediately trips a second:

assert old_global_expert_indices.shape[1] == new_global_expert_indices.shape[1]

in rebalance_execute.py:592. Root cause: _map_new_expert_indices_with_rank_mapping hard-codes num_local_physical_experts = new_num_physical_experts // new_ep_size, i.e., the new per-GPU density is assumed to equal the old per-GPU density. Growing the density violates this assumption; the mapping function produces a tensor of a different shape than the caller expects.

Fixing this end-to-end cascades into at least five places (see Track 1b in the Exp B README):

  1. The two mapping functions (_map_{new,old}_expert_indices_with_rank_mapping) need the actual old shape as an explicit parameter, not derived from the new shape.
  2. Per-rank expert_weights allocation — each surviving GPU must grow its expert-slot count (32 → 64 for DP=4→2). Current allocation is fixed at model load time.
  3. physical_to_logical_map per-rank shape is similarly sized at init.
  4. expert_load_window indexing under the grown density.
  5. move_to_buffer/move_from_buffer assume slot i stays on the same GPU across scale events; grow-density migrates slots across GPUs, requiring extended transfer semantics.

This is a multi-file upstream redesign, not a one-line fix. It is deferred as Track 1b in the Exp B scope and is the principal remaining obstacle to making scale-down work with the default num_redundant_experts=0 configuration.

6.4 When the redundancy redirect matters (alternate deployment path)

The validated cycle (Config C) uses num_redundant_experts=0 because it starts at DP=2; scale-up internally grows physical experts to 256 (R=128 effective) as a consequence of the per-GPU invariant (§6.5), and scale-down reverses that. Patch 0001’s error-message redirect (--eplb-config.num_redundant_experts=128) is therefore not on the critical path for Phase 3’s 2 → 4 → 2 cycle.

The redirect is still a real artifact in the codebase, because some deployments cannot cold-start at their minimum DP: for example, an operator on a quota system who must provision all 4 GPUs at once, or a service that receives peak traffic immediately on boot. Those deployments face a cold DP=4 start that cannot reach DP=2 without pre-allocated redundancy. Patch 0001 converts the resulting crash (a cryptic Ray-worker AssertionError) into a ValueError naming the exact CLI flag to set. A prior end-to-end run documented in the lab notebook (results/20260421-1258/analysis.md, earlier addendum) validated the redirect path: cold DP=4 with num_redundant_experts=128 → scale-down to DP=2 succeeds in 4.56 s, DP=4 throughput lands at 119.65 tok/s (−6.4% vs unpatched). The cost of that path is ~35.8 GB per-GPU expert load instead of ~16.4 GB, which fits on 40 GB A100s and would bite harder on smaller GPUs.

The framing shift matters: 0001 is a diagnostic fallback, not a workaround the main elasticity story relies on.

Limitation: the ValueError raised by patch 0001 currently propagates through the Ray worker and kills the engine core — the HTTP response to a bad /scale_elastic_ep call is a timeout rather than a 4xx with the error message. The message shows up in server logs, which is actionable for a human operator but not for a programmatic controller. A proper HTTP-layer rejection requires reaching into the elastic-EP request handler before dispatch; tracked in the Track 1b sketch list.

6.5 Mechanism: expert placement under scale events

Two concrete questions motivate this subsection, both with the same underlying answer:

  • Q1. Why does a cold DP=2 → 4 → 2 cycle succeed with num_redundant_experts=0, while a cold DP=4 → 2 scale-down from the same R=0 fails at the num_redundant ≥ 0 assertion?
  • Q2. Does Phase 3 (per-GPU containers) need more redundancy than Phase 1 (native)? Both validated cycles used R=128; is there a mechanism difference?

Both resolve to one observation about how EPLB sizes physical experts across scale events.

The per-GPU invariant

The scale-down path in vllm/distributed/eplb/eplb_state.py:rearrange (the site patch 0001 guards) computes:

num_physical_new = num_physical_old // old_ep_size * new_ep_size

Which is equivalent to:

per_gpu_new  =  num_physical_new / new_ep_size
             =  (num_physical_old // old_ep_size * new_ep_size) / new_ep_size
             =  num_physical_old / old_ep_size  =  per_gpu_old

The formula preserves per-GPU expert count across scale events. Scale-up applies the symmetric direction: new GPUs each receive per_gpu expert slots, which EPLB populates by replicating logical experts via load-aware bin packing (uniformly at cold-start, when no load statistics have been observed yet). The invariant is not a performance choice — every downstream buffer in vLLM (expert_weights, physical_to_logical_map, expert_load_window, the shape assertions in rebalance_execute.py) is sized against per_gpu. Changing it mid-cycle is the five-point cascade that Exp B’s Track 1b documents.

The invariant is concretely enforced by elastic_ep_execute.switch_and_prepare, which runs during every scale event. On scale-up it F.pads expert_load_pass and expert_load_window from the old to the new num_local × ep_size, and mutates parallel_config.eplb_config.num_redundant_experts in place so subsequent rearrange() calls see the grown total. On scale-down it slices the same buffers down symmetrically. The formula above lives inside rearrange(); switch_and_prepare is what makes the new num_physical_experts value visible to the rest of the model state.

Scenario A — cold DP=2 with R=0, full 2 → 4 → 2 cycle (Phase 1 native reference)

State num_log num_redundant num_phy ep_size per_gpu Notes
cold DP=2 128 0 128 2 64 GPU 0 holds a 64-expert slice of the 128 logical set; GPU 1 holds the complement.
scale-up to DP=4 128 128 256 4 64 switch_and_prepare sets num_physical = 64 × 4 = 256 and pads per-rank buffers. rearrange() then populates the 256 slots by bin-packing logical experts; each surviving GPU still holds 64.
scale-down to DP=2 128 0 128 2 64 Formula: 256 // 4 × 2 = 128. The 128 redundant copies built by scale-up are consumed; back to the start state.

Scale-up conjures redundancy out of thin air to preserve per_gpu; scale-down consumes exactly what scale-up built. The cycle is symmetric and closes cleanly with no operator-supplied redundancy. This is why Exp A3 Phase 1 validates 2 → 4 → 2 at R=0.

Scenario B — cold DP=4 with R=0, 4 → 2 (the failure)

State num_log num_redundant num_phy ep_size per_gpu Notes
cold DP=4 128 0 128 4 32 Each GPU holds 32 of the 128 logical experts, one copy each.
attempted scale-down to DP=2 128 −64 64 2 32 Formula: 128 // 4 × 2 = 64. But num_redundant = num_phy − num_log = 64 − 128 = −64 — assert num_redundant ≥ 0 fires in replicate_experts (vLLM policy/default.py:93).

There is no pre-built redundancy to consume. The formula produces a value below num_log because per_gpu was already at its minimum (1 copy per logical expert per GPU). The invariant preservation policy is fundamentally at odds with the absence of redundancy on a cold start above DP=num_log / per_gpu.

Counterfactual with grow-density. If the formula were instead max(num_logical, num_physical_old // old_ep × new_ep), the attempted scale-down would compute max(128, 64) = 128 and each surviving GPU would grow to hold 64 physical experts (double its previous 32). That preserves the num_phy ≥ num_log invariant — but breaks the per_gpu invariant that the rest of the reshuffle pipeline relies on. Five buffer-sizing assumptions downstream (_map_{new,old}_expert_indices_with_rank_mapping, per-rank expert_weights allocation, physical_to_logical_map shape, the load-window indexing, and the slot-migration semantics in move_to_buffer/move_from_buffer) all assume per_gpu is constant across a scale. This is the five-point cascade in Section 6.3, which is why Track 1b is a multi-file refactor rather than a one-liner.

Scenario C — cold DP=4 with R=128, 4 → 2 (validated Exp B cycle)

State num_log num_redundant num_phy ep_size per_gpu Notes
cold DP=4, R=128 128 128 256 4 64 Operator pre-allocates 128 redundant slots at startup, so each GPU holds 64 physical experts = 2 copies per logical expert on average. Memory footprint doubles vs Scenario B (~35.8 GB vs 16.4 GB per GPU).
scale-down to DP=2 128 0 128 2 64 Formula: 256 // 4 × 2 = 128. num_redundant = 0 ≥ 0, assertion passes. The 128 redundant copies placed at startup are consumed during the scale-down.

The post-scale-down state of Scenario C is structurally identical to the final state of Scenario A. Different paths to the same place: Scenario A lets scale-up build the redundancy implicitly; Scenario C asks the operator to build it up-front at cluster launch. Patch 0001’s error message exists to tell an operator on the failed Scenario B path exactly how to convert their deployment into Scenario C.

Side-by-side summary

Step A: 2→4→2 (R=0) B: cold 4→2 (R=0) C: cold 4→2 (R=128)
Initial (num_phy, num_log, per_gpu) 128, 128, 64 @ DP=2 128, 128, 32 @ DP=4 256, 128, 64 @ DP=4
After scale-up (if any) 256, 128, 64 @ DP=4 — —
After scale-down (num_phy, num_log, per_gpu) 128, 128, 64 @ DP=2 ✓ 64, 128, 32 ✗ assertion fires 128, 128, 64 @ DP=2 ✓
Peak per-GPU expert memory ~16.4 GB (@ DP=2) / ~16.4 GB (@ DP=4) ~16.4 GB (scale-down never completes) ~35.8 GB (both DPs)

The asymmetry between cycling and cold-starting is visible at a glance: Scenario A reaches per_gpu=64 by going through DP=4’s intermediate state (which implicitly requires memory for 64 experts/GPU anyway); Scenario B tries to reach per_gpu=32 at DP=2, which would require shrinking num_phy below num_log, and that’s the invariant violation.

Answer to Q2: Phase 3 does not need more redundancy than Phase 1

The EPLB formula is the same in Phase 1 native and Phase 3 per-GPU containers — it lives in vLLM core, not in any container-specific code. The Exp B headline cycle runs at R=0 (Scenario A) in per-GPU containers, exactly matching Exp A3 Phase 1 native’s 2→4→2 pattern. Patch 0002 unlocks the cold DP=2 start that Phase 3 previously could not do; once DP=2 is reachable, Phase 1 Scenario A behaviour applies unchanged. The redundancy requirement is a function of the starting DP, not the containerisation regime.

Where Phase 3 does interact with the redundancy choice is memory budget: each container has exactly one GPU, so the R=128 load (~35.8 GB per rank for Qwen3-30B-A3B) must fit on that single GPU if the cold-DP=4 fallback path (Scenario C) is used. On 40 GB A100s this works (documented in Section 3.3). On smaller GPUs the feasibility constraint would bite harder in Phase 3 than in Phase 1 multi-GPU deployments, where the same redundancy could be spread across several GPUs via TP; this is a memory- budget concern, not an EPLB one.

Tie to Track 1b. The precise reason Track 1b is a multi-file refactor rather than a one-line formula change is exactly the per-GPU invariant demonstrated above: grow-density scale-down breaks the invariant that every downstream buffer sizing currently assumes (Section 6.3 enumerates the five sites). Any serious fix has to propagate the new per-GPU count through all of them atomically.

7. Conclusions

  1. Exp B’s success criterion is met. The full 2 → 4 → 2 elastic cycle succeeds end-to-end in per-GPU containers with num_redundant_experts=0 (matching Exp A3 Phase 1): scale-up 49.93 s, scale-down 12.92 s, DP=2 throughput returns from 62.21 tok/s (initial) to 67.53 tok/s (post-scale-down), service health maintained throughout.
  2. Scheduler-level reclamation works as designed. Post-scale-down, 2 of 4 GPUs drop to 4 MiB with all 4 containers still Up and DeviceIDs=['N'] — an external coordinator could docker stop the idle containers to release GPUs to the host scheduler with no vLLM state hazard. Which specific two GPUs are freed is run-dependent (Ray placement non-determinism), not a correctness issue.
  3. Patch 0002 is the load-bearing patch for the primary cycle. It enables the cold DP=2 start that Phase 3 previously could not perform (Finding #2 placement-group count assertion). The two-line outer-loop break mirrors the pattern already present in the scale-up sibling add_dp_placement_groups; the cold-start path simply missed it.
  4. Patch 0001 is a diagnostic redirect, off the critical path for the primary cycle. It matters for operators that cold- start above their minimum DP (see §6.4). It raises a precise ValueError naming the --eplb-config.num_redundant_experts value to set; the cold-DP=4 path it enables was validated separately in the lab notebook. A proper repair (Track 1b) is a multi-file vLLM refactor across the EPLB mapping functions and per-rank buffer allocations.
  5. DP=4 post-scale-up is 89.7% of the unpatched Phase 3 baseline. The gap is structural, not a patch cost: scale-up preserves per_gpu = 64, raising total physical experts to 256 (effective R=128). The unpatched Phase 3 baseline runs cold DP=4 with R=0 and per_gpu = 32. This is the per-GPU invariant at work (§6.5); it applies identically on Phase 1 native.
  6. NCCL transport is unchanged and remains the biggest single throughput lever. Per-GPU container channels still use NET/Socket/0; the NCCL gate shim from Exp A (v5 §4.1) is the orthogonal track that would recover most of the 23% Phase 3 penalty.

Next steps. (i) Track 1b: implement the grow-density scale-down path properly — removes the redundancy-config requirement. (ii) Track 3 (v5 §4.2): external container lifecycle coordinator that calls docker stop on departing ranks and docker run on arriving ones, closing the loop to an actual orchestrator-visible scaling operation. (iii) NCCL shim integration (v5 §4.1): apply the Exp A LD_PRELOAD shim to per-GPU containers and re-measure to see how much of the 23% gap it recovers in a vLLM workload.
