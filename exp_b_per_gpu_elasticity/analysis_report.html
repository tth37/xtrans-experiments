in Per-GPU Container Clusters

1. Objective

Exp A3 Phase 3 established that per-GPU containers eliminate the multi-GPU container’s “GPU-trap” problem at the architectural level — each container holds exactly one GPU, so releasing a container truly releases that GPU to the scheduler. But the same phase surfaced three concrete bugs in vLLM Elastic EP that made the scaling API unusable in that topology. Exp B’s objective is to turn Phase 3’s architectural-only elasticity story into a functionally working elastic cycle by patching the vLLM install inside a derived Docker image — without touching the upstream submodule (3rdparty/vllm/), without modifying the research plan’s model or hardware, and without requiring the NCCL bandwidth-recovery shim (v5 §4.1 is an orthogonal track).

Result: Two local patches (0001: EPLB scale-down precondition, 0002: placement-group outer-loop early exit) plus a --eplb-config.num_redundant_experts=128 deployment-time redirect turn Phase 3 into a working elastic cycle. Full DP=4 → 2 → 4 completes in 4.56 s + 41.14 s of service interruption, DP=4 throughput recovers to 94.7% of pre-scale, and 2 of 4 GPUs are fully released to 4 MiB (container-scoped reclamation) while all 4 containers remain Up.

2. Background

Research plan v5 argues that per-GPU containers are the only path to cluster-visible elasticity for in-place systems like vLLM Elastic EP — multi-GPU containers keep the freed GPUs “memory-free but container-trapped” because Docker’s HostConfig.DeviceRequests.DeviceIDs is immutable for the container’s lifetime. Exp A3 Phase 3 built the per-GPU-container topology (four containers on a Docker bridge network, Ray cluster spanning them) and confirmed the architectural premise. It then uncovered three failure sites in vLLM 0.19.0 that Exp B addresses (one each for startup, cold-start, and scaling).

2.1 Finding #1 — DP-master-node resource assumption

Launching with --data-parallel-size 4 --data-parallel-backend ray fails immediately with ValueError: Not enough resources to allocate 4 DP ranks on DP master node, possible to fit 1. vLLM’s Ray placement logic assumes the DP master is a multi-GPU host; in a per-GPU setup it has a single GPU. The workaround is --data-parallel-size-local 1, which tells vLLM “this node contributes 1 rank, find the others elsewhere in the Ray cluster.” This is a design-time CLI assumption, not a bug; no patch is needed, but every deployment must pass the flag. Already documented in Exp A3 §5.6.2.

2.2 Finding #2 — Placement-group count mismatch for DP < Nnodes

Even with --data-parallel-size-local 1 set, starting with --data-parallel-size 2 against a 4-Ray-node cluster asserts:

AssertionError: Created 4 DP placement groups, expected 2

The cold-start function RayDPClient.make_dp_placement_groups walks every Ray node allocating placement groups. An inner break stops the per-node bundle loop once len(placement_groups) == dp_size, but the outer loop over nodes has no early exit, so it keeps visiting remaining nodes and over-creating PGs. The same file’s scale-up sibling add_dp_placement_groups already has the outer break (if num_pg_created >= num_pg_to_create: break) — the cold-start path simply missed it.

2.3 Finding #3 — EPLB num_redundant ≥ 0 on scale-down

Starting at DP=4 works, but POST /scale_elastic_ep {new_data_parallel_size: 2} crashes inside a Ray worker with:

assert num_redundant >= 0
AssertionError

Triggered by policy/default.py:93 in replicate_experts, which requires num_phy >= num_log. The feeder code at eplb_state.py computes num_replicas = num_replicas // ep_group.size() * num_gpus: a proportional shrink that preserves per-GPU slot count and reduces total. When num_redundant_experts=0 (the default), the total already equals num_logical, so any proportional shrink drops below it — for DP=4 → 2 with Qwen3’s 128 logical experts, 128//4×2 = 64 < 128, and the assertion fires. The serving process dies; in the multi-container topology the failure cascades across four Ray actors and the cluster is unrecoverable without a full teardown.

Three observations:

  • The redundancy cost at DP=4 is modest (−6.4% vs unpatched Phase 3). Phase 3’s Socket-NCCL bandwidth is already the bottleneck; doubling physical experts adds routing and memory pressure but does not add a new communication bottleneck.
  • Throughput recovery post-scale-up is 94.7% (113.34 / 119.65) of the pre-scale number, meeting the README’s success-criterion threshold of “within 10%”. The small shortfall is within bench-run variance on this shared host.
  • DP=2 post-scale (91.43 tok/s) runs faster than DP=2 cold (68.53 tok/s) on the same workload. The delta is almost entirely TTFT (233 ms vs 982 ms), driven by compile/warmup caches inherited from the DP=4 run. This is a warmup effect, not a parallelism advantage of the post-scale state.

6. Analysis

6.1 What patch 0001 does

Patch 0001 inserts a 7-line precondition check at the top of eplb_state.rearrange()’s scale-down branch. The check detects the invariant violation before the buggy formula fires:

new_num_replicas = num_replicas // ep_group.size() * num_gpus
if new_num_replicas < model.num_logical_experts:
    raise ValueError(
        f"Scale-down from ep_size={ep_group.size()} to {num_gpus} "
        f"requires num_redundant_experts ≥ {model.num_logical_experts} "
        f"(current: {current_redundant}). Pass "
        f"--eplb-config.num_redundant_experts={model.num_logical_experts} "
        f"at vllm serve startup."
    )
num_replicas = new_num_replicas

This is a refuse-with-actionable-message, not a repair: when num_redundant_experts=0 the patch still cannot scale down (a proper repair would require growing per-GPU expert density across the cluster, see §6.3). What it does is replace a cryptic Ray-worker AssertionError at policy/default.py:93 with a ValueError that names the exact CLI flag the operator needs to set. Combined with --eplb-config.num_redundant_experts=128, this is a complete and reproducible scale-down path on the default vLLM deployment — which we validated end-to-end in Config C.

The math behind the required value: for scale-down from DP=N to DP=M to keep new_num_phy ≥ num_log, the operator needs num_redundant_experts ≥ num_log × (N − M) / M. For N=4, M=2, num_log=128 that is 128 redundant experts. The patch suggests num_logical_experts as a conservative upper bound that satisfies any supported scale-down pair; a TODO comment in the patch notes that the tight per-pair minimum could be computed inline.

6.2 What patch 0002 does

Patch 0002 adds two lines after the inner bundle-allocation loop inside RayDPClient.make_dp_placement_groups:

            if len(placement_groups) == dp_size:
                break
+           if len(placement_groups) ≥ dp_size:
+               break

        if len(placement_groups) < dp_size:
            raise ValueError(...)

The inner break (already present) exits the bundle-allocation loop for one Ray node; the new outer break exits the node loop. Safe across all three VLLM_RAY_DP_PACK_STRATEGY values (strict, fill, span): in the first two, each iteration synchronously emits placement groups with no accumulated state; in span, placement groups are emitted only after collected_bundles reaches world_size, at which point collected_bundles = []. An outer break can therefore never leave a partial bundle remainder behind.

The correctness of the fix is additionally supported by the observation that the file’s scale-up sibling add_dp_placement_groups already has the identical pattern (if num_pg_created ≥ num_pg_to_create: break). The cold-start path simply forgot the outer break; we are aligning with upstream intent, not working around it.

6.3 Why the grow-density repair is a structural refactor

The obvious “real fix” for Finding #3 is to clamp num_replicas to max(num_logical_experts, ...) so per-GPU expert density grows on scale-down: at DP=4→2 the two surviving GPUs would each hold 64 physical experts instead of 32. This was attempted (see the original patch 0001 iteration) and clears the first assertion — but immediately trips a second:

assert old_global_expert_indices.shape[1] == new_global_expert_indices.shape[1]

in rebalance_execute.py:592. Root cause: _map_new_expert_indices_with_rank_mapping hard-codes num_local_physical_experts = new_num_physical_experts // new_ep_size, i.e., the new per-GPU density is assumed to equal the old per-GPU density. Growing the density violates this assumption; the mapping function produces a tensor of a different shape than the caller expects.

Fixing this end-to-end cascades into at least five places (see Track 1b in the Exp B README):

  1. The two mapping functions (_map_{new,old}_expert_indices_with_rank_mapping) need the actual old shape as an explicit parameter, not derived from the new shape.
  2. Per-rank expert_weights allocation — each surviving GPU must grow its expert-slot count (32 → 64 for DP=4→2). Current allocation is fixed at model load time.
  3. physical_to_logical_map per-rank shape is similarly sized at init.
  4. expert_load_window indexing under the grown density.
  5. move_to_buffer/move_from_buffer assume slot i stays on the same GPU across scale events; grow-density migrates slots across GPUs, requiring extended transfer semantics.

This is a multi-file upstream redesign, not a one-line fix. It is deferred as Track 1b in the Exp B scope and is the principal remaining obstacle to making scale-down work with the default num_redundant_experts=0 configuration.

6.4 Why the redundancy redirect is not a workaround tax

Deploying with num_redundant_experts=128 doubles per-GPU expert memory (32 → 64 experts/GPU, ~35.8 GB vs 16.4 GB load) and adds ~6% throughput overhead at DP=4. In return the deployment gains full scale-down/scale-up capability. For a deployment that explicitly wants elasticity, this is a sensible trade — redundancy is already a supported configuration in vLLM, and EPLB’s load-balancer uses it productively when available. Exp B’s contribution is not to eliminate this cost but to make the elasticity path reliably usable when the cost is accepted.

Limitation: the ValueError raised by patch 0001 currently propagates through the Ray worker and kills the engine core — the HTTP response to a bad /scale_elastic_ep call is a timeout rather than a 4xx with the error message. The message shows up in server logs, which is actionable for a human operator but not for a programmatic controller. A proper HTTP-layer rejection requires reaching into the elastic-EP request handler before dispatch; tracked in the Track 1b sketch list.

7. Conclusions

  1. Exp B’s success criterion is met. The full 4 → 2 → 4 elastic cycle succeeds end-to-end in per-GPU containers: scale-down 4.56 s, scale-up 41.14 s, DP=4 throughput recovery 94.7%, no service restart.
  2. Scheduler-level reclamation works as designed. Post-scale-down, 2 of 4 GPUs drop to 4 MiB with all 4 containers still Up and DeviceIDs=['N'] — an external coordinator could docker stop the idle containers to release GPUs to the host scheduler with no vLLM state hazard.
  3. Patch 0002 aligns with upstream intent. The two-line outer-loop break mirrors the pattern already present in the scale-up sibling add_dp_placement_groups; the cold-start path simply missed it.
  4. Patch 0001 is a diagnostic redirect, not a repair. It raises a precise ValueError naming the --eplb-config.num_redundant_experts value the operator must set; combined with that CLI flag the elasticity path is complete. A proper repair (Track 1b) is a multi-file vLLM refactor across the EPLB mapping functions and per-rank buffer allocations.
  5. The redundancy redirect costs ~6% throughput at DP=4 and doubles per-GPU expert memory. Acceptable for a deployment that wants elasticity; Track 1b would eventually remove this tax.
  6. NCCL transport is unchanged and remains the biggest single throughput lever. Per-GPU container channels still use NET/Socket/0; the NCCL gate shim from Exp A (v5 §4.1) is the orthogonal track that would recover most of the 23% Phase 3 penalty.

Next steps. (i) Track 1b: implement the grow-density scale-down path properly — removes the redundancy-config requirement. (ii) Track 3 (v5 §4.2): external container lifecycle coordinator that calls docker stop on departing ranks and docker run on arriving ones, closing the loop to an actual orchestrator-visible scaling operation. (iii) NCCL shim integration (v5 §4.1): apply the Exp A LD_PRELOAD shim to per-GPU containers and re-measure to see how much of the 23% gap it recovers in a vLLM workload.
